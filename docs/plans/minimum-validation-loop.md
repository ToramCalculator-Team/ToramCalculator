# 最小验证闭环推进计划

- **状态**: 产品阶段 0–5 与 SimulatorSession 边界重构已有实现，待性能与最终人工验收
- **日期**: 2026-07-13
- **目标用户**: 带明确验证假设的进阶验证型玩家
- **相关决策**: ADR 0024、0026、0027、0028、0029、0030、0031、0032、0034、0035、0036、0037、0038、0043、0044
- **相关迁移计划**: [AUI 行为状态机迁移计划](./0021-aui-interface-state-migration.md)

本计划负责交付第一个真实用户闭环。AUI 与 SimulatorSession 的职责、运行产出所有权已经落实到 ADR 0037、0038、代码契约和自动化测试；本文继续保留产品范围、性能测量和最终人工验收。

## 目标

交付以下完整路径，而不是一组互不连通的底层能力：

```text
加载持久 Simulator 设计
  -> 手动开始并操作一次木桩验证
  -> 手动结束并查看本次摘要
  -> 自由调整 Character 或 Simulator 设计
  -> 形成新设计副本并再次验证
  -> 用户选择两条运行记录并查看结果、配置与操作差异
  -> 用户选择一个设计副本应用为正式设计
```

闭环完成后，用户应能回答一个具体问题：两种 Character 或 Simulator 设计在这两次实操中有哪些输入、时长、操作与结果差异；系统不会把这些差异裁定为某个变量的纯因果效果。

## 首版边界

### 纳入范围

- 最小闭环的验收 fixture 使用一个 Player Member 对一个固定木桩 Mob Member；通用 Simulator 不限制 Camp、Team 或 Member 数量。
- Player 未设置成员流程时由用户手动释放技能；场景解析为其提供 idle 运行行为，不抢夺控制权。
- 木桩不移动、不攻击、不施加异常，并以极高普通 HP 保证在合理的手动验收时长内不会被耗尽。
- 每次测试由用户手动开始和结束，不限制固定时长。
- 同一 Simulator 会话内可保留多条运行记录，由用户选择两条对比。
- 每 Tick 捕获完整分析快照；实时 UI 可以在捕获后独立节流。
- 顶层结果摘要、技能过程摘要、配置差异和操作差异。
- 记录手动输入与被接纳的语义行动，并允许用户显式把行动录制保存为 Player 成员流程。
- 当前会话保留多个完整设计副本；RunRecord 直接关联产生它的副本，用户可以选择一个副本应用为正式设计。
- Simulator 设计持久化并参与跨端同步；运行产出仅保留在当前会话内存。

### 明确非目标

- 完整流程编辑器或要求用户先配置自动流程；首版只提供从实操录制生成线性成员流程的入口。
- 完整的逐 Tick 时间轴 UI；首版只完成其可靠数据底座。
- 固定时长、自动配对、自动剔除记录或系统裁定“有效可比”。
- 运行历史、Tick 历史或结果摘要的数据库持久化与跨端同步。
- ReplayRecipe、输入重放、历史规则版本兼容或刷新后的结果恢复。
- 设计副本写回时的跨设备并发冲突检测与合并；它属于数据库写入冲突，首版沿用现有数据层语义。
- 正式游戏的等级到能力点预算计算公式，以及任何基于控制变量法的设计合法性判断。
- 首版不提供多成员分析范围编辑、组队聚合视图、移动目标专项分析、攻击型 Mob 专项分析、异常状态专项分析和自动 benchmark；底层不得因此拒绝多成员场景。

## 实施结果与待验收

以下状态来自 2026-07-13 的 Prisma 源 schema、生成结果、运行时代码和自动化验证。数据库最终为 schema v4；v3 只作为迁移链中的历史中间版本存在。

| 能力 | 当前实施结果 |
|---|---|
| 数据模型 | Simulator 独占 Team/Member；Player Member 直接引用 Character；Player 流程可空、Mob Member 流程必须为空；`formationOrder` 全链路生效 |
| 数据库生成 | `DB_SCHEMA_VERSION = 4`；生成器支持列可空性收紧/放宽；v1 到 v4 客户端迁移链通过测试 |
| 场景解析 | DesignCopy 经唯一边界同时产出逻辑输入与带模型、外观、动画描述的 `worldResources`；引擎只消费 `resolvedBehavior` |
| 木桩 fixture | 固有行为仅等待；使用普通 `maxhp = 2,000,000,000`，没有不可击败字段、特殊规则或 GameEngine 分支 |
| 会话与 AUI | AUI actor system 常驻持有 SimulatorSession child；AUI 并行区域唯一表达设计、验证、分析阶段；路由卸载不结束会话 |
| Worker 事实 | 每 Tick 记录严格扁平属性快照，并独立记录输入结果、已接纳行动、技能释放和逐段伤害 |
| 分析与再设计 | 支持单次摘要、显式 A/B、DesignCopy 差异、Simulator 内设计编辑、录制生成成员流程和事务写回 |

尚未完成的是阶段 3 的短/中/长性能样本测量和文末九步真实浏览器验收；它们属于验收工作，不再要求新增第二套领域或运行时设计。

## 推进原则

1. 每一阶段都形成可独立验收的纵向切片；不先并行建设一套数据层、一套 Worker 层和一套空 UI。
2. 一个领域事实只保留一个所有者：Simulator 保存正式设计，DesignCopy 保存会话内候选，Member 保存可选成员流程，Mob 模板保存固有 AI，Worker 记录 Tick，分析器只派生结果；摘要缓存不是新的权威事实。
3. 不从当前 HUD、控制器投影或节流帧反推分析事实；输入接纳、行动录制和结果分析必须读取 Worker 的权威运行事实。
4. 页面、路由和 3D 场景都是会话投影，卸载投影不得结束 Simulator 会话。
5. 木桩和 idle Player 通过普通数据与场景解析表达，不在 GameEngine 增加 `practiceMode` 特殊分支。
6. 每一阶段完成后先删除已被替代的旧读取路径和状态源，再进入下一阶段，禁止长期双写或运行时 fallback。
7. 首版验收 fixture 不上升为场景合法性；分析对象由 DesignCopy 内的分析范围确定。
8. DesignCopy 完整定义运行输入；Worker payload 只做纯派生，不升级为 RunSnapshot、RunParameters 或第二份可编辑设计。

## 阶段 0：数据模型收敛门

此阶段不是用户闭环，但它决定后续是否会建立第二套行为和成员选择模型。最终实现已经收敛到 schema v4。

### 已完成基线

- [x] 从 Character schema 删除 `actions`。
- [x] 将 Member schema 的阵型字段从 `sequence` 改为 `formationOrder`。
- [x] 将 `member.behavior` 收敛为可空成员流程，并在设计解析边界约束 Player/Mob 语义。
- [x] 更新默认数据、备份、生成 schema、数据库 schema 版本和客户端迁移链结构测试到 v4。

### 收敛项

- [x] 按 ADR 0032 将 `member.behavior` 改为可空：Player Member 可以保存流程或为空，Mob Member 必须为空。
- [x] 完成 ADR 0026、0031：Team/Member 由单个 Simulator 独占；Member 使用按类型区分的显式关系，Player Member 直接关联 Character，不引入多态 `relationId`。
- [x] 将 Mob 难度收紧为 Mob Member 专属可选配置；非 Mob Member 不保存无语义的默认难度。
- [x] 按 ADR 0036 的完整运行输入边界实现：Simulator 增加随机种子、预期手动控制成员和分析范围；结果影响配置不得在运行开始时临时附加。
- [x] 审计现有 `RuntimeConfig`、`runtimeSelection` 和引擎默认值：`logicHz` 等会改变逻辑结果的可配置值归入 Simulator；驱动模式、输出策略、UI 节流和手动开始/结束等执行事实继续由运行时宿主持有。运行入口不得再用 `createRealtimeConfig()` 或“第一个成员”补出设计语义。
- [x] 为 Member 类型建立一致性校验：与 `member.type` 对应的关系必须存在，其他类型关系必须为空；Mob Member 的 `behavior` 必须为空；无效组合在进入引擎前失败。
- [x] 只修改 `db/schema/main.prisma`、`db/schema/models/*.prisma` 和 `db/schema/enums.ts` 中的源定义；运行 `pnpm generate`，由生成器产出迁移、客户端逻辑、类型和 repository，不直接编辑生成产物。
- [x] 更新默认数据、备份 fixture、repository、预览数据、场景 schema 和引擎构造调用方；Mob AI 继续只保留在 Mob 模板，不复制到 Member。
- [x] 场景解析按 MemberType 产出统一 `resolvedBehavior`：Player 读取可选成员流程或解析为手动 idle，Mob 读取 `mob.actions`；引擎运行时不再感知持久来源差异。
- [x] 阵型计算、查询排序、预览数据和 UI 显示统一读取 `formationOrder`；删除 `player.useIn`、`character.actions` 及其他被替代的引擎运行时读取路径。

### 验收

- 一个持久 Simulator 可以表达一名无成员流程的手动 Player 和一名使用模板固有 AI 的 Mob，不复制 Mob AI。
- 同一个 Character 可被两个 Simulator 引用，并持有不同的可选成员流程；修改一方不改变另一方。
- 引擎场景输入不再需要加载 Player 的全部 Character 后读取 `player.useIn`。
- Member 写入和场景解析会拒绝类型与关系不匹配、多填关系或永久悬空的引用。
- 搜索确认 `character.actions` 和成员阵型 `sequence` 不再参与场景执行；`mob.actions` 只由场景解析作为 Mob 固有 AI 读取。
- `pnpm generate` 成功，生成后的客户端迁移链测试通过，相关 schema、repository、预览与引擎路径通过 Biome 检查。

## 阶段 1：从持久设计创建可操作木桩世界

本阶段交付第一条端到端输入链：用户选择一个真实 Simulator 设计，应用从它创建干净且可手动操作的世界。

### 工作项

- [x] 用明确的方案选择与创建入口替代 `defaultSimulatorId`；无既有方案时可以从账号 Character 创建持久木桩验证设计，路由参数只表达当前投影的方案身份，不拥有会话。
- [x] 建立 ADR 0027、0036 的唯一场景解析边界，从 DesignCopy 一次派生 `EngineScenarioInput` 和 `worldResources`，不创建长期 RunSnapshot。
- [x] 场景解析保持通用的 Camp/Team/Member 输入，不增加一对一人数校验或练习模式分支。
- [x] 提供一份普通的持久验收 fixture：一名手动 Player 与一个使用静止固有 AI 的 Mob；它只服务测试和首次体验。
- [x] Player Member 不保存空行为树；`behavior = null` 经场景解析得到手动 idle 运行行为，只有用户控制器命令可以发起技能。
- [x] 木桩使用 Mob 模板固有 idle AI，不移动、不攻击、不施加异常；`maxhp = 2,000,000,000` 只是普通 Mob 数据。
- [x] 不增加不可击败字段、不可击败规则、木桩 ID 判断或练习模式分支；木桩耐久完全由普通 `maxhp` 表达。
- [x] 设计阶段可以直接按阵型显示解析后的模型、外观和动画资源，不要求先创建或启动逻辑世界。

### 验收

- 从数据库选择 Simulator 后，可以稳定加载正确的 Character、Mob、成员行为和模型资源。
- 连续两次解析未修改的设计得到等价逻辑输入；Player 运行行为可追溯到成员流程状态，木桩运行行为可追溯到 Mob 模板固有 AI。
- 启动后用户可以手动释放技能，Player 不会自行行动，木桩不会还击或结束测试。
- 无效方案在创建世界前失败，不产生半初始化引擎或残留场景实体。
- 一个包含额外 Team 或 Member 的结构合法方案不会因首版木桩验收假设而被场景解析拒绝。

## 阶段 2：建立会话与运行生命周期

本阶段把“页面里的引擎演示”变成可跨工作面持续存在的 Simulator 会话，并建立多次验证的容器。

### 工作项

- [x] 将 Simulator 会话所有权迁出 `RealtimeSimulator`，由应用级唯一入口显式创建、切换和结束。
- [x] AUI 状态机唯一持有当前会话的设计、验证、分析阶段；删除独立全局 `businessPhaseMachine` 的重复阶段事实。
- [x] 页面、HUD 与 3D 场景只订阅会话读面并发送语义意图；卸载时只释放自己的订阅和视觉资源。
- [x] 加载正式 Simulator 后建立完整当前 DesignCopy；随机种子、预期主控和分析范围均来自副本，不从页面状态临时拼装。
- [x] “开始验证”直接选择当前 DesignCopy，派生 Worker 输入并重建干净世界；副本从此保持原值，后续编辑通过 copy-on-write 创建新副本。
- [x] 首版 Simulator 默认把预期主控 Player 设为分析来源、全部敌对成员设为分析目标；配置不完整时在设计边界暴露，而不是运行时猜测第一个成员。
- [x] 会话生命周期、RunRecord 所有权、显式切换/结束和验证失败决策已经迁入 SimulatorSession child，并删除旧 Provider 与页面旁路。

### 验收

- 离开 Simulator 进入 Character 工作面再返回，仍是同一会话，既有运行记录没有丢失。
- 每次点击开始都从 Tick 0、初始资源和初始成员状态创建新世界，不继承上一条运行的 HP、MP、Buff、行为树变量或随机状态。
- 相同 DesignCopy 的重复运行使用相同随机种子；系统不会在开始运行时自动换种子。
- 相同 DesignCopy 派生出的结果影响配置完全一致；改变宿主输出频率或 UI 节流不会改变 Tick 历史和摘要。
- 用户可以连续完成至少三条不同时长的运行，记录互不覆盖。
- 已经产生运行记录的副本不会被后续设计修改；继续编辑形成具有新身份的副本。
- 切换到另一个 Simulator 才会替换会话，并明确清理旧会话内存数据。

## 阶段 3：建立逐 Tick 权威记录

本阶段先保证分析事实完整，再让实时 UI 从这些事实中降频投影。完整时间轴 UI 不在本阶段实现。

### 工作项

- [x] 按 ADR 0029 为 `ModifierSource`、modifier 条目、`AttributeSnapshotLeaf` 和属性快照建立统一 Zod schema，公开类型全部从 schema 推导。
- [x] StatContainer 直接导出扁平 `AttributePath -> AttributeSnapshotLeaf`，保留当前值、基础值及完整 modifier 来源，不先构造嵌套树。
- [x] `MemberSerializeData`、`FrameSnapshot` 和 Worker 协议共用该属性契约；快照覆盖分析所需成员，不依赖当前控制器绑定或 UI 选中对象。
- [x] 调整 Tick 收尾顺序：结算完成后无条件创建一次权威 FrameSnapshot，并写入当前 Engine 运行产出收集器；之后才判断是否向 UI 推送最新帧。
- [x] 从 `frame_snapshot` 热路径移除 `JSON.stringify/parse` 式清洗；不可结构化克隆的数据在 schema 与构造边界解决。
- [x] 在权威结算点记录成功技能和伤害事实，至少保留来源 Member、目标 Member、来源技能或“其他来源”、伤害值及相对模拟时间。
- [x] 技能释放事实与伤害事实分离：一次多段技能只计一次成功释放，但可以产生多条伤害；当前控制器 `hit` 投影不得作为分析数据源。
- [x] 分开记录输入轨迹与行动录制：前者包含接纳或拒绝结果，后者只包含 Member FSM 真正接纳的语义行动。
- [x] 行动接纳事实由实际开始或入队的运行时边界产生，并绑定引擎模拟时间；不得用 MessageRouter 派发成功、Tick 序号或主线程时间代替。
- [x] 行动录制保存相对运行起点的模拟时间和完整语义 payload，不把时间压缩成无间隔的技能 ID 数组。
- [x] 按 ADR 0038 完成 `EngineRunOutput` 幂等移交并删除旧 Worker 历史目录；本文继续验收逐 Tick 完整性与性能。
- [x] 按 ADR 0044 将逐 Tick 对象数组迁移为 SAB-only 分段历史，删除结束阶段的完整历史映射、深冻结、复制和重复解析；浏览器与性能验收见 [Simulator Tick 状态历史 SAB 性能重构计划](./simulator-tick-state-history-sab-performance.md)。

### 正确性验收

- 对任意正常结束的运行，分段目录声明的 Tick 数等于已执行 Tick 数；读取器可以精确还原每个连续 Tick，且不存在重复或缺口。
- 把实时 UI 推送频率分别设为低频和高频，得到的 `EngineRunOutput` Tick 记录与最终摘要相同。
- 未绑定控制器或未被 UI 选中的成员仍有完整权威记录。
- 同一多段技能的释放次数、命中次数和总伤害可以被分别验证；非技能伤害进入“其他来源”。
- 重复点击冷却中的技能会出现在输入轨迹中，但只有实际接纳的技能行动进入行动录制。
- 相同录制转换出的流程保持相同计划模拟时间；逻辑频率只影响执行量化精度，UI 推送频率不影响调度。
- 属性值及 modifier 来源在分段目录完成跨线程移交后仍可按任意 Tick 完整还原。

### 性能验收

- 首个统一参考负载为 `60 Hz`、8 个成员、单条运行 10 分钟，并在同一 SimulatorSession 连续保留 3 条运行记录；它只用于性能验收，不构成运行时长、成员数量或记录数量上限。
- 短、中、长样本分别使用 30 秒、3 分钟和 10 分钟，并使用相同的测量点与统计口径。
- 从 SimulatorSession 接受 `validation.finish.requested` 到 RunRecord 已提交且分析面板首次绘制并可交互，参考负载下 P95 不超过 1 秒、单次不超过 2 秒；结束处理不得在主线程形成超过 200 ms 的单个任务。
- 从 SimulatorSession 接受 `validation.returnToDesign.requested` 到设计面板首次绘制并可交互，参考负载下 P95 不超过 100 ms、单次不超过 250 ms；返回路径不得遍历、校验或复制 Tick 记录，也不得产生超过 50 ms 的主线程长任务。
- 分别测量 Tick 历史写入、Worker 分段留存、结束时分段目录移交和 Session 持有三条记录时的内存成本，不能只记录总帧率。
- 用短、中、长三种人工运行样本记录 CPU、内存增长和 GC 行为；产品不因此强制固定用户测试时长。
- 不得静默丢弃旧 Tick。按 ADR 0044 使用字典化、分段基准、增量列与经测量选择的全量列控制容量，并以 SAB 在完成边界共享同一份历史；不提供对象数组或 ArrayBuffer 兜底路径。

## 阶段 4：交付第一次可用分析

本阶段把一条 RunRecord 转换为用户能理解的摘要，再允许用户明确选择两条记录进行比较。

### 工作项

- [x] 建立纯派生分析器，以 RunRecord 关联的 DesignCopy 分析范围与 Tick 记录为输入，不读取当前 Character、正式 Simulator 或 HUD 状态。
- [x] 摘要可以在当前会话中缓存以支持快速切换，但必须能从 DesignCopy 与 Tick 记录重新生成；缓存不持久化，也不参与结果真实性判断。
- [x] 单条记录输出验证时长、总伤害和平均 DPS；时长使用模拟时间，DPS 明确定义为 `总伤害 / 验证秒数`。
- [x] 单条记录输出成功技能顺序，以及每技能释放次数、总伤害、伤害占比和平均单次伤害。
- [x] 无法归因到技能的伤害显式聚合为“其他来源”，不能丢弃或错误归给最近技能。
- [x] 会话 UI 列出所有 RunRecord 的开始顺序、时长和核心摘要，用户显式选择恰好两条记录；不得自动使用最近两条。
- [x] 对比视图同时展示 A、B 原始值、`B - A` 绝对差和相对 A 的百分比差；A 为零时显示不可计算，不生成无穷值。
- [x] 展示两个 DesignCopy 中的等级与五项基础能力差异，以及成功技能顺序、释放次数和验证时长差异。
- [x] 操作差异逐条展示被拒绝输入的模拟时刻、语义动作和原因，并与真正接纳的行动序列分开呈现。
- [x] 文案只陈述“这两次运行的差异”，不宣称配置变化产生了纯因果收益。

### 验收

- 用户结束一次运行后立即看到时长、总伤害、平均 DPS 和技能过程摘要。
- 用户完成多次运行后可以任意选择两条，交换 A/B 时绝对差和百分比基准按定义变化。
- 摘要中的技能伤害加“其他来源”等于顶层总伤害；各技能伤害占比与总量一致。
- 分析范围外成员的伤害不会混入摘要，但其 Tick 数据仍完整保留。
- 两条时长或技能顺序不同的运行会明确显示这些差异，但仍允许用户继续比较。
- 刷新页面后运行记录消失，持久 Simulator 设计仍然存在并可重新加载。

## 阶段 5：闭合“再设计”路径

本阶段把完整设计副本编辑、比较和差异应用接回活跃 Simulator 会话，完成首个可重复循环。

### 工作项

- [x] 活跃 Simulator 会话只通过 Simulator 自己的设计 CUI 读取和修改当前 DesignCopy；Character 工作面只维护正式 Character，不读取或修改 DesignCopy。两处可以复用无状态表单和纯校验函数，但不能共享编辑状态或提交通道；应用副本之前不得把候选值写入 repository。
- [x] Character 与 Simulator 编辑不为控制变量法增加等级锁定、点数预算、局部提交草稿或自动修正。
- [x] 用户可以自由改变等级、五项基础能力及总和；结构合法的数据始终可以进入 DesignCopy、运行和参与记录对比。
- [x] 对比视图同时展示等级、五项基础能力各自数值和总和差异；未覆盖的其他输入不得被暗示为相同。
- [x] 当前 DesignCopy 尚未产生运行记录时可以继续编辑；一旦被 RunRecord 引用，任何 Character、Simulator 或 Member 修改都先 copy-on-write 形成新副本。
- [x] 用户选择“应用为正式设计”时，先展示目标副本相对正式基线的实体与字段差异，再在一个明确写入边界应用全部 insert、update 和 delete。
- [x] 副本中共享 Character 的变化直接更新原 Character，并自然影响其他引用它的 Simulator；只有显式“复制 Character”操作才创建新实体。
- [x] 用户可以从一条 RunRecord 显式选择“保存为成员流程”；系统只转换行动录制，不包含被拒绝输入，也不在运行结束时自动覆盖设计。
- [x] 生成的初始成员流程严格保留行动相对模拟时间，不推断技能可用条件或自动重试；计划时刻无法执行时只记录该行动失败，其他行动仍在各自原定时刻独立尝试。
- [x] 保存结果先写入新的当前 DesignCopy 的 Player Member `behavior`；只有应用该副本时才进入正式 Simulator 持久化与同步。
- [x] 返回 Simulator 时保留同一会话、DesignCopy 集合和旧 RunRecord；下一次验证运行当前选择的副本。
- [x] 对比视图从两条 RunRecord 关联的 DesignCopy 展示设计差异，不读取当前正式数据来重写旧记录。

### 验收

- 改变等级或五项总和不会阻止保存、运行或比较；无效关系、错误类型和不可解析行为仍会在结构边界失败。
- 修改 Character 时，已经结束和正在分析的 RunRecord 保持原始输入与结果不变。
- 新运行使用新 DesignCopy，旧运行继续关联并显示修改前副本。
- 保存行动录制后，当前 Simulator 获得可再次执行的成员流程；未执行保存时，运行不会改变 Simulator 设计。
- 修改配置导致行动在原计划时刻不可用时，新运行明确显示失败；系统不会终止整条流程，也不会改变、跳过或延后其他计划行动。
- 用户可以完整走通“设计 -> 验证 -> 分析 -> 再设计 -> 再验证 -> 选择两条记录对比”，中途无需学习流程编辑器。

## 最终验收脚本

以下步骤尚待真实浏览器人工验收，不因自动化测试通过而提前标记完成。

1. 创建或加载一个包含 Player 与木桩的持久 Simulator，确认 Player 使用可选成员流程、木桩使用 Mob 模板固有 AI。
2. 开始验证，按用户自己的节奏手动释放若干技能，手动结束并查看第一条摘要。
3. 在同一会话的 Simulator 设计界面修改当前 DesignCopy 中的一项或多项 Character 设计输入；验收者可以自愿采用同等级、同总点数的能力重分配，但此时正式数据库数据不得改变。另行打开 Character 工作面时只能维护正式 Character，不能读取或改写该 DesignCopy。
4. 返回 Simulator，确认第一条 RunRecord 及其 DesignCopy 仍存在；从修改后的副本开始第二次干净验证，手动结束。
5. 在记录列表中明确选择第一、第二条，查看时长、总伤害、平均 DPS、技能过程和能力分配差异。
6. 再执行一条故意使用不同时长或技能顺序的运行，确认系统展示操作差异但不阻止选择，也不自动裁定可比性。
7. 从一条手动运行记录显式生成 Player 成员流程，确认它先进入新的 DesignCopy；使用该副本再次运行，核对计划模拟时间、实际接纳结果和失败记录，并确认单个行动失败不妨碍后续行动在原定时刻尝试。
8. 选择一个候选 DesignCopy 应用为正式设计，确认差异预览与实际 insert、update、delete 一致，共享 Character 的变化按原实体身份写回。
9. 刷新页面，确认会话内 DesignCopy 和 RunRecord 丢失，而刚应用的 Simulator 与 Character 正式设计可从持久数据恢复。

满足以上步骤，并同时满足阶段 3 的逐 Tick 完整性断言，才视为最小验证闭环完成。仅完成 schema、会话类、快照 schema 或摘要组件中的任意一部分都不算闭环。

## 验证策略

- schema / 枚举变化：`pnpm generate`。
- 数据库生成变化：运行客户端迁移链测试，确认生成迁移可从 baseline 连续执行到当前 schema。
- TS / UI 小步变更：`pnpm biome check <相关路径>`。
- 纯分析器、记录目录、设计差异和状态机：增加 Vitest 单元测试，覆盖零时长、零基准、多段技能、其他来源、不同运行选择和非法阶段事件。
- Worker Tick 记录：增加确定性集成测试，断言 Tick 数、索引连续性、UI 节流独立性和多次运行隔离。
- 每个纵向阶段完成后执行对应人工验收；只有影响包入口、Vite、Service Worker 或生成产物引用时才运行完整 `pnpm build`。

## 主要风险与止损点

| 风险 | 观察信号 | 处理顺序 |
|---|---|---|
| 每 Tick 完整属性来源导致内存增长过快 | 长运行中 SAB 分配量过大或频繁 GC | 按 ADR 0044 测量字段贡献，调整来源字典、增量列与全量列布局，不回退到对象数组 |
| 技能伤害无法稳定归因 | 聚合总伤害与引擎结算不一致 | 回到伤害结算事实补来源契约，不在分析器按时间猜测 |
| 页面和会话仍有双重生命周期 | 路由切换后运行丢失或控制器重复绑定 | 删除页面级 load/reset 所有权，保留唯一会话入口 |
| 设计修改污染旧结果 | 旧记录显示正式数据库或当前副本的最新值 | RunRecord 只读绑定的 DesignCopy；已运行副本禁止原地修改 |
| 场景解析留下多行为来源 | 引擎运行时仍直接读取 Member 与 Mob 持久字段 | 阶段 0 不通过就不进入会话实现；引擎只消费解析后的单一行为 |
| 范围重新膨胀到流程编辑或历史恢复 | 首次验证依赖新编辑器或版本模型 | 回到非目标清单，相关需求另立后续计划，不阻塞本闭环 |

## 完成定义

- 产品层：真实用户无需开发工具即可完成最终验收脚本。
- 数据层：Simulator 设计可持久化和同步，运行产出不进入数据库。
- 生命周期：普通工作面切换不销毁会话，显式切换方案或结束会话才清理。
- 设计层：DesignCopy 完整定义运行输入；RunRecord 直接关联副本，不存在独立 RunSnapshot 或 RunParameters。
- 写回层：候选副本只有经用户确认后才按实体差异应用为正式设计，完整支持 insert、update 和 delete。
- 引擎层：每 Tick 权威记录完整，实时 UI 节流不影响分析。
- 分析层：顶层与技能摘要守恒，两条记录的输入、操作和结果差异同时可见。
- 领域层：成员流程与 Mob 固有 AI 边界明确，不存在 Character 流程、引擎多来源 fallback、页面级会话或 HUD 分析事实等第二套所有权。
