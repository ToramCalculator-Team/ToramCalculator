# 最小验证闭环推进计划

- **状态**: 可实施
- **日期**: 2026-07-13
- **目标用户**: 带明确验证假设的进阶验证型玩家
- **相关决策**: ADR 0024、0026、0027、0028、0029、0030、0031、0032、0033、0034、0035、0036
- **相关迁移计划**: [AUI 行为状态机迁移计划](./0021-aui-interface-state-migration.md)

本计划负责交付第一个真实用户闭环，不替代 AUI 迁移计划。AUI 计划继续负责跨模态状态源迁移；本计划负责把数据、会话、引擎记录和分析 UI 串成可实际使用的纵向切片。

## 目标

交付以下完整路径，而不是一组互不连通的底层能力：

```text
加载持久 Simulator 设计
  -> 手动开始并操作一次木桩验证
  -> 手动结束并查看本次摘要
  -> 自由调整 Character 或 Simulator 设计
  -> 形成新设计副本并再次验证
  -> 用户选择两条运行记录并查看结果、配置与操作差异
  -> 用户选择一个设计副本应用为正式设计
```

闭环完成后，用户应能回答一个具体问题：两种 Character 或 Simulator 设计在这两次实操中有哪些输入、时长、操作与结果差异；系统不会把这些差异裁定为某个变量的纯因果效果。

## 首版边界

### 纳入范围

- 最小闭环的验收 fixture 使用一个 Player Member 对一个固定木桩 Mob Member；通用 Simulator 不限制 Camp、Team 或 Member 数量。
- Player 未设置成员流程时由用户手动释放技能；场景解析为其提供 idle 运行行为，不抢夺控制权。
- 木桩不移动、不攻击、不施加异常，并且不会因被击杀而强制结束测试。
- 每次测试由用户手动开始和结束，不限制固定时长。
- 同一 Simulator 会话内可保留多条运行记录，由用户选择两条对比。
- 每 Tick 捕获完整分析快照；实时 UI 可以在捕获后独立节流。
- 顶层结果摘要、技能过程摘要、配置差异和操作差异。
- 记录手动输入与被接纳的语义行动，并允许用户显式把行动录制保存为 Player 成员流程。
- 当前会话保留多个完整设计副本；RunRecord 直接关联产生它的副本，用户可以选择一个副本应用为正式设计。
- Simulator 设计持久化并参与跨端同步；运行产出仅保留在当前会话内存。

### 明确非目标

- 完整流程编辑器或要求用户先配置自动流程；首版只提供从实操录制生成线性成员流程的入口。
- 完整的逐 Tick 时间轴 UI；首版只完成其可靠数据底座。
- 固定时长、自动配对、自动剔除记录或系统裁定“有效可比”。
- 运行历史、Tick 历史或结果摘要的数据库持久化与跨端同步。
- ReplayRecipe、输入重放、历史规则版本兼容或刷新后的结果恢复。
- 设计副本写回时的跨设备并发冲突检测与合并；它属于数据库写入冲突，首版沿用现有数据层语义。
- 正式游戏的等级到能力点预算计算公式，以及任何基于控制变量法的设计合法性判断。
- 首版不提供多成员分析范围编辑、组队聚合视图、移动目标专项分析、攻击型 Mob 专项分析、异常状态专项分析和自动 benchmark；底层不得因此拒绝多成员场景。
- 提前引入 SAB；是否采用由逐 Tick 捕获的真实性能数据决定。

## 当前基线与缺口

以下基线来自 2026-07-12 的工作区。数据迁移正在进行，不能把“文件中已有改动”视为阶段 0 已验收。

| 能力 | 当前状态 | 进入闭环前必须收敛到 |
|---|---|---|
| 行为解析 | 已删除 `character.actions`，`member.behavior` 当前必填；Player/Mob 运行时仍直接读取不同持久字段 | Player 流程可空，Mob Member 流程必须为空；场景解析按类型产出统一运行行为 |
| 阵型顺序 | schema 已改为 `formationOrder`，引擎阵型代码仍读取 `sequence` | 全链路只使用 `formationOrder` |
| 成员选择 | Member 仍通过 `playerId -> player.useIn/characters[]` 间接选择 Character | 按 MemberType 使用显式关系；Player Member 直接关联 Character |
| 聚合所有权 | Simulator 与 Team 仍是多对多关系 | Team/Member 归单个 Simulator 独占 |
| 方案入口 | `/simulator` 硬编码跳转 `defaultSimulatorId` | 用户可明确加载或创建持久方案 |
| 场景输入 | 路由直接查询引擎专用深关系树 | 唯一场景解析边界同时产出逻辑输入与 `worldResources` |
| 会话生命周期 | `RealtimeSimulator` 挂载时加载，卸载时 `engine.reset()` 并解绑控制器 | 页面只负责输入和投影，会话由应用级边界拥有 |
| 业务阶段 | 独立全局 `businessPhaseMachine` 保存 `designing/simulating/analyzing` | 当前 Simulator 会话的 AUI 状态唯一表达设计、验证、分析 |
| Tick 数据 | `GameEngine` 在创建 `FrameSnapshot` 之前按 UI 频率节流 | 每个逻辑 Tick 先捕获并记录，再派生实时 UI 投影 |
| 属性协议 | `attrs` 是嵌套 `Record<string, unknown>` | 统一的扁平严格属性快照 schema |
| 技能归因 | 控制器 `hit` 投影不包含来源技能，不能形成权威分析 | 从结算事实保留技能、来源成员、目标与伤害归因 |
| 运行分析 | 只有最新帧和 HUD 投影 | 会话内多运行记录、摘要聚合器和两条记录对比 |
| 设计差异 | Character 允许自由编辑，但旧运行尚无稳定输入差异来源 | 保持设计开放，并从运行实际输入展示等级、能力总和及各项差异 |
| 设计版本 | 当前没有完整本地设计副本或副本差异应用能力 | 设计副本完整定义运行；已运行副本保持原值，编辑产生新副本 |
| 运行输入 | 随机种子、主控和分析口径尚未收拢为完整 Simulator 设计 | 所有结果影响配置归入 Simulator；不建立 RunSnapshot 或 RunParameters |

## 推进原则

1. 每一阶段都形成可独立验收的纵向切片；不先并行建设一套数据层、一套 Worker 层和一套空 UI。
2. 一个领域事实只保留一个所有者：Simulator 保存正式设计，DesignCopy 保存会话内候选，Member 保存可选成员流程，Mob 模板保存固有 AI，Worker 记录 Tick，分析器只派生结果；摘要缓存不是新的权威事实。
3. 不从当前 HUD、控制器投影或节流帧反推分析事实；输入接纳、行动录制和结果分析必须读取 Worker 的权威运行事实。
4. 页面、路由和 3D 场景都是会话投影，卸载投影不得结束 Simulator 会话。
5. 木桩和 idle Player 通过普通数据与场景解析表达，不在 GameEngine 增加 `practiceMode` 特殊分支。
6. 每一阶段完成后先删除已被替代的旧读取路径和状态源，再进入下一阶段，禁止长期双写或运行时 fallback。
7. 首版验收 fixture 不上升为场景合法性；分析对象由 DesignCopy 内的分析范围确定。
8. DesignCopy 完整定义运行输入；Worker payload 只做纯派生，不升级为 RunSnapshot、RunParameters 或第二份可编辑设计。

## 阶段 0：数据模型收敛门

此阶段不是用户闭环，但它决定后续是否会建立第二套行为和成员选择模型。你当前进行的 schema 迁移在本阶段统一验收。

### 工作项

- [ ] 完成 ADR 0032：删除 `character.actions`，把 `member.behavior` 改为可空的成员流程，并把 `member.sequence` 全链路改为 `formationOrder`。
- [ ] 完成 ADR 0026、0031：Team/Member 由单个 Simulator 独占；Member 使用按类型区分的显式关系，Player Member 直接关联 Character，不引入多态 `relationId`。
- [ ] 完成 ADR 0036：Simulator 增加随机种子、预期手动控制成员和分析范围；结果影响配置不得在运行开始时临时附加。
- [ ] 审计现有 `RuntimeConfig`、`runtimeSelection` 和引擎默认值：`logicHz` 等会改变逻辑结果的可配置值归入 Simulator；驱动模式、输出策略、UI 节流和手动开始/结束等执行事实继续由运行时宿主持有。运行入口不得再用 `createRealtimeConfig()` 或“第一个成员”补出设计语义。
- [ ] 为 Member 类型建立一致性校验：与 `member.type` 对应的关系必须存在，其他类型关系必须为空；Mob Member 的 `behavior` 必须为空；无效组合在进入引擎前失败。
- [ ] 将 Mob 难度收紧为 Mob Member 专属配置；非 Mob Member 不保存无语义的默认难度。
- [ ] 更新迁移、默认数据和同步 schema：既有 Character 行为迁移到对应 Player Member；Mob AI 继续只保留在 Mob 模板，不复制到 Member。
- [ ] 运行 `pnpm generate`，更新生成类型后再修正 repository、预览数据、场景 schema 和引擎构造调用方。
- [ ] 场景解析按 MemberType 产出统一 `resolvedBehavior`：Player 读取可选成员流程或解析为手动 idle，Mob 读取 `mob.actions`；引擎运行时不再感知持久来源差异。
- [ ] 阵型计算、查询排序和 UI 显示统一读取 `formationOrder`，仓库中不再把 `sequence` 当作成员顺序。

### 验收

- 一个持久 Simulator 可以表达一名无成员流程的手动 Player 和一名使用模板固有 AI 的 Mob，不复制 Mob AI。
- 同一个 Character 可被两个 Simulator 引用，并持有不同的可选成员流程；修改一方不改变另一方。
- 引擎场景输入不再需要加载 Player 的全部 Character 后读取 `player.useIn`。
- Member 写入和场景解析会拒绝类型与关系不匹配、多填关系或永久悬空的引用。
- 搜索确认 `character.actions` 和成员阵型 `sequence` 不再参与场景执行；`mob.actions` 只由场景解析作为 Mob 固有 AI 读取。
- `pnpm generate` 成功，相关 schema、repository 与引擎路径通过 Biome 检查。

## 阶段 1：从持久设计创建可操作木桩世界

本阶段交付第一条端到端输入链：用户选择一个真实 Simulator 设计，应用从它创建干净且可手动操作的世界。

### 工作项

- [ ] 用明确的方案选择或创建入口替代 `defaultSimulatorId`；路由参数只表达当前投影的方案身份，不拥有会话。
- [ ] 建立 ADR 0027、0036 的唯一场景解析边界，从 DesignCopy 一次派生 `EngineScenarioInput` 和 `worldResources`，不创建长期 RunSnapshot。
- [ ] 场景解析保持通用的 Camp/Team/Member 输入，不增加一对一人数校验或练习模式分支。
- [ ] 提供一份普通的持久验收 fixture：一名手动 Player 与一个使用静止固有 AI 的 Mob；它只服务测试和首次体验。
- [ ] Player Member 不保存空行为树；`behavior = null` 经场景解析得到手动 idle 运行行为，只有用户控制器命令可以发起技能。
- [ ] 木桩使用 Mob 模板固有 AI；为首个场景选择本身不移动、不攻击、不施加异常且不会触发运行自动结束的目标数据。
- [ ] 若现有行为或属性机制无法表达不会被击杀的木桩，补足通用领域能力；不得把木桩 ID 或练习模式判断硬编码进 GameEngine。
- [ ] 设计阶段可以直接显示解析后的成员与静态世界资源，不要求先启动逻辑引擎。

### 验收

- 从数据库选择 Simulator 后，可以稳定加载正确的 Character、Mob、成员行为和模型资源。
- 连续两次解析未修改的设计得到等价逻辑输入；Player 运行行为可追溯到成员流程状态，木桩运行行为可追溯到 Mob 模板固有 AI。
- 启动后用户可以手动释放技能，Player 不会自行行动，木桩不会还击或结束测试。
- 无效方案在创建世界前失败，不产生半初始化引擎或残留场景实体。
- 一个包含额外 Team 或 Member 的结构合法方案不会因首版木桩验收假设而被场景解析拒绝。

## 阶段 2：建立会话与运行生命周期

本阶段把“页面里的引擎演示”变成可跨工作面持续存在的 Simulator 会话，并建立多次验证的容器。

### 工作项

- [ ] 将 Simulator 会话所有权迁出 `RealtimeSimulator`，由应用级唯一入口显式创建、切换和结束。
- [ ] AUI 状态机唯一持有当前会话的设计、验证、分析阶段；删除独立全局 `businessPhaseMachine` 的重复阶段事实。
- [ ] 页面、HUD 与 3D 场景只订阅会话读面并发送语义意图；卸载时只释放自己的订阅和视觉资源。
- [ ] 加载正式 Simulator 后建立完整当前 DesignCopy；随机种子、预期主控和分析范围均来自副本，不从页面状态临时拼装。
- [ ] “开始验证”直接选择当前 DesignCopy，派生 Worker 输入并重建干净世界；副本从此保持原值，后续编辑通过 copy-on-write 创建新副本。
- [ ] 首版 Simulator 默认把预期主控 Player 设为分析来源、全部敌对成员设为分析目标；配置不完整时在设计边界暴露，而不是运行时猜测第一个成员。
- [ ] “结束验证”由用户显式触发：停止推进 Tick，封存当前 RunRecord，转入分析；暂停和页面切换都不等价于结束。
- [ ] 会话内维护 DesignCopy 集合、RunRecord 目录与关联；多条运行可以共享一个副本，记录和未应用副本都不写入数据库或同步层。
- [ ] RunRecord 只保存 `designCopyId` 与运行事实；删除独立 RunSnapshot、RunParameters 和每条记录的 Character 副本。
- [ ] 显式切换 Simulator 或结束会话时清理 Worker 记录、控制器和场景；普通路由切换不清理。

### 验收

- 离开 Simulator 进入 Character 工作面再返回，仍是同一会话，既有运行记录没有丢失。
- 每次点击开始都从 Tick 0、初始资源和初始成员状态创建新世界，不继承上一条运行的 HP、MP、Buff、行为树变量或随机状态。
- 相同 DesignCopy 的重复运行使用相同随机种子；系统不会在开始运行时自动换种子。
- 相同 DesignCopy 派生出的结果影响配置完全一致；改变宿主输出频率或 UI 节流不会改变 Tick 历史和摘要。
- 用户可以连续完成至少三条不同时长的运行，记录互不覆盖。
- 已经产生运行记录的副本不会被后续设计修改；继续编辑形成具有新身份的副本。
- 切换到另一个 Simulator 才会替换会话，并明确清理旧会话内存数据。

## 阶段 3：建立逐 Tick 权威记录

本阶段先保证分析事实完整，再让实时 UI 从这些事实中降频投影。完整时间轴 UI 不在本阶段实现。

### 工作项

- [ ] 按 ADR 0029 为 `ModifierSource`、modifier 条目、`AttributeSnapshotLeaf` 和属性快照建立统一 Zod schema，公开类型全部从 schema 推导。
- [ ] StatContainer 直接导出扁平 `AttributePath -> AttributeSnapshotLeaf`，保留当前值、基础值及完整 modifier 来源，不先构造嵌套树。
- [ ] `MemberSerializeData`、`FrameSnapshot` 和 Worker 协议共用该属性契约；快照覆盖分析所需成员，不依赖当前控制器绑定或 UI 选中对象。
- [ ] 调整 Tick 收尾顺序：结算完成后无条件创建一次权威 FrameSnapshot，并写入当前 Worker RunRecord；之后才判断是否向 UI 推送最新帧。
- [ ] 从 `frame_snapshot` 热路径移除 `JSON.stringify/parse` 式清洗；不可结构化克隆的数据在 schema 与构造边界解决。
- [ ] 在权威结算点记录本 Tick 的成功技能和伤害事实，至少保留来源 Member、目标 Member、来源技能或“其他来源”、伤害值及 Tick。
- [ ] 技能释放事实与伤害事实分离：一次多段技能只计一次成功释放，但可以产生多条伤害；当前控制器 `hit` 投影不得作为分析数据源。
- [ ] 分开记录输入轨迹与行动录制：前者包含接纳或拒绝结果，后者只包含 Member FSM 真正接纳的语义行动。
- [ ] 行动接纳事实由实际开始或入队的运行时边界产生，并绑定逻辑 Tick；不得用 MessageRouter 派发成功或主线程时间代替。
- [ ] 行动录制保存相对运行起点的 Tick 偏移和完整语义 payload，不把时间压缩成无间隔的技能 ID 数组。
- [ ] Worker 继续拥有 Tick 数组；主线程会话只持有 RunRecord 身份、`designCopyId`、历史查询句柄和可丢弃的摘要缓存，避免复制一份无界完整历史。
- [ ] 提供按运行、Tick 范围读取的内部接口，为后续时间轴 UI 保留精确到 Tick 的查询能力。

### 正确性验收

- 对任意正常结束的运行，`记录快照数 = 已执行 Tick 数`，Tick 索引连续且无重复。
- 把实时 UI 推送频率分别设为低频和高频，得到的 Worker Tick 记录与最终摘要相同。
- 未绑定控制器或未被 UI 选中的成员仍有完整权威记录。
- 同一多段技能的释放次数、命中次数和总伤害可以被分别验证；非技能伤害进入“其他来源”。
- 重复点击冷却中的技能会出现在输入轨迹中，但只有实际接纳的技能行动进入行动录制。
- 相同录制转换出的流程在相同 DesignCopy 下产生相同计划 Tick；UI 推送频率不影响调度。
- 属性叶节点通过 schema 校验，modifier 来源在跨线程查询后仍完整可辨识。

### 性能验收

- 分别测量快照构造、Worker 留存、历史查询传输和主线程投影成本，不能只记录总帧率。
- 用短、中、长三种人工运行样本记录 CPU、内存增长和 GC 行为；产品不因此强制固定用户测试时长。
- 不得静默丢弃旧 Tick。若出现容量问题，先采用字典化、列式、增量或分块表示；只有测量证明通信或共享最新帧是瓶颈时才评估 SAB。

## 阶段 4：交付第一次可用分析

本阶段把一条 RunRecord 转换为用户能理解的摘要，再允许用户明确选择两条记录进行比较。

### 工作项

- [ ] 建立纯派生分析器，以 RunRecord 关联的 DesignCopy 分析范围与 Tick 记录为输入，不读取当前 Character、正式 Simulator 或 HUD 状态。
- [ ] 摘要可以在当前会话中缓存以支持快速切换，但必须能从 DesignCopy 与 Tick 记录重新生成；缓存不持久化，也不参与结果真实性判断。
- [ ] 单条记录输出验证时长、总伤害和平均 DPS；时长使用模拟时间，DPS 明确定义为 `总伤害 / 验证秒数`。
- [ ] 单条记录输出成功技能顺序，以及每技能释放次数、总伤害、伤害占比和平均单次伤害。
- [ ] 无法归因到技能的伤害显式聚合为“其他来源”，不能丢弃或错误归给最近技能。
- [ ] 会话 UI 列出所有 RunRecord 的开始顺序、时长和核心摘要，用户显式选择恰好两条记录；不得自动使用最近两条。
- [ ] 对比视图同时展示 A、B 原始值、`B - A` 绝对差和相对 A 的百分比差；A 为零时显示不可计算，不生成无穷值。
- [ ] 展示两个 DesignCopy 中的等级与五项基础能力差异，以及成功技能顺序、释放次数和验证时长差异。
- [ ] 操作差异可以展示被拒绝输入，但必须与真正接纳的行动序列分开呈现。
- [ ] 文案只陈述“这两次运行的差异”，不宣称配置变化产生了纯因果收益。

### 验收

- 用户结束一次运行后立即看到时长、总伤害、平均 DPS 和技能过程摘要。
- 用户完成多次运行后可以任意选择两条，交换 A/B 时绝对差和百分比基准按定义变化。
- 摘要中的技能伤害加“其他来源”等于顶层总伤害；各技能伤害占比与总量一致。
- 分析范围外成员的伤害不会混入摘要，但其 Tick 数据仍完整保留。
- 两条时长或技能顺序不同的运行会明确显示这些差异，但仍允许用户继续比较。
- 刷新页面后运行记录消失，持久 Simulator 设计仍然存在并可重新加载。

## 阶段 5：闭合“再设计”路径

本阶段把完整设计副本编辑、比较和差异应用接回活跃 Simulator 会话，完成首个可重复循环。

### 工作项

- [ ] 活跃 Simulator 会话内的 Character 与 Simulator 工作面读取和修改当前 DesignCopy；表单可以复用通用编辑语义，但应用副本之前不得把候选值写入 repository。
- [ ] Character 与 Simulator 编辑不为控制变量法增加等级锁定、点数预算、局部提交草稿或自动修正。
- [ ] 用户可以自由改变等级、五项基础能力及总和；结构合法的数据始终可以进入 DesignCopy、运行和参与记录对比。
- [ ] 对比视图同时展示等级、五项基础能力各自数值和总和差异；未覆盖的其他输入不得被暗示为相同。
- [ ] 当前 DesignCopy 尚未产生运行记录时可以继续编辑；一旦被 RunRecord 引用，任何 Character、Simulator 或 Member 修改都先 copy-on-write 形成新副本。
- [ ] 用户选择“应用为正式设计”时，先展示目标副本相对正式基线的实体与字段差异，再在一个明确写入边界应用全部 insert、update 和 delete。
- [ ] 副本中共享 Character 的变化直接更新原 Character，并自然影响其他引用它的 Simulator；只有显式“复制 Character”操作才创建新实体。
- [ ] 用户可以从一条 RunRecord 显式选择“保存为成员流程”；系统只转换行动录制，不包含被拒绝输入，也不在运行结束时自动覆盖设计。
- [ ] 生成的初始成员流程严格保留行动相对 Tick，不推断技能可用条件或自动重试；指定 Tick 无法执行时只记录该行动失败，其他行动仍在各自原定 Tick 独立尝试。
- [ ] 保存结果先写入新的当前 DesignCopy 的 Player Member `behavior`；只有应用该副本时才进入正式 Simulator 持久化与同步。
- [ ] 返回 Simulator 时保留同一会话、DesignCopy 集合和旧 RunRecord；下一次验证运行当前选择的副本。
- [ ] 对比视图从两条 RunRecord 关联的 DesignCopy 展示设计差异，不读取当前正式数据来重写旧记录。

### 验收

- 改变等级或五项总和不会阻止保存、运行或比较；无效关系、错误类型和不可解析行为仍会在结构边界失败。
- 修改 Character 时，已经结束和正在分析的 RunRecord 保持原始输入与结果不变。
- 新运行使用新 DesignCopy，旧运行继续关联并显示修改前副本。
- 保存行动录制后，当前 Simulator 获得可再次执行的成员流程；未执行保存时，运行不会改变 Simulator 设计。
- 修改配置导致行动在原 Tick 不可用时，新运行明确显示失败；系统不会终止整条流程，也不会改变、跳过或延后其他计划行动。
- 用户可以完整走通“设计 -> 验证 -> 分析 -> 再设计 -> 再验证 -> 选择两条记录对比”，中途无需学习流程编辑器。

## 最终验收脚本

1. 创建或加载一个包含 Player 与木桩的持久 Simulator，确认 Player 使用可选成员流程、木桩使用 Mob 模板固有 AI。
2. 开始验证，按用户自己的节奏手动释放若干技能，手动结束并查看第一条摘要。
3. 在同一会话进入 Character 工作面，自由修改当前 DesignCopy 中的一项或多项设计输入；验收者可以自愿采用同等级、同总点数的能力重分配，但此时正式数据库数据不得改变。
4. 返回 Simulator，确认第一条 RunRecord 及其 DesignCopy 仍存在；从修改后的副本开始第二次干净验证，手动结束。
5. 在记录列表中明确选择第一、第二条，查看时长、总伤害、平均 DPS、技能过程和能力分配差异。
6. 再执行一条故意使用不同时长或技能顺序的运行，确认系统展示操作差异但不阻止选择，也不自动裁定可比性。
7. 从一条手动运行记录显式生成 Player 成员流程，确认它先进入新的 DesignCopy；使用该副本再次运行，核对计划 Tick、实际接纳结果和失败记录，并确认单个行动失败不妨碍后续行动在原定 Tick 尝试。
8. 选择一个候选 DesignCopy 应用为正式设计，确认差异预览与实际 insert、update、delete 一致，共享 Character 的变化按原实体身份写回。
9. 刷新页面，确认会话内 DesignCopy 和 RunRecord 丢失，而刚应用的 Simulator 与 Character 正式设计可从持久数据恢复。

满足以上步骤，并同时满足阶段 3 的逐 Tick 完整性断言，才视为最小验证闭环完成。仅完成 schema、会话类、快照 schema 或摘要组件中的任意一部分都不算闭环。

## 验证策略

- schema / 枚举变化：`pnpm generate`。
- TS / UI 小步变更：`pnpm biome check <相关路径>`。
- 纯分析器、记录目录、设计差异和状态机：增加 Vitest 单元测试，覆盖零时长、零基准、多段技能、其他来源、不同运行选择和非法阶段事件。
- Worker Tick 记录：增加确定性集成测试，断言 Tick 数、索引连续性、UI 节流独立性和多次运行隔离。
- 每个纵向阶段完成后执行对应人工验收；只有影响包入口、Vite、Service Worker 或生成产物引用时才运行完整 `pnpm build`。

## 主要风险与止损点

| 风险 | 观察信号 | 处理顺序 |
|---|---|---|
| 每 Tick 完整属性来源导致内存增长过快 | 长运行中堆内存线性增长过大或频繁 GC | 先测量字段贡献，再做来源字典、列式或增量存储，最后才评估 SAB |
| 技能伤害无法稳定归因 | 聚合总伤害与引擎结算不一致 | 回到伤害结算事实补来源契约，不在分析器按时间猜测 |
| 页面和会话仍有双重生命周期 | 路由切换后运行丢失或控制器重复绑定 | 删除页面级 load/reset 所有权，保留唯一会话入口 |
| 设计修改污染旧结果 | 旧记录显示正式数据库或当前副本的最新值 | RunRecord 只读绑定的 DesignCopy；已运行副本禁止原地修改 |
| 场景解析留下多行为来源 | 引擎运行时仍直接读取 Member 与 Mob 持久字段 | 阶段 0 不通过就不进入会话实现；引擎只消费解析后的单一行为 |
| 范围重新膨胀到流程编辑或历史恢复 | 首次验证依赖新编辑器或版本模型 | 回到非目标清单，相关需求另立后续计划，不阻塞本闭环 |

## 完成定义

- 产品层：真实用户无需开发工具即可完成最终验收脚本。
- 数据层：Simulator 设计可持久化和同步，运行产出不进入数据库。
- 生命周期：普通工作面切换不销毁会话，显式切换方案或结束会话才清理。
- 设计层：DesignCopy 完整定义运行输入；RunRecord 直接关联副本，不存在独立 RunSnapshot 或 RunParameters。
- 写回层：候选副本只有经用户确认后才按实体差异应用为正式设计，完整支持 insert、update 和 delete。
- 引擎层：每 Tick 权威记录完整，实时 UI 节流不影响分析。
- 分析层：顶层与技能摘要守恒，两条记录的输入、操作和结果差异同时可见。
- 领域层：成员流程与 Mob 固有 AI 边界明确，不存在 Character 流程、引擎多来源 fallback、页面级会话或 HUD 分析事实等第二套所有权。
