# 概念说明：交互情境（InteractionContext）是一个模态无关的抽象 UI 模型

- **状态**: 已收敛（独立层，与业务阶段正交，从属于 ADR 0014 的应用三分基准）
- **日期**: 2026-06-15
- **性质**: 澄清「交互情境」是什么、不是什么。它是一个模态无关的抽象 UI 层；视觉层（场景 + UI）是它的投影。
- **相关 ADR**: 0012（意图优先视觉控制）、0014（主业务/配套/瞬时 UI 三分基准）、0016（渲染层架构，下游）
- **相关代码**: `src/machines/intent/types.ts`、`src/machines/intent/visualIntentMachine.ts`、`src/machines/projections/SceneIntentBridge.tsx`、`src/machines/businessPhaseMachine.ts`、`src/routes/(app)/(features)/character/characterPageModel.ts`

---

## 1. 结论先行

**交互情境 = 每个界面（态）对应的一个模态无关抽象 UI 模型，由三要素构成：**

1. **resources**：这个界面操作哪些**种类**的资源
2. **inputScheme**：这个界面允许哪些操作
3. **attentionTargets**：注意力焦点可落在哪些目标

它是一个**独立层**，不是任何东西的派生物。它在视觉与 UI 之上更底层——视觉层（场景 + 面板）是它的**投影器**，假想的听觉层会是另一个投影器。

它**与业务阶段正交**：交互情境覆盖所有界面（含 wiki/profile，它们不在设计-验证-分析里）；业务阶段只在 simulator 这类界面内部调制它。

不要为它建可插拔框架或应用级根容器。它的全部新增物只有一张「每界面（态）的能力声明表」。

---

## 2. 它是什么：模态无关的抽象 UI（Abstract UI）

这个概念在 HCI 工程里有成熟对应——Cameleon 参考框架的分层：

```
任务与概念  →  抽象UI(AUI)  →  具体UI(CUI)  →  最终UI(FUI)
              模态无关          绑定模态         平台实现
              ↑ 交互情境在这层   ↑ 视觉/听觉在这里分叉
```

- **抽象 UI（AUI）**：描述交互单元——有哪些可操作对象、允许哪些操作、注意力落在哪——但**不规定**它是视觉按钮还是语音命令。
- **具体 UI（CUI）**：才绑定模态。同一个 AUI 可投影成视觉（场景 + 面板），也可投影成听觉（音频焦点 + 语音指令）。

> 本项目**不实现听觉**。引入「模态无关」纯粹是作为**划分职责的标尺**，不是要搭多模态框架（违背 YAGNI）。

### 判据：模态无关性测试

> **能在「听觉版本」里说得通的，才属于交互情境；一旦绑定了视觉数值，就留在视觉层。**

验证：
- `intent/types.ts` 的 `Target`（focus 落在 `equipmentSlot:weapon`）—— 听觉版成立（「聚焦武器槽」可以是一句语音）。✅ **属于交互情境（AUI）**
- `SceneIntentBridge` 的 `SLOT_CAMERA_POSES`（`alpha: π/2.6, beta: 1.35, radius: 2.0`）—— 听觉版无意义，是纯视觉相机姿态。❌ **属于视觉层（CUI），绝不进交互情境**

即：交互情境只说「聚焦武器槽」，视觉层才决定「相机怼到 (π/2.6, 1.35, 2.0)」。

## 3. 三要素：AUI 侧 vs 视觉投影侧

| 要素 | AUI 侧（属交互情境，模态无关） | 视觉投影侧（属场景/UI 层） |
|------|------------------------------|---------------------------|
| **resources** | 操作哪些**种类**资源（成员 / 词条） | 渲染成什么 mesh、什么列表组件 |
| **inputScheme** | 允许哪些操作（移动 / 换装 / 浏览） | 绑 WASD / 按钮 / 手势（视觉）、语音（听觉） |
| **attentionTargets** | 焦点可落在哪（weapon 槽 / 某成员） | 相机姿态、高亮描边 |

inputScheme 举例：
- simulator 验证态：移动、放技能、切主控、焦点切换
- simulator 布阵态：放成员、场景配置、焦点切换；**不允许**实时控制
- character 机体设计：换装、换 character、焦点切换；**不允许**移动 / 技能
- wiki / 搜索：浏览、搜索；无世界操作

## 4. 单元：每界面（态）一个，与业务阶段正交

交互情境的键是**界面**，不是业务阶段。证据：wiki / profile 不在设计-验证-分析三态里，却同样有三要素——若交互情境挂在业务阶段上，它们就无处安放。所以：

- 多数界面（wiki / profile / character）：**界面 = 一个交互情境**（无业务阶段）。
- 主业务界面（simulator）：**界面 × 业务阶段 = 多个交互情境**（布阵态、实时态各一）。业务阶段在此**调制** inputScheme。

```
交互情境层（统一管理所有界面，含 wiki/profile）
   每界面（态） → (resources, inputScheme, attentionTargets)
        │
        └── 正交：业务阶段（design-verify-analyze）
              仅在 simulator 类界面"内部"调制 inputScheme，
              对 wiki/profile 等界面 N/A
```

> 因此「交互情境」与 `businessPhaseMachine`（设计-验证-分析）是**正交并列**，不是从属关系。业务阶段只覆盖主业务的若干界面；交互情境覆盖全部界面。

## 5. 现成样本：注意力链已是 AUI→CUI 分离的活样本

ADR 0012 的注意力链，已经是三要素中「attentionTargets」这一个要素的完整 AUI/CUI 分离实现：

```
visualIntentMachine        SceneIntentBridge          相机/高亮
(产出"焦点应落在 X"   →    (投影器：语义→视觉)   →    (视觉 CUI)
 = AUI 的 attention)        AUI → 视觉CUI
```

意图机产出**模态无关**的注意力意图（AUI），桥把它投影成视觉（CUI），相机姿态是 CUI 参数。

> 要做的，只是把这个**已验证**的分离模式，从「注意力」一个要素，推广到三要素全体（resources、inputScheme 同样走「语义声明 → 投影器 → 视觉」）。这不是发明新架构，是把现有成功模式提升为通则。

## 6. 世界内容（渲染什么）与交互情境的关系

注意区分「资源的种类声明」与「资源的数据与延续」：

- **resources（种类）属于交互情境**：声明这个界面操作「成员」还是「词条」。
- **资源的实际数据与跨界面延续，由 simulator 上下文管，与交互情境正交**：

> **世界内容延续 / 重建的开关 = 当前携带的 simulator 上下文有没有变，与「切了哪个界面/情境」无关。**

- 从 simulator 设计走到 character 机体设计：界面变了（inputScheme/attention 变），但 simulator 上下文没变 → **世界内容延续**。
- 选木桩 / 打开分享的 character URL：simulator 上下文变了 → **世界内容重建**。

交互情境切换频繁，世界内容延续是常态——这就是「世界内容跨页面延续」的精确定义。

### character 资源定性 = 丙（ADR 0014 代价段未决项的回填）

机体是 **account 级可复用资源**（资源身份）**且** 从场景设计进入时**携带「为当前 simulator 编辑」的会话上下文**；编辑结果既存入资源库，又回填到该场景。两个事实正交，同时成立。

- 资源身份 → 进 `character` 表、可寻址 URL（0014 A 类资源）。
- 会话上下文（为哪个 simulator 的哪个 member 编辑）→ 瞬时态，由进入时携带，不进 character 资源本身、不进 URL。
- 现状 `characterPageModel.ts:createPreviewScenario` 无条件造独立预览场景（只实现「独立资源」一面）；「携带 simulator 上下文延续」是**目标态，待实现**。

## 8. 纪律：防模态泄漏

本模型唯一的失败模式是**模态泄漏**——把相机距离、颜色、组件名等视觉细节塞进交互情境。一旦发生，AUI 退化成「换皮的视觉层」，抽象白做。守则始终是第 2 节那句判据：**说不出听觉版本的，不许进交互情境。**

## 9. 附录：被否决的探索（防止重走弯路）

本文更早的两版分别错在：

**版本一**：提出「四正交维度（Subject × Phase × Surface × Attention）+ ApplicationContext 应用情境根」。否决理由：
- **Phase 不通用**：设计-验证-分析只对模拟业务有意义，character 把它压扁、wiki/搜索根本没有。不通用 → 不是维度。
- **Surface 不是维度**：URL 只保留资源定位能力，不表示状态和关系。
- **Subject 不存在平行切换**：simulator/character/wiki 是业务树里的**嵌套包含**关系，不是平行主体。
- **根容器是重复发明**：ADR 0014 三分法已是那个「应用级模型」。

**版本二**：把交互情境降格为「业务树节点的派生物」（交互情境 = 节点 + inputScheme + attention）。否决理由：
- 这把交互情境**挂在了业务阶段上**，导致 wiki/profile（无业务阶段）无处安放。
- 正确关系是**正交**：交互情境按界面键入、覆盖全部界面；业务阶段只调制其中主业务的若干界面。

教训：
- 当一个「维度」对多数取值是 N/A 时，它不是维度，是某一类节点的内部结构。
- 交互情境的键是**界面（态）**，不是业务阶段；二者正交。
- 交互情境是**模态无关的独立层**，不是任何结构的派生物。
