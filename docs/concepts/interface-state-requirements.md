# 跨模态界面状态需求

- **状态**: 当前需求基线
- **日期**: 2026-07-10
- **相关**: ADR 0021、`docs/concepts/task-model.md`

## 问题

2D UI、3D 场景、键盘等输入可以表达同一个用户意图。如果各模态分别保存“当前在做什么”，再通过互相通知保持同步，系统会出现重复事实源、循环更新和生命周期漂移。

真正需要统一的不是 2D 与 3D 的全部状态，而是它们共享的**交互语义**：用户当前处于哪个有限交互模式、正在作用于哪个对象，以及此时哪些意图合法。

## 需求

1. **单一事实源**：跨模态共享的交互状态只能由一个应用级模型持有，2D UI 与 3D 场景不得各自保存副本。
2. **语义输入**：UI 操作、场景拾取和快捷键先转换为模态无关的语义事件，再进入状态模型；输入源不能直接命令另一个投影。
3. **显式模式**：有限交互模式必须能从状态路径直接识别；合法意图由状态转移和 guard 表达。
4. **动态上下文**：角色 ID、装备槽、成员 ID 等运行时目标放在状态上下文中，不为每个实体扩展状态结构。
5. **可恢复投影**：任一投影重新挂载后，只依赖当前状态快照和领域数据就能恢复表现，不要求另一投影重放操作。
6. **单向投影**：2D UI 与 3D 场景分别消费同一快照。相机补间、overlay 动画和资源加载完成等具体表现状态不反向推进共享交互状态。
7. **可追踪性**：一次用户意图应能沿“输入适配 -> 语义事件 -> 状态转移 -> 多端投影”追踪，不引入第二套 capability 或注意力状态。
8. **投影有据**：AUI 语义不隐含具体视觉参数。只有资产或渲染契约提供明确映射时，投影才能改变相机、动画或材质；缺少映射时保持现状。

## 状态边界

统一管理的内容仅限于多个模态共同需要解释的行为状态，例如当前交互空间、检查或编辑模式，以及这些模式所需的动态目标标识。

以下内容继续由原有模块管理：

- 角色、装备、场景成员和模拟结果等领域数据；
- `SceneRuntime`、引擎 worker、场景资源和路由生命周期；
- 相机姿态、补间进度、模型加载状态和 overlay handle；
- 只影响单个组件的表单草稿、hover、展开项等局部 UI 状态。

## 非目标

- 不定义抽象组件树、布局或控件形态；
- 不从 AUI 自动生成现有 2D UI 或 3D 场景；
- 不在短期内采用完整的自上而下 MDD 流程；
- 不把所有应用状态收编进一台全局状态机。

## 最小验证

首个纵向切片应证明：同一装备意图可由 UI 和 3D 输入产生同一种语义事件；当前快照足以恢复 sheet 和有资产 metadata 支撑的选中高亮；非法事件不会改变状态；缺少视角契约时相机保持不变；任何动画完成回执都不是状态继续前进的前提。
