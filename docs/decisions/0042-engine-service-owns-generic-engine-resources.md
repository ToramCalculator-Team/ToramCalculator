# 0042 - EngineService 拥有通用引擎资源

- **状态**: Accepted
- **日期**: 2026-07-14
- **决策层**: 跨层（应用组合 / 引擎基础设施 / Session 生命周期）
- **相关 ADR**: Supersedes 0015；Depends on 0024

## 决策问题

应用既有跨路由持续的实时模拟，也有执行完成后结束的一次性模拟任务。系统需要决定物理引擎资源由具名业务服务、各 Session 或通用 EngineService 拥有，以及业务使用生命周期和依赖装配由谁负责。

## 决策驱动

- 引擎基础设施不能编码 Character、Simulator 或业务阶段。
- Session 负责业务使用生命周期，但不能关心 Worker、通信通道和故障替换。
- 物理资源必须有唯一所有者，业务能力必须通过受限边界暴露。
- 应用级服务必须在唯一显式入口构造和装配，不能被组件惰性查询或启动。

## 候选方案

### A. EngineService 预建并公开具名业务引擎

- 优点：业务页面可以按名称直接取得预热资源。
- 缺点：基础设施认识业务角色，资源生命周期不再由实际使用者表达。

### B. Session 直接创建和管理物理引擎资源

- 优点：业务所有者直接控制资源。
- 缺点：Session 必须理解 Worker、transport、协议校验和故障恢复，基础设施职责进入业务编排。

### C. EngineService 拥有资源，Session 拥有受限使用句柄

- 优点：物理资源与业务使用生命周期各有清晰所有者，基础设施保持业务无关。
- 缺点：需要显式句柄生命周期和应用组合根装配。

## 决议

选择 C：**EngineService 拥有物理引擎资源和通用基础设施；Session 拥有业务使用生命周期和受限实时句柄；bootstrap 组合根是唯一构造与依赖装配入口。**

- EngineService 不命名或登记具体业务引擎，也不保存 Session、路由、阶段或领域状态。
- Session 通过受限能力编排业务流程，不直接创建 Worker、通信对象或资源池，也不解释 transport envelope。
- 句柄表达 Session 对实时引擎的使用权和关闭责任；底层物理资源、通信与故障恢复仍由 EngineService 拥有。
- 组合根以显式构造参数装配 EngineService 与 Session，不建立组件级服务定位器或隐式全局单例。
- CUI 只消费对应 Session 的输入边界和只读投影，不能绕过 Session 取得 EngineService。
- 生命周期启动必须有唯一显式入口，读取能力不能惰性创建或修复应用级资源。

## 代价

组合根需要显式装配基础设施和多个 Session，测试也需要提供对应能力替身。实时句柄增加资源身份与关闭协议，必须保证 Session 停止、资源故障和应用关闭时所有权一致。

## 重新评估条件

- 动态 Session 数量需要容量限制、休眠或抢占，现有句柄生命周期无法表达。
- 实时引擎与池化任务的状态和调度语义完全收敛，可以由同一种资源管理器表达。
- 应用引入正式依赖注入容器，并能继续保证唯一组合根、静态依赖和禁止组件级服务定位。
