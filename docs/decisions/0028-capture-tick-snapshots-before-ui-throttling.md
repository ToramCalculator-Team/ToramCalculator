# 0028 - 每 Tick 捕获权威分析快照并在捕获后节流 UI

- **状态**: Accepted
- **日期**: 2026-07-11
- **决策层**: 跨层（引擎 / 通信 / 分析数据 / UI）
- **相关代码**: `src/lib/engine/core/GameEngine.ts`、`src/lib/engine/core/types.ts`、`src/lib/engine/core/thread/Simulation.worker.ts`
- **相关 ADR**: Depends on 0024

## 背景

运行记录未来需要支持精确到逻辑 Tick 的时间轴分析。当前引擎只在满足 UI 推送频率时创建 `FrameSnapshot`，未发送的 Tick 不存在快照；同一发送路径还会先对深层对象执行 JSON 序列化清洗。若继续把捕获频率与 UI 消费频率绑定，后续无法从运行记录恢复完整时间轴；若把每 Tick 的完整历史都立即传到主线程，又会让分析保真度与线程通信实现耦合。

## 决策驱动

- 分析记录必须独立于 UI 刷新频率，精确保留每个逻辑 Tick。
- Worker 应拥有权威运行事实，主线程只接收可替换的查询或实时投影。
- 存储和传输实现需要能够独立优化，而不改变分析语义。

## 候选方案

### A. 继续按 UI 频率创建并发送快照

- 优点：对象创建、序列化和主线程更新次数较少，现有实现改动小。
- 缺点：被节流的 Tick 永久缺失，无法形成权威运行记录；UI 性能策略反向决定分析数据精度。

### B. 每 Tick 创建并立即发送完整快照

- 优点：主线程随时拥有完整历史，分析 UI 接入直接。
- 缺点：每 Tick 深层对象分配、JSON 清洗、结构化克隆和主线程消费会形成固定成本；用户自定运行时长使历史传输没有固定上限。

### C. 每 Tick 捕获到 Worker 内记录，UI 在捕获后独立节流

- 优点：分析记录不丢 Tick，UI 仍可按自身刷新需求消费最新投影；存储与通信优化可以在不改变快照语义的前提下演进。
- 缺点：Worker 需要管理随运行增长的历史记录，并提供分块查询、结束时移交或持久化能力。

## 决议

选 C：**逻辑引擎在每个 Tick 完成后捕获权威 `FrameSnapshot` 并写入当前运行记录；实时 UI 只消费该事实的节流投影，节流不得阻止源快照生成。**

1. `FrameSnapshot` 的分析语义以逻辑 Tick 为边界，由 Worker 在本 Tick 所有结算完成后统一捕获。
2. 完整历史首先由 Worker 内的运行记录拥有，不要求每 Tick 都跨线程发送到主线程。
3. 实时 UI 可以只接收最新快照、降低发送频率或覆盖过期帧，但不得改变已经捕获的运行记录。
4. 权威快照的属性形状由 ADR 0029 独立定义；本决议只要求捕获对象从类型和 schema 层保证可结构化克隆，不经 JSON stringify/parse 清洗。
5. SAB 只是一种可替换的传输实现。先测量每 Tick 捕获、Worker 内留存、分块传输和 UI 投影的真实成本；只有确认结构化克隆或 GC 构成瓶颈后，再决定 SAB 的布局与适用通道。
6. 若采用 SAB，优先服务实时“最新帧”共享视图；不可丢失且时长不定的历史记录仍需独立的有界分块或持久化策略。

## 代价

每 Tick 捕获完整属性及来源会增加 Worker 内 CPU、内存和记录增长速度，特别是大量来源元数据在相邻 Tick 间重复时。运行记录必须解决容量、分块与版本迁移问题；显式快照 schema 也会使协议演进需要正式兼容策略。若实测表明完整快照无法满足长时间实操，允许在保持每 Tick 语义和查询结果不变的前提下改用列式、字典化、增量或分段压缩表示，但不能重新用 UI 节流丢弃分析 Tick。

## 重新评估条件

- 即使采用字典化、增量或分块表示，每 Tick 权威记录仍无法满足目标运行时长或内存上限。
- 引擎改为跨进程或远程执行，Worker 不再适合作为运行期间的权威记录所有者。
- 产品明确放弃 Tick 级分析，只保留语义不同的低频采样分析。

## 影响范围

- `GameEngine` 的 Tick 收尾顺序调整为先捕获并交给运行记录，再进行实时 UI 投影。
- `FrameSnapshot` 从宽泛 UI 数据对象收紧为显式、可版本化的分析契约；实时 UI 可从中派生更小的视图。
- Worker 通信协议需要区分实时最新帧和历史记录查询/移交，二者不再共用同一频率语义。
- 性能验收必须分别测量快照捕获、Worker 留存、跨线程传输和主线程投影，不能只观察网络式“带宽”。
