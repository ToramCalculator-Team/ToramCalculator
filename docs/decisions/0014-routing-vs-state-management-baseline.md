# 0014 - 路由与状态管理：主业务 vs 配套服务的分类基准

- **状态**: Accepted
- **日期**: 2026-06-12
- **决策层**: 跨层（应用层状态架构）
- **相关代码**: `src/routes/(app)/`、`src/machines/businessPhaseMachine.ts`、`src/store.ts`
- **相关 ADR**: 0012、0013、0015

## 背景

路由按资源组织，业务状态机按设计、验证、分析阶段组织。两者不是同一维度：资源需要可分享和刷新恢复，阶段可能只是同一资源的不同视图，局部面板又完全不需要 URL。需要统一判断哪些状态进入路由。

## 候选方案

### A. URL 直接等同业务阶段

- 优点：阶段一眼可见。
- 缺点：把瞬时视图当成资源地址，也无法自然容纳 wiki、character 等配套功能。

### B. 路由和状态机各自演进

- 优点：无迁移成本。
- 缺点：状态权属继续模糊，路由和业务逻辑会互相驱动。

### C. 按可寻址性分类，状态模型为真值源

- 优点：URL 只承载需要分享、历史记录或刷新恢复的状态；主业务、配套资源和瞬时 UI 边界明确。
- 缺点：需要持续维护 URL 只是状态投影的纪律，具体归类依赖产品语义。

## 决议

选 C：**状态模型是应用真值源，URL 只投影其中需要被寻址的子集。**

分类基准：

1. 资源身份进入 path。
2. 资源视图阶段只有在需要分享、历史或刷新恢复时才进入 path/query。
3. 瞬时 UI 不进入 URL。
4. 主业务 simulator 可以拥有阶段机；配套资源路由和瞬时 UI 不因此被强行纳入同一生命周期。
5. URL 变化必须被状态模型消费和校验，不能绕过业务约束直接驱动逻辑。

character 同时具有两种身份：它是可复用、可寻址的账号资源；从 simulator 进入时也可以携带“为当前场景编辑”的瞬时会话上下文。资源身份进入 URL，会话上下文不进入 URL。

## 代价

- 状态是否需要寻址依赖产品语义，不能只靠技术规则判断。
- analyze 只有在结果具备可恢复归属后才能支持深链接。
- 若路由直接承担业务转移，本决议的单向关系会被破坏。

## 影响范围

- simulator 的业务阶段由状态机管理，路由只表达被选中的可寻址部分。
- character/wiki 等资源保持真实 URL；setting、注意力强度和局部 panel 状态保持内存态。
- 跨页面世界内容是否延续由携带的业务上下文决定，不由页面名称决定。
