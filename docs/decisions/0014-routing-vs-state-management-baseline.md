# 0014 - 路由与状态管理：主业务 vs 配套服务的分类基准

- **状态**: Accepted
- **日期**: 2026-06-12
- **决策层**: 跨层（应用层状态架构）
- **相关代码**: `src/routes/(app)/`、`src/machines/businessPhaseMachine.ts`、`src/store.ts`
- **相关 ADR**: 0012、0013、0015

## 背景

应用的状态建模（业务阶段机：设计→验证→分析）与现有路由结构（wiki / character 配置 / simulator）存在交叉与冲突：

- 路由按「操作什么对象」组织（wiki=数据、character=机体、simulator=模拟场景）。
- 状态机按「处于什么工作阶段」组织（designing / simulating / analyzing）。

二者是正交的两个维度，强行让一方映射另一方会扭曲建模。具体表现：
- 「验证」阶段散落在多处——character 页内部有属性/技能预览（`panelMode`），simulator 页有实时战斗模拟。
- 「分析」阶段在路由里没有对应物。
- 把业务阶段（瞬时视图状态）直接当作顶层并列路由，或反过来让路由驱动业务逻辑，都会导致状态与 URL 互相打架。

需要先确立「哪些状态进 URL、哪些不进」「路由与状态机谁是真值源」的基准规则，否则每个页面都要重新讨论一次归类。

## 候选方案

### A. 路由即业务阶段（URL 直接 = designing/simulating/analyzing）

把三个业务阶段提升为顶层并列路由，导航跟随阶段切换。

- 优点：URL 直接反映业务流，导航语义直白。
- 缺点：wiki/character 是横切的只读参考与数据配套，根本不属于这个工作流，硬塞进阶段体系会扭曲；阶段是「同一个资源的不同视图」而非不同资源，并列路由割裂了「它们共享同一份数据」的事实；把瞬时视图状态当资源标识，违反「URL = 可寻址资源」。

### B. 维持现状（路由按对象，状态机按阶段，各管各）

不调整，接受两套维度并存。

- 优点：零改动。
- 缺点：冲突不消除——business phase 机无处安放、character 页被迫感知不属于它的阶段概念；「分析」无归属；新人无法从结构判断一个状态该放哪。

### C. 三分基准：主业务（有阶段）/ 配套服务（资源）/ 瞬时 UI（不进 URL），状态机为真值源

确立分类理念而非具体路由表：
- **状态按「是否需要被 URL 寻址」分三类**：资源标识（进 path）、资源的视图阶段（视分享需求进嵌套 path 或 query）、瞬时 UI（绝不进 URL）。
- **判据**：一个状态是否需要 URL 的三种能力——可分享、可被历史记录前进后退、可刷新恢复。三者都不需要就不进 URL。
- **应用结构三分**：主业务（simulator，有生命周期与 phase 机）/ 配套服务（character 配置、装备配置、wiki，无生命周期、导航可达、面向数据）/ 瞬时 UI（setting 浮层、注意力强度、panelMode）。
- **真值源单向**：状态机/store 是唯一真值源，URL 是其「可寻址子集」的投影。状态变化 → 命令式 `navigate()`；URL 变化 → `ROUTE_CHANGED` 事件 → 机器消费并校验（可拒绝非法深链接并降级）。

- 优点：每个页面归类变成「套判据」而非「重新辩论」；business phase 机归属收窄到 simulator 一支，其它路由不感知 phase，冲突消失；wiki/character 作为资源型路由保持独立且可寻址；与 0013 的 deps 注入 / ROUTE_CHANGED 范式一致。
- 缺点：需要团队接受「URL 是状态投影而非导航菜单」这一理念，否则容易退回「路由驱动业务」的旧习惯；判据需要产品语义输入（如 character 是否为可复用资源）才能完成具体归类，无法纯技术决定。

## 决议

选 C。

理由：

1. **路由的本质是「被提升到 URL 的那部分应用状态」**，不是页面切换器。因此「路由 vs 状态管理」是同一个问题：全部状态里哪些需要被寻址。一个状态进不进 URL，只问它需不需要被分享 / 历史记录 / 刷新恢复。
2. **状态机是真值源，URL 是其可寻址子集的投影，关系单向。** 禁止「路由驱动业务逻辑」——那会让业务规则散落进路由守卫、让深链接进入非法状态。机器永远可以拒绝非法 URL 并降级（呼应 0013 决议第 3 条的降级精神）。
3. **应用三分：主业务 / 配套服务 / 瞬时 UI。** 核心业务对象是「模拟」（simulator）；character 配置、装备配置是「造轮子」——面向数据、与某次模拟无关的配套服务；wiki 是只读参考；三者正交，本就不该共用一套结构。最初的「冲突」根源正是把配套服务（character）误当成与主业务（simulator）平级、并让业务流缺失 phase 层。
4. **business phase 机只绑定 simulator 一支**（`/simulator/:id/{design|simulate|analyze}`），配套服务路由与 phase 无关。这消除了「character 页要不要感知 phase」的矛盾——它根本不感知。
5. **配套服务用导航跳转 + 真 URL（A 类资源），不是 setting 式浮层。** 形态差异源于状态性质差异：setting 是瞬时 UI（C 类，浮层 + store 信号）；character/wiki 是可寻址资源（A 类，真路由）。

## 代价

- **判据依赖产品语义**：character 究竟是「account 级可复用资源库」还是「某次模拟的一次性草稿」，决定它是独立资源路由还是 simulator 子资源。本 ADR 给出判据但不替产品定性；定性变化会改变 character 的路由归属。
- **「分析模拟结果」的产物持久化悬空**：当前 `statistic` 模型仅记录访问元数据（createdAt/usageTimestamps 等），不存模拟结果；simulator 也无结果字段。analyze 作为 simulator 的全局阶段，其分析对象目前只是运行时内存态，无持久化归属。路由推进到 analyze 时必须先解决产物归属，否则该阶段不可深链接/刷新恢复。
- **理念落地依赖纪律**：「URL 是投影、机器是真值」需要持续维护单向关系；一旦有人图省事让路由直接驱动逻辑，本 ADR 的收益会被侵蚀。

## 影响范围

- 代码层面：`businessPhaseMachine` 的消费绑定到 `/simulator/:id` 一支，通过 deps 注入 `navigate()` 与消费 `ROUTE_CHANGED`；配套服务路由（character/wiki）保持资源型 URL，不引入 phase；瞬时 UI（setting/注意力/panelMode）继续用 store/signal/状态机内存，禁止进 URL。
- 文档层面：与 0012（意图优先视觉控制）、0013（注意力机阶段无关）、0015（运行时资源架构）共同构成应用层基准。
- 迁移：本 ADR 只确立基准，不要求立即重排路由。后续路由改造（character 内部 panelMode、simulator 子路由化、analyze 产物归属）按本判据分步推进，各自可独立立项。

## 参考

- 与用户的架构讨论（2026-06-12）：经五轮澄清收敛——核心业务对象是「模拟」、character/wiki/装备是「造轮子」配套服务、setting 范式（store 信号 + Show）、URL=可寻址资源。
- ADR 0013：注意力机阶段无关、ROUTE_CHANGED / deps 注入 navigate 范式、PHASE_CHANGED 降级。
- REST / web 平台基本原则：URL 标识可寻址资源，浏览器历史与刷新恢复语义。

## 补遗（2026-06-15）：character 资源定性 = 丙

代价段曾留未决项：「character 究竟是 account 级可复用资源库，还是某次模拟的一次性草稿」。现定性为**丙（两者皆是）**：

- character 配的机体是 **account 级可复用资源**（资源身份）——进 `character` 表、可寻址 URL（A 类资源），可被任意 simulator 引用。
- **且** 从场景设计环节进入 character 页时，**携带「为当前 simulator 的某个 member 编辑」的会话上下文**——编辑结果既存入资源库，又回填到该 simulator 场景。
- 两个事实正交、同时成立：资源身份进 URL；会话上下文是瞬时态（C 类），由进入时携带，**不进** character 资源本身、**不进** URL。

层级澄清：character 页是「主业务 → 设计 → 初始场景设计 → 机体设计」的**内部特化工具**，因复杂度高 + 需分享而独立成 URL，但**未脱离主业务设计环节**。它在配套服务里的归类（决议#3）是就「可寻址资源型路由」而言；就业务语义而言，它是设计环节的嵌套节点。二者不矛盾。

衍生影响：
- 「世界内容跨页面延续」的精确定义 = **携带的 simulator 上下文未变则世界内容延续，变则重建**（与「在哪个页面」无关）。
- 现状 `characterPageModel.ts:createPreviewScenario` 无条件造独立预览场景，仅实现「独立资源」一面；「携带 simulator 上下文延续」为目标态、待实现。
- 详见 `docs/concepts/interaction-context.md`（交互情境为轻派生概念）与 ADR 0016（渲染层只吃 worldResources）。
