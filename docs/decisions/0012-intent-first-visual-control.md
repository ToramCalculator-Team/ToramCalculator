# 0012 - 意图优先的视觉控制：意图层作为 UI 与场景的单一事实源

- **状态**: Accepted
- **日期**: 2026-06-09
- **决策层**: 跨层（应用层 UI / 渲染 / 状态）
- **相关代码**: `src/store.ts`、`src/lib/engine/render/SceneRuntimeCore.tsx`、`src/lib/engine/render/sceneStateMachine.ts`、`src/routes/(app)/(features)/character/[characterId].tsx`（现状触点）；`src/machines/`（规划落点）
- **相关 ADR**: 0009（常驻渲染运行时）、0013

## 背景

应用需要 UI 与 3D 场景配合表达同一交互语义：例如进入装备编辑时，UI 打开编辑面板，同时场景相机对焦到角色对应装备部位。该语义有多个等价输入源——从 UI 菜单选择、点击角色模型上的部位、将部位置于视角中心（未来还可能有快捷键等）。

现状中不存在承载这类语义的状态层：

- `store.ts` 只有数据型状态（session / settings / sw）和零散 dialog 布尔值，无跨页面的"模式/意图"状态。
- 角色页的面板切换是路由本地 signal（`panelMode`），UI 卸载即丢失，场景无法感知。
- 场景侧相机已有命令面（`ThirdPersonCameraController`）和过渡状态机（`sceneStateMachine`），但只被场景内部生命周期驱动，与 UI 无共享语义。

必须决定：这类"UI 与场景共同表现的交互状态"由谁拥有、如何流动。

## 候选方案

### A. 双向同步（UI 与场景互发通知）

UI 状态变化时通知场景执行相机动作，场景输入（拾取）时回调 UI 更新面板。两侧各自持有自己的状态副本。

- 优点：不引入新层，每个功能点的实现路径最短；贴近"事件回调"直觉。
- 缺点：状态权属不清，两份副本必然漂移；回声问题（A 通知 B，B 的变化又通知 A）需要逐处手工抑制；每新增一个输入源或投影端，同步边数平方增长；交互语义散落在回调链里，无法独立测试。

### B. UI 为主（UI 状态驱动场景，场景输入注入 UI）

把交互状态放在 UI 层（页面 signal 或 store.ts），场景作为 UI 的渲染附属，场景输入通过 props/回调写回 UI 状态。

- 优点：符合现有 SolidJS 心智，改动集中在 UI 侧；单一事实源成立（在 UI 内）。
- 缺点：状态生命周期被路由绑架——跨页面的注意力连续性无法表达；场景输入必须了解 UI 组件树结构才能写回；"场景不是 UI 的下属而是平级表达面"这一产品语义被架构否定；交互语义与渲染框架耦合，无法脱离 Solid 测试。

### C. 意图层作为上层单一事实源，UI 与场景为纯投影

新增应用级意图状态机（XState actor）。任何模态的输入都归一化为语义意图事件（提案）送入意图机；意图机是唯一写者；UI 与场景作为两个对称、无特权的投影，各自从快照纯函数派生表现并自行收敛。

概念模型把意图正交分解为三个维度，避免事件枚举爆炸：

- **Target（目标）**：语义路径，如 `{ kind: "equipment"; slot }`、`{ kind: "timelineEvent"; eventId }`，可随产品扩展；
- **Operation（操作强度）**：`inspect`（只读检视）/ `engage`（主动操纵）；
- **注意力生命周期**（状态结构，目标无关）：

```
atRest ──ATTEND(target)──▶ acquiring ──settle──▶ attending ──ENGAGE──▶ engaging
  ▲                           │  ▲                    │                   │
  └──────── releasing ◀───────┘  └──新ATTEND自打断─────┴───RELEASE─────────┘
```

投影契约：`π_ui(s)` 即时渲染，`π_scene(s)` 以补间动画收敛并回执 `SETTLED`；回执是建议性的（驱动 `acquiring → attending`），不是权威的（新意图不等回执直接抢占）。

- 优点：单向数据流根除双向同步与回声问题；输入源对称可扩展（新模态只需一个输入适配器）；交互语义集中且可脱离 UI/babylon 独立测试；注意力状态生命周期独立于路由。
- 缺点：新增一层间接性，简单交互也要绕经意图层；"哪些状态进意图机、哪些留本地"需要持续判断；投影回执协议给场景实现增加义务。

## 决议

选 C。

理由：

1. **双输入源是产品需求而非偶然**——"UI 选择"与"场景点击/视角驻留"必须产生同一语义。只有把语义提到两者之上，输入对称性才是结构保证而不是逐处约定。
2. **场景与 UI 是平级表达面**。方案 B 把场景降格为 UI 附属，与产品语义矛盾；方案 A 没有事实源。
3. **正交分解抑制状态爆炸**：注意力生命周期只有五个状态且目标无关，功能增长只扩展 Target 词汇与 capability 表（见 ADR 0013），不增加状态结构。
4. 与项目既有范式同构：deps 注入隔离副作用（仿 `sceneStateMachine`）、actor 化（仿 Member），不引入新范式。

附五条不变量作为实现护栏：

1. **单一写者**：只有意图机能改变意图状态。
2. **投影无状态**（对意图而言）：删除任一投影再重建，从快照可完整恢复表现；投影内部瞬态（补间进度）永不回写。
3. **输入与投影不直连**：UI 手势不得直接命令相机，必须绕经意图层。
4. **用户主权**：新意图无条件抢占进行中的过渡，永不排队等动画。
5. **回执建议性**：`SETTLED` 驱动状态推进，但缺失/迟到不阻塞意图层。

## 代价

- **间接成本**：最简单的"点按钮开面板"也要走 事件 → 机器 → 快照 → 派生 的全链路。对纯 UI 本地状态（如 AttrPreview/SkillPreview 切换）这是过度设计，因此必须维持"意图态 vs 本地瞬态"的边界判断——这条边界没有机械规则，判断失误会导致意图机膨胀成第二个全局 store（这正是我们明确反对的"XState 当 store 用"反模式）。
- **投影义务**：场景投影必须实现收敛 + 回执 + 可取消三件套，比"被动收命令"复杂。
- 如果产品最终只有"装备编辑"这一个 UI-场景配合点，本架构的抽象成本（Target 泛化、注意力生命周期）将得不偿失。我们押注的是配合点会随三阶段（设计/验证/分析）持续增多。

## 影响范围

- 代码层面：新增 `src/machines/`（意图机、AppActorContext provider）；`SceneRuntimeCore.tsx` 增加投影订阅与拾取输入适配器；角色页 `panelMode` 中装备编辑相关部分迁入意图机；`store.ts` 不迁移（数据型状态原样保留）；`characterPageModel` 的命令流不动，意图机只管"在编辑哪个槽"，不管数据修改。
- 文档层面：本 ADR 与 0013 共同构成应用层状态架构基准；落地后在 `SceneRuntimeCore.tsx` 顶部加引用注释。
- 迁移：首版输入源只做 UI 选择 + 场景点击两种；"视角驻留即聚焦"留待第二迭代（事件协议已留 `source` 字段）。
