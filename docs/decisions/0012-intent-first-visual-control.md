# 0012 - 意图优先的视觉控制：意图层作为 UI 与场景的单一事实源

- **状态**: Superseded by 0021
- **日期**: 2026-06-09
- **决策层**: 跨层（应用层 UI / 渲染 / 状态）
- **相关代码**: `src/machines/intent/visualIntentMachine.ts`、`src/machines/projections/SceneIntentBridge.tsx`
- **相关 ADR**: 0009、0013、0021

## 背景

UI 选择、场景点击和未来快捷键可能表达同一个交互语义，例如“检查角色的武器槽”。如果 UI 与场景分别保存状态并互相通知，会产生重复事实源、回声和跨页面漂移。需要确定共享交互语义由谁拥有。

## 候选方案

### A. UI 与场景双向同步

- 优点：局部实现直接。
- 缺点：两份状态需要持续对齐，输入源和投影端增加后同步关系迅速增长。

### B. UI 状态驱动场景

- 优点：符合现有组件心智，单一事实源明确。
- 缺点：共享语义被路由和 UI 组件生命周期绑定，场景输入必须反向了解 UI。

### C. 独立意图层，UI 与场景都是投影

- 优点：输入源统一为语义事件；状态可跨投影存活；UI 与场景互不直连。
- 缺点：增加中间层，并要求持续区分共享意图与局部 UI 状态。

## 决议

选 C：**跨 UI 与场景的交互语义由应用级意图状态机唯一持有，所有输入归一化为语义事件，UI 与场景只投影快照。**

当前模型以 `Target`、`Operation` 和注意力生命周期表达意图。必须遵守以下不变量：

1. 只有意图机可以修改共享意图状态。
2. 删除任一投影后，可仅凭当前快照恢复表现。
3. 输入适配器不能直接命令另一投影。
4. 新用户意图可以抢占进行中的视觉过渡，不排队等待动画。
5. 场景完成回执只用于建议性推进，迟到或缺失不能阻塞新意图。

纯 UI 局部状态和领域数据不进入意图机，避免把 XState 当成第二个全局 store。

## 代价

- 简单交互也增加一次事件和投影间接层。
- 场景投影需要处理收敛、取消和迟到回执。
- 如果 UI/场景协同点很少，这层抽象的维护成本可能高于收益。

## 影响范围

- UI 与场景共享语义必须通过意图机通信，不能互相调用。
- `store.ts` 继续保存数据型状态；页面局部瞬态保持本地管理。
- 新输入模态只增加语义事件适配器，不新增事实源。
