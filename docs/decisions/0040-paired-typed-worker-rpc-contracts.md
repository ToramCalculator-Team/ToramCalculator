# 0040 - Worker RPC 使用成对类型化契约

- **状态**: Accepted
- **日期**: 2026-07-14
- **决策层**: 通信（主线程 / Worker / 引擎协议）

## 决策问题

跨线程消息在运行时是不可信数据，而实时与池化引擎调用需要共享相同的请求、响应和错误语义。系统需要决定由通用 Worker transport 理解业务结果，还是由 Engine 协议边界统一恢复并校验业务类型。

## 决策驱动

- 每类 Engine 请求必须唯一对应其成功响应。
- 编译期类型不能替代跨线程边界的运行时校验。
- 通用 Worker transport 不能依赖 Engine 或具体业务类型。
- 实时与池化调用必须共享相同的协议和错误语义。

## 候选方案

### A. 各调用方自行断言响应类型

- 优点：协议层改动少。
- 缺点：请求与响应没有统一配对，调用方会形成不同的兼容和错误处理分支。

### B. 通用 Worker transport 拥有 Engine 结果映射

- 优点：通用执行接口可以直接返回业务类型。
- 缺点：基础设施被 Engine 协议污染，静态映射也不能证明实际消息合法。

### C. Transport 返回 unknown，Engine 协议边界恢复类型

- 优点：通用层只保证传输，Engine 边界集中拥有配对、校验和错误分类。
- 缺点：每种 Engine 调用都需要明确的运行时协议定义。

## 决议

选择 C：**通用 Worker transport 只保证传输 envelope 并返回 `unknown`；Engine 协议边界统一拥有请求与响应配对、运行时校验和错误分类，实时与池化路径共享该契约。**

- 调用方不能用自由泛型或类型断言声明未经验证的业务结果。
- 通用 transport 不导入 Engine 请求、响应或业务错误类型。
- Engine 的执行失败、transport 故障、protocol 故障和合法空值必须保持不同语义。
- 请求与响应 RPC 和单向流式投影保持不同协议，不通过同一抽象相互模拟。

## 代价

Engine 协议层需要维护成对类型和运行时校验，并验证实时与池化路径的一致性。通用 transport 的直接消费者必须显式处理 `unknown`，这是保持基础设施边界所接受的成本。

## 重新评估条件

- Engine RPC 被能够生成双端类型、运行时编解码器和错误联合的协议框架替代。
- 多个非 Engine 消费者都需要相同的“任务自带运行时解析器”能力，足以形成新的通用契约。
- 运行时协议校验被测量为显著瓶颈，且替代方案能保持同等错误可见性。
