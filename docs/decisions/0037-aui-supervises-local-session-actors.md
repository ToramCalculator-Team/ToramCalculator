# 0037 - AUI 统辖局部交互 Session

- **状态**: Accepted
- **日期**: 2026-07-13
- **决策层**: 应用层（AUI / 局部 Session / CUI）
- **相关 ADR**: Refines 0021、0024

## 决策问题

AUI 已是跨模态交互状态的唯一来源，局部 Session 则拥有持续的业务会话状态。系统需要决定两者是由外部代码平行协调、把所有输入集中到 AUI，还是建立明确的父子所有权和协议方向。

## 决策驱动

- AUI 不能膨胀为理解所有局部规则的全局命令总线。
- 局部会话状态需要独立于页面和单个 CUI 投影持续存在。
- 上下级状态必须各有唯一所有者，不能通过外部 effect 双向同步。
- 局部意图应先由最了解其规则的 Session 接纳、拒绝或升级。

## 候选方案

### A. 所有输入先进入 AUI

- 优点：入口集中，应用级状态可以观察全部操作。
- 缺点：AUI 必须重复局部 Session 的规则和编排职责。

### B. AUI 与 Session 平行存在，由外部代码协调

- 优点：生命周期可以分别创建。
- 缺点：长期所有权和协议方向落在第三个协调位置，形成第二套状态同步机制。

### C. AUI 统辖局部 Session

- 优点：状态所有权、生命周期和协议方向由 actor 层级直接表达。
- 缺点：需要维护清晰的父子协议，并分别观察父子快照。

## 决议

选择 C：**AUI 拥有应用级交互控制，具有持续局部状态的 Session 由 AUI actor system 统辖；具体输入先进入对应 Session，控制向下，提案与事实向上。**

- AUI 只拥有模态无关的应用级交互状态，不复制 Session 的局部状态和规则。
- Session 拥有局部会话、业务操作及其状态，并决定输入在本地处理、拒绝或上报。
- 上行信息只能表达提案或已发生事实，下行信息只能表达应用级控制；双方不得依赖对方的内部状态路径。
- 页面和其他 CUI 只是投影与输入适配器，其卸载不改变 Session 的业务所有权。
- 只有确实需要持续局部状态和上下行协议的交互空间才建立 Session，不按目录或页面对称预建。

## 代价

跨层操作需要同时观察 CUI、Session 和 AUI，父子协议也需要独立验证。停止顶层 actor system 会连带停止受其统辖的 Session，因此普通路由卸载不能被当作顶层生命周期结束。

## 重新评估条件

- 局部 Session 必须在 AUI actor system 停止后继续独立存活。
- 产品需要同时运行多个彼此独立的顶层 AUI。
- 某类 Session 不再拥有持续状态、上行决策或跨 CUI 投影，只剩转发职责。
