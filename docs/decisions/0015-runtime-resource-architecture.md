# 0015 - 运行时资源架构：双引擎常驻、共享场景、注意力机为相机摆位触发器

- **状态**: Superseded by 0042
- **日期**: 2026-06-12
- **决策层**: 跨层（引擎线程 / 渲染 / 视觉意图）
- **相关代码**: `src/lib/engine/core/thread/EngineService.ts`、`src/lib/3dScene/SceneRuntime.tsx`、`src/machines/interfaceStateMachine.ts`
- **相关 ADR**: Depends on 0009、0014；Superseded by 0042

## 背景

character 配置和 simulator 曾共用同一个默认引擎，页面间导航会互相覆盖场景状态。两者还需要共享昂贵的 Babylon 场景实例，但展示的世界内容互斥。相机同时面对注意力摆位、成员跟随和用户拖拽，需要明确控制边界。

## 候选方案

### A. 单主引擎、页面级 character 引擎、按阶段仲裁相机

- 优点：常驻资源较少。
- 缺点：character 状态无法跨 URL 保持；需要额外相机仲裁层。

### B. 双引擎常驻、共享场景切换内容、注意力只触发一次摆位

- 优点：引擎状态隔离且跨页面保持；Babylon 实例只创建一次；相机无需长期控制权仲裁。
- 缺点：两个引擎持续占用资源；场景内容切换需要统一边界。

## 决议

选 B：

1. simulator 与 character 使用两个具名、应用级常驻引擎，状态互相隔离。
2. Babylon 场景实例唯一且常驻；当前活跃世界内容在 character 与 simulator 之间互斥切换。
3. 注意力目标变化只触发一次“切目标并摆位”，随后释放相机控制；用户拖拽和成员跟随可以继续改变相机。
4. phase 只决定目标由谁设置，不决定相机长期服从哪个控制源。
5. 常驻引擎必须在唯一显式生命周期入口创建，不能通过 `ensure*` 惰性启动。

## 代价

- 即使停留在无需引擎的页面，双 worker/GPU 资源仍可能常驻。
- 决议依赖 character 与 simulator 不同时展示；未来分屏会要求重新设计内容模型。
- 场景内容切换的具体编排边界需要由渲染架构继续明确。

## 影响范围

- 引擎服务对外暴露明确的 simulator/character 实例，不再提供可被双方覆写的默认实例。
- 页面只申请共享 SceneRuntime 的内容能力，不拥有场景实例。
- 相机表现参数留在渲染层，注意力层只表达语义目标。
