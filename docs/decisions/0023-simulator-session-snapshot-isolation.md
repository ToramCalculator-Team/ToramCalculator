# 0023 - Simulator 会话采用方案快照隔离

- **状态**: Superseded by 0025
- **日期**: 2026-07-10
- **决策层**: 跨层（数据 / 应用状态 / 引擎）
- **相关代码**: `src/routes/(app)/(features)/simulator/`、`src/lib/engine/core/thread/`
- **相关 ADR**: Depends on 0015、0021

## 背景

Simulator 会话可以在打开 Wiki 或 Character 工作面时完整保留。Character 工作面允许修改当前模拟方案所引用角色的装备和技能，因此必须决定这些持久数据变更是否立即进入已经加载的会话。运行结果需要能够由同一份输入和随机种子复现，不能在一次运行中混合修改前后的角色规则。

## 候选方案

### A. 持久数据变更热更新到当前会话

- 优点：Character 修改可以立即反映到正在运行的模拟中，无需重新加载。
- 缺点：一次运行会混合多个版本的配置，运行结果无法再由单一输入复现；还需要定义每类变更在运行中的安全应用时机。

### B. 当前会话隔离于持久数据变更

- 优点：会话始终有一份明确输入，运行与分析结果可复现；Character 与 Simulator 两条数据流不需要双向同步。
- 缺点：用户修改 Character 后必须显式重新加载最新方案，当前会话不会自动获得修改。

## 决议

选 B：**Simulator 会话在加载时捕获一份方案快照；Character 或其他持久源随后的变更不得热更新这份快照。**

1. 方案快照包含本次会话所需的模拟方案、成员及其角色配置；它与会话运行中持续变化的 HP、位置、时钟等运行状态是两类数据。
2. Character 工作面只更新持久角色数据和自身预览，不向当前 Simulator 会话派发配置补丁。
3. 当前会话继续基于原方案快照运行和分析。只有用户显式重新加载最新方案或开始新会话，持久数据变更才进入 Simulator。
4. 运行结果的可复现输入是该方案快照与随机种子，而不是查询时刻的最新数据库状态。

## 代价

当前会话可能落后于持久数据，用户需要主动重新加载才能验证刚完成的 Character 修改。界面必须能避免把旧快照的结果误解为最新配置的结果；若未来确实需要实时协同编辑，只能在保留版本化输入与确定性边界的前提下重新评估。

## 影响范围

- Character 与 Simulator 不建立持久数据变更的热更新通道。
- Simulator 会话边界必须同时拥有方案快照身份、业务阶段和运行状态，不能只保留阶段标签。
- 引擎加载、重置、分析与回放都必须引用当前会话的方案快照。
- 迁移与验收见 `docs/plans/0021-aui-interface-state-migration.md`。

## 后续

ADR 0025 将冻结边界从“会话加载”修正为“从设计进入验证”：会话在设计阶段维护可变草稿，验证和分析使用随后捕获的不可变运行快照。Character 修改不热更新当前运行输入的隔离原则继续有效。
