# 0034 - 录制生成的成员流程保留相对模拟时间

- **状态**: Accepted
- **日期**: 2026-07-12
- **决策层**: 跨层（运行记录 / 行为生成 / 引擎）
- **相关 ADR**: Refines 0043

## 决策问题

运行时以离散 Tick 推进，但规则系统以模拟时间表达冷却、持续时间和事件时刻。行动录制需要决定以 Tick 序号、模拟时间还是自适应条件表达原运行节奏。

## 决策驱动

- 行动时刻不能因逻辑频率变化而改变领域含义。
- 运行记录必须使用引擎规则系统的权威时间，不能读取主线程或墙钟时间。
- 机械转换必须忠实保存原运行节奏，不能替用户发明等待可用、延后或重试策略。
- Tick 只用于离散帧排序与定位，不能成为持续时间的权威单位。

## 候选方案

### A. 保存相对 Tick

- 优点：可以直接定位原运行中的离散帧。
- 缺点：相同 Tick 在不同逻辑频率下代表不同持续时间，改变频率会改变行动时刻。

### B. 保存相对模拟时间

- 优点：行动时刻与引擎规则时间一致，不依赖宿主执行速度和逻辑频率。
- 缺点：执行仍只能发生在离散逻辑边界，无法命中两个 Tick 之间的连续时刻。

### C. 自动生成适应型流程

- 优点：配置变化后更可能完成全部行动。
- 缺点：系统会自行引入等待、延后或重试策略，生成结果不再是原录制的忠实转换。

## 决议

选择 B：**行动录制和由其机械生成的初始成员流程保留相对于运行起点的引擎模拟时间，不保存 Tick 作为行动时间，也不自动等待、延后或重试。**

- 行动的权威时刻是运行时接纳边界读取的模拟时间减去本次运行起始模拟时间。
- 重新执行时，行动在第一个满足“当前模拟时间不早于计划时刻”的逻辑边界尝试。
- `tickIndex` 可以继续用于帧排序、诊断和 checkpoint 定位，但不能用于推导行动持续时间。
- 单个行动被拒绝不能改变其他行动的计划时间；失败必须保留为运行事实。
- 条件、等待可用、重试和基于前序结果的控制流只能由用户显式编辑；编辑后的流程是新设计。

## 代价

模拟仍是离散执行，计划时刻若不落在当前逻辑步长边界上，只能在其后的第一个边界执行；逻辑频率决定量化精度，但不改变计划时间本身。运行产出需要同时维护模拟时间事实和帧 Tick 定位，两者职责必须保持清晰。

## 重新评估条件

- 引擎规则时间不再以单调模拟时间表达。
- 产品目标从忠实转换改为自动适应并尽量完成。
- 引擎支持连续时间事件调度，并需要在两个逻辑 Tick 之间精确执行行动。
