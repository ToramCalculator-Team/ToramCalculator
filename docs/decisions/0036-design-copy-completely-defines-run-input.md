# 0036 - Simulator DesignCopy 完整定义运行输入

- **状态**: Accepted
- **日期**: 2026-07-12
- **决策层**: 跨层（数据 / 应用状态 / 引擎 / 分析）
- **相关 ADR**: Supersedes 0025；Refines 0024

## 决策问题

DesignCopy 已能表示一套可运行的 Simulator 设计。系统需要决定每次运行是否还要建立独立运行快照和临时参数，还是让单个 DesignCopy 身份完整拥有全部可配置运行输入。

## 决策驱动

- 每条运行必须能由一个设计身份完整解释其可配置输入和分析口径。
- 相同结果影响值不能同时由设计、运行快照和临时参数拥有。
- 跨线程输入只能是设计的派生传输形状，不能成为第二个可编辑领域对象。

## 候选方案

### A. DesignCopy 之外保留运行快照与临时参数

- 优点：单次运行可以覆盖设计值，延续既有运行冻结模型。
- 缺点：同一运行输入由多个对象拼装，身份、比较和再次执行语义不稳定。

### B. DesignCopy 完整拥有可配置运行输入

- 优点：运行记录、设计差异和再次执行都指向同一设计身份。
- 缺点：所有会影响结果和分析口径的可配置值都必须纳入设计边界。

## 决议

选择 B：**Simulator DesignCopy 完整定义全部可配置运行输入；RunRecord 关联产生它的 DesignCopy，不再建立第二份运行快照或临时参数对象。**

- DesignCopy 一旦用于运行就保持原值，继续编辑通过新的候选表达。
- 跨线程运行输入是 DesignCopy 的纯派生传输形状，不具有独立领域身份和编辑权。
- 运行身份、时间、输入判决和结果属于运行事实，不进入 DesignCopy。
- 不改变逻辑结果的宿主策略不进入 DesignCopy；固定游戏规则仍由引擎规则集拥有。

## 代价

DesignCopy 必须持续覆盖全部用户可配置的结果影响因素，任何遗漏都会破坏运行可解释性。派生传输仍会产生物理复制，但不得演变为第二个长期事实源。

## 重新评估条件

- 单次运行必须合法覆盖设计值，且覆盖不应成为可比较的设计事实。
- DesignCopy 的数据图无法在目标规模下以可接受成本跨线程派生。
- 产品需要独立、可版本化并跨会话持久化的运行输入身份。
