# 0034 - 录制生成的成员流程保留相对 Tick 时序

- **状态**: Accepted
- **日期**: 2026-07-12
- **决策层**: 跨层（运行记录 / 行为生成 / 引擎）
- **相关代码**: `src/lib/engine/core/World/Member/runtime/BehaviourTree/`、`src/lib/engine/core/thread/`
- **相关 ADR**: Refines 0033；Related to 0036

## 背景

行动录制可以机械生成 Player 成员流程，但仍需决定录制时间表示什么。若严格保留原运行的相对 Tick，生成流程可以忠实复现用户节奏并揭示新配置下的行动冲突；若自动等待技能可用或上一行动完成，流程更容易执行完整，却会在不同配置下自行改变时间轴，混入录制器推断的行为策略。

## 决策驱动

- 机械转换必须忠实保存原运行时序，不能替用户发明控制策略。
- 配置变化导致的行动冲突应成为可分析事实，而不是由自动延后掩盖。
- 后续条件化编辑必须与原始录制明确区分。

## 候选方案

### A. 严格保留相对 Tick

- 优点：忠实表达原实操时间轴；适合作为方案对比的控制变量；配置变化造成的不可执行行动会显式暴露。
- 缺点：更换配置后，原本被接纳的行动可能在指定 Tick 失败；生成流程不具备自适应能力。

### B. 转换为条件适应型流程

- 优点：技能冷却或动作时长变化后仍更可能执行完整序列；更接近成熟自动战斗行为。
- 缺点：系统会替用户发明等待、重试和完成条件；不同配置实际执行的时间轴不再相同，无法视为原录制的忠实转换。

## 决议

选 A：**行动录制和由它机械生成的初始成员流程严格保留相对于运行起点的逻辑 Tick；行动条件变化不得导致系统自动延后。**

1. 每个被接纳行动记录相对本次运行起点的 Tick 偏移，而不是主线程时间戳或与上一行动完成事件的隐式关系。
2. “保存为成员流程”按这些偏移生成确定的定时行动，不推断“等待可用”“等待上一技能结束”或自动重试。
3. 到达指定 Tick 时，若行动因冷却、资源、状态或目标条件不满足而未被接纳，必须把失败写入新运行的输入轨迹；不得静默延后到以后执行。
4. 行动失败不能改变其他行动在录制中保存的 Tick 偏移；具体流程终止策略必须显式定义，不能通过时间漂移补救。
5. 后续流程编辑器可以由用户主动把固定 Tick 改成条件节点、等待节点或重试策略；编辑后的流程是新设计，不再等同于原始行动录制。
6. 运行使用的逻辑时钟配置由 RunRecord 关联的 DesignCopy 固定，分析同时展示计划 Tick、实际接纳结果和失败原因。

## 代价

机械生成的流程可能在 Character 配置、技能冷却或目标状态变化后部分失败，不能承诺“录一次便永远完整执行”。用户若需要适应型自动流程，仍需学习后续编辑能力。行为执行层还必须支持以逻辑 Tick 调度动作，并明确单个行动失败后是否继续执行后续计划；无论采用哪种终止策略，都不能修改其他行动原本的时间偏移。

## 重新评估条件

- 产品把生成流程的目标从“忠实转换”改为“自动适应并尽量完成”。
- 引擎不再以稳定逻辑 Tick 表达行为调度。
- 流程编辑器建立了可验证的条件、等待和重试语义，并明确允许录制器生成这些策略。

## 影响范围

- 行动录制 schema 保存相对 Tick 和完整语义行动 payload。
- 录制到 MemberBTSchema 的转换是无推断的机械编译，不承担流程优化职责。
- 分析器区分计划行动、实际接纳行动和被拒绝行动。
- 实施与验收纳入 `docs/plans/minimum-validation-loop.md` 的 Tick 记录和成员流程保存阶段。
