# 0025 - 在验证边界捕获不可变运行快照

- **状态**: Superseded by 0036
- **日期**: 2026-07-10
- **决策层**: 跨层（数据 / 应用状态 / 引擎）
- **相关代码**: `src/machines/`、`src/routes/(app)/(features)/simulator/`、`src/lib/engine/core/thread/`
- **相关 ADR**: Supersedes 0023；Refines 0024；Superseded by 0036

## 背景

Simulator 会话包含设计、验证和分析三个阶段。若会话加载时就冻结方案，设计阶段对阵容和流程的修改无法进入本次验证；若验证期间继续使用可变设计数据，一次运行又会混合多个输入版本，无法可靠复现。

## 候选方案

### A. Simulator 会话加载时捕获快照

- 优点：整个会话始终引用同一输入，实现边界直接。
- 缺点：会话内的设计修改无法进入验证，设计阶段名存实亡。

### B. 设计阶段维护草稿，进入验证时捕获运行快照

- 优点：设计保持可编辑，验证与分析仍绑定单一、可复现的输入。
- 缺点：会话必须明确区分设计草稿和运行快照，并处理二者版本不一致的展示与再次验证。

## 决议

选 B：**Simulator 会话在设计阶段维护可变的设计草稿；只有从设计进入验证时，才捕获本次验证与分析共用的不可变运行快照。**

1. 设计草稿表达用户当前正在编排的模拟输入，设计阶段的合法意图可以修改它。
2. 进入验证是唯一的冻结边界；引擎从该时点的设计草稿构造运行快照，并以运行快照和随机种子推进本次运行。
3. 验证和分析只引用本次运行快照，不读取此后变化的设计草稿、Character 持久数据或其他源数据。
4. Character 工作面的修改不得热更新当前运行快照。它是否以及如何进入设计草稿，必须由后续明确的合并语义决定，不能隐式同步。
5. 运行结果归属于产生它的运行快照与随机种子，不归属于查询时刻的最新持久方案。

## 代价

设计草稿与运行快照可以同时存在且内容不同，界面需要清楚区分“正在编辑的设计”和“本次结果实际使用的输入”。重新验证会产生新的运行快照，旧结果若继续保留，也必须继续绑定旧快照。

## 影响范围

- AUI 状态转移需要把“进入验证”作为显式冻结边界，但运行快照本身仍由 Simulator 会话领域模块拥有。
- 引擎加载、运行、分析与回放必须使用同一运行快照身份。
- Character、Wiki 和模拟任务不得修改活跃会话的运行快照。
- 迁移与验收见 `docs/plans/0021-aui-interface-state-migration.md`。
