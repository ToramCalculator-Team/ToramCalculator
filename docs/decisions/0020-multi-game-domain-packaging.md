# 0020 - 多游戏领域以源码目录隔离并由生成器发布命名空间

- **状态**: Accepted
- **日期**: 2026-07-10
- **决策层**: 跨层（数据 / 应用 / 路由 / 生成器）
- **相关代码**: `db/generator/utils/schemaCollector.ts`、`db/generator/enumInjector.ts`、`db/generator/generator.ts`、`src/routes/`、`src/lib/engine/`
- **相关 ADR**: Depends on 0019；Related to 0014、0017

## 背景

当前数据库、路由、组件和引擎默认 Toram 是唯一领域。第二个游戏不会共享相同的账号、角色、技能和资产结构；把 Toram 模型提升为平台通用模型会产生大量 nullable 字段和特殊分支，手工给所有名称添加前缀又会把领域边界散落在仓库中。

## 候选方案

### A. 建立通用 game_account/game_character

- 优点：跨游戏查询和通用 UI 直接。
- 缺点：迫使不同游戏服从当前 Toram 结构，形成错误平台抽象。

### B. 每个游戏独立数据库

- 优点：隔离最强。
- 缺点：需要多连接、多同步实例和跨库身份映射，超出首轮多游戏改造范围。

### C. 单数据库、领域源码隔离、生成器发布命名空间

- 优点：领域源码保留自然名称，物理表和生成产物全局唯一，继续复用现有本地优先基础设施。
- 缺点：生成器成为关键基础设施，并产生一次大规模命名迁移。

### D. PostgreSQL schema 隔离

- 优点：数据库命名整洁。
- 缺点：现有 Prisma、PGlite、Electric 和 SQL 生成链没有多 schema 边界。

## 决议

选 C：**平台继续使用单个 PostgreSQL/PGlite 数据库；每个游戏在独立源码领域中使用自然名称，由生成器发布带领域命名空间的公共 schema 和产物。**

核心边界：

1. 平台只统一到 `user -> game_profile`；不创建通用 `game_account` 或 `game_character` 表。
2. 领域源码中的模型、枚举和关系由 AST 转换统一加命名空间，禁止使用正则批量改写 schema。
3. 领域配置是模型入口、前缀、同步分类、server-only 分类和所有权解析的唯一事实源；生成器据此发布 domain manifest。
4. 依赖具体游戏数据结构、术语、规则或内容的代码都属于该游戏领域。
5. 当前 simulation engine 整体归入 Toram 领域；只有出现第二个真实实现后，才从两个实现中提取公共运行时。
6. 平台核心只保留认证、游戏目录、`game_profile`、同步/启动基础设施和真正通用的 UI 机制，不能反向依赖具体游戏模型。
7. Auth.js 认证表属于 server-only 平台模型，不进入 PGlite、Electric 或通用 changes 白名单。
8. 路由使用领域前缀；平台根路由只承担游戏入口，具体游戏页面和工具进入自己的命名空间。
9. 旧的无领域前缀 URL 不提供兼容重定向。

迁移策略也是本决议的一部分：该改造在 v1.0 后作为一次整体数据库里程碑完成，不设计从当前结构到新结构的在线增量迁移，也不在 v1.0 前进行零散前缀改名。具体命令、版本号和数据转换顺序不属于本 ADR。

## 代价

- 领域编译器错误会影响全部数据库和生成产物，必须具备 AST 转换、冲突检查和生成验证。
- 一次性重建会丢弃尚未上传的 local-only 数据。
- 旧 URL 不提供兼容重定向，书签和外链会失效。
- 当前完整 engine 先归入 Toram，暂时放弃“已经通用”的命名。
- 游戏数量或隐私要求增长后，单数据库和全量下行可能需要重新评估。

## 影响范围

- schema、生成器、路由、应用目录和引擎必须遵守平台/领域单向依赖。
- domain manifest 取代各模块重复维护的游戏表和入口清单。
- 数据库发布采用一次性领域迁移策略，不在 ADR 中维护执行 runbook。
