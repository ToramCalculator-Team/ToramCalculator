# 0035 - 本地设计副本按实体身份写回正式设计

- **状态**: Accepted
- **日期**: 2026-07-12
- **决策层**: 跨层（应用状态 / 数据 / 同步）

## 决策问题

会话内可以保留多个候选 Simulator 设计分别运行和比较。用户选中一个候选后，系统需要决定是创建一套新的持久实体，还是按已有实体身份把候选差异应用回正式设计。

## 决策驱动

- 已产生运行记录的候选输入必须保持不变。
- 写回正式设计时必须保留共享实体的身份和关系语义。
- 设计写回应复用统一的数据变更与同步边界。

## 候选方案

### A. 为候选设计创建新的持久实体图

- 优点：候选之间完全隔离，不修改原实体。
- 缺点：隐式复制会改变共享实体身份，并产生新的关系与版本管理语义。

### B. 按实体身份把候选差异应用回正式设计

- 优点：保留已有身份和关系，继续使用统一数据写入语义。
- 缺点：共享实体的修改会影响其他引用者，写回需要明确的差异与并发边界。

## 决议

选择 B：**用户显式应用候选 DesignCopy 时，系统按稳定实体身份计算差异，并把变化写回对应的正式持久实体。**

- 已用于运行的 DesignCopy 保持原值；后续编辑形成新的候选。
- 写回不隐式克隆共享实体，也不把完整对象作为新的匿名持久副本覆盖。
- 未应用的候选属于当前会话，不自动进入持久化或同步边界。
- 需要独立实体身份时，必须由用户执行显式复制或新建设计操作。

## 代价

应用一个候选可能同时改变多个相关实体，并通过共享身份影响其他设计。系统需要提供可理解的差异和原子写入边界；并发冲突继续由统一数据层处理，而不是在 DesignCopy 中建立第二套合并协议。

## 重新评估条件

- 候选设计需要被持久化、分享或跨设备协作。
- 正式设计改为不可变版本模型，不再允许按实体身份原地更新。
- 共享实体的影响无法由统一差异与冲突语义安全表达。
