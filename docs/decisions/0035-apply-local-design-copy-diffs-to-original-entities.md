# 0035 - 本地设计副本按差异更新原持久实体

- **状态**: Accepted
- **日期**: 2026-07-12
- **决策层**: 跨层（应用状态 / 数据 / 同步）
- **相关代码**: `src/routes/(app)/(features)/simulator/`、`db/repositories/`
- **相关 ADR**: Depends on 0036；Refines 0026

## 背景

用户需要在当前会话中保留若干相似 Simulator 设计，分别运行并比较，再选择其中一个作为正式设计。若每次编辑直接写数据库，旧设计会立刻丢失；若为共享 Character 自动创建版本或克隆实体，又会改变现有显式关系语义。需要明确本地副本包含什么，以及选择副本保存时如何更新正式数据。

## 决策驱动

- 已产生运行记录的设计输入必须保持不变，后续编辑形成独立候选。
- 保存候选设计时必须保留 Character 等共享实体的原有身份和关系语义。
- 副本写回应复用统一数据库变更语义，不建立第二套同步协议。

## 候选方案

### A. 每次修改直接持久化，运行记录各自保存旧输入

- 优点：没有本地设计分支，数据库始终是当前设计。
- 缺点：用户无法在多个候选之间切换和选择；每条运行记录需要重复保存旧设计数据；编辑与分析相互牵制。

### B. 本地复制完整设计，选中后按差异更新原实体

- 优点：多个运行可以共享同一设计副本；编辑形成新候选而不污染旧运行；用户决定何时把哪套设计写回数据库。
- 缺点：应用需要管理副本身份、结构化差异、批量写入和并发数据变化；共享实体的更新会影响其他引用者。

## 决议

选 B：**Simulator 会话在本地维护完整设计副本；用户选择一个副本作为最终设计时，系统按实体身份计算差异并更新对应的原持久实体。**

1. 设计副本包含该 Simulator 设计和运行所需的完整数据图，而不是只有 `simulator` 表字段或关系 ID。
2. 用户修改当前副本时直接修改副本中的对应数据；已经产生运行记录的副本保留原值，后续编辑通过 copy-on-write 形成新副本。
3. 多条 RunRecord 可以关联同一个设计副本，不为每条运行重复保存一份 Character 数据。
4. 用户显式选择“应用为正式设计”后，系统把该副本与正式设计基线做结构化差异，并把变化写回具有相同实体身份的数据库记录。
5. 修改共享 Character 时直接更新原 Character；所有引用它的 Simulator 随正式数据更新而变化。系统不得在保存副本时隐式克隆 Character。
6. 如果用户需要 Simulator 专属 Character，必须显式复制 Character 并修改 Member 关系，这是一项独立设计操作。
7. 未应用的设计副本只属于当前会话，不持久化或跨端同步；刷新或结束会话后允许丢失。

## 代价

保存一个副本可能同时更新 Simulator、Member、Character 和其他关联实体，并通过共享关系影响当前 Simulator 之外的设计。应用必须提供清晰的差异预览和原子写入边界；跨设备并发更新属于数据库写入冲突，沿用数据层统一语义，不在设计副本领域内建立第二套合并模型。数据写入层已有 insert、update 和 delete 语义，设计副本无需再建立一套特殊的删除协议。

## 重新评估条件

- 产品需要把候选设计本身持久化、分享或跨设备协作。
- 共享 Character 的写回影响无法由现有差异预览和数据库冲突语义安全表达。
- 正式设计改为不可变版本模型，不再允许按实体身份原地更新。

## 影响范围

- Simulator 会话拥有本地设计副本集合、当前编辑副本和副本到运行记录的关联。
- 数据写入层按实体身份生成 insert/update/delete 差异，不能把完整深对象当作单行 JSON 覆盖。
- Character 继续是独立共享实体；设计副本只复制其值到本地工作图，不改变持久关系所有权。
- 实施与验收纳入 `docs/plans/minimum-validation-loop.md` 的会话与再设计阶段。
