# 0030 - 只持久化 Simulator 设计而非运行产出

- **状态**: Accepted
- **日期**: 2026-07-12
- **决策层**: 跨层（数据 / 应用状态 / 引擎 / 同步）
- **相关代码**: `db/schema/models/data.prisma`、`src/routes/(app)/(features)/simulator/`、`src/lib/engine/core/`
- **相关 ADR**: Refines 0024；Related to 0036

## 背景

持久化模型需要在 Simulator 设计和运行产出之间划定边界。项目的高价值创作数据是 Simulator 设计本身：用户编排初始场景、成员配置与行为流程；运行产出是当前设计循环中的分析材料。若持久化运行输入、摘要和 Tick 历史，还必须同时承担规则版本、输入日志、历史兼容与恢复真实性，不能只把当前会话对象写入数据库。

## 决策驱动

- 持久化与同步层应只承载用户长期创作和跨设备恢复所需的权威事实。
- 运行历史若持久化，必须连同规则版本和恢复真实性整体设计，不能渐进堆叠缓存表。
- 当前会话应能自由生成和丢弃分析产物，而不扩大数据库与同步协议。

## 候选方案

### A. 持久化可重放运行历史

- 优点：刷新后仍可恢复测试结果和 Tick 时间轴，也可长期追溯旧运行。
- 缺点：需要输入日志、结果缓存、规则版本和历史兼容模型，扩大首个闭环并与 Simulator 设计职责重叠。

### B. 只持久化 Simulator 设计，运行产出限于会话

- 优点：持久模型只表达用户设计；验证和分析围绕当前会话形成闭环，不提前承担历史数据与规则版本管理。
- 缺点：刷新或结束会话会丢失测试记录，用户不能跨会话选择或恢复旧结果。

## 决议

选 B：**当前架构只持久化并同步 Simulator 设计；DesignCopy、RunRecord、结果摘要和逐 Tick 快照只属于当前 Simulator 会话。**

1. Simulator 保存完整设计事实，包括初始场景、成员配置和行为流程；它不保存某次运行的产出。
2. Character 等被引用资源只保存自身配置，不拥有某个 Simulator Member 的行为流程。
3. RunRecord 直接关联 ADR 0036 定义的不可变 DesignCopy，不创建可持久化的独立运行快照。
4. 当前会话可以积累多条运行记录并由用户选择比较，但刷新或结束会话后允许全部丢失。
5. 每 Tick 捕获与 UI 节流遵守 ADR 0028，属性形状遵守 ADR 0029；两者都只服务当前运行和当前会话分析。
6. 当前持久化架构不建立重放配方、历史运行目录、规则版本身份或跨版本恢复能力。
7. 未来若需要持久化运行产出，必须连同数据版本和引擎规则版本管理重新决策，不能把当前会话对象直接写入数据库。

## 代价

用户误刷新、关闭应用或结束会话会失去尚未使用的测试结果，无法跨设备或跨时间比较运行历史。Simulator 设计可以同步，但同一方案过去的运行表现不会随之同步。未来增加历史记录时仍需设计版本化与兼容边界。

## 重新评估条件

- 用户需要跨会话、跨设备或长期分享运行结果与 Tick 时间轴。
- 产品需要审计、排行榜或可验证基准，要求保存不可抵赖的运行证据。
- 引擎能够提供稳定规则版本身份和经过验证的输入重放协议。

## 影响范围

- 数据库与同步层只新增或迁移 Simulator 设计所需字段，不新增运行结果、重放输入或 Tick 历史表。
- Simulator 会话领域模块拥有 DesignCopy 和多条候选运行记录。
- Character 的行为字段迁移到 Simulator 所拥有的 Member；具体字段命名和模板继承语义另行决定。
- 引擎确定性仍是质量目标，但不是当前持久化边界的前置条件。
