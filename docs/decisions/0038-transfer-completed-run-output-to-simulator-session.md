# 0038 - 完成运行时把引擎产出移交 SimulatorSession

- **状态**: Accepted
- **日期**: 2026-07-13
- **决策层**: 跨层（引擎 / 通信 / 应用状态 / 分析）
- **相关 ADR**: Depends on 0024、0030

## 决策问题

运行期间由 Engine 产生权威结果。运行完成后，系统需要决定由 Engine 长期保存多条历史并提供查询，还是把完整产出可靠地移交给拥有业务分析生命周期的 SimulatorSession。

## 决策驱动

- Engine 负责执行活动运行，不应管理 Simulator 的历史、选择和比较。
- SimulatorSession 需要拥有当前会话中的多条运行记录。
- 通信或资源释放失败不能直接丢失刚完成的产出。
- 运行产出仍是会话数据，不进入持久化和同步边界。

## 候选方案

### A. Engine 保存当前 Session 的全部已完成运行

- 优点：避免在结束时完整移交，Engine 可以按需提供历史。
- 缺点：Engine 与 Session 共同承担运行历史和清理生命周期。

### B. Engine 在完成边界把产出移交 SimulatorSession

- 优点：活动计算和业务历史各有唯一所有者，数据流在完成边界明确交接。
- 缺点：需要可靠传输完整产出，并处理接收成功后的资源释放。

### C. 运行期间持续镜像到 SimulatorSession

- 优点：结束时没有集中移交。
- 缺点：通信和消费速度进入权威捕获路径，并引入持续双写与背压协议。

## 决议

选择 B：**Engine 只拥有活动运行和尚未确认移交的产出；SimulatorSession 拥有已经接收的 RunRecord。移交必须幂等，资源释放失败不能撤销已经接收的结果。**

- Engine 不拥有已完成运行的业务目录、比较选择或分析语义。
- Session 先校验并按运行身份幂等接收结果，再触发 Engine 释放待移交资源。
- 业务所有权移交与 Engine 资源释放是两个可区分的结果；释放失败只能重试清理，不能删除或重复建立已接收记录。
- Engine 在移交未收敛前必须保护唯一待移交产出，不能用新运行覆盖它。
- Session 结束时清理其运行记录；运行结果不因此进入数据库或同步协议。

## 代价

完整产出会增加完成边界的传输成本和 Session 内存占用。接收或资源释放失败时，Engine 需要暂时保留不可变产出，并可能阻止新运行，直到所有权和资源状态重新收敛。

## 重新评估条件

- 目标运行规模使完整结果移交持续超过交互延迟或内存预算。
- Engine 迁移到不适合一次性返回完整结果的远程执行环境。
- 产品要求跨会话持久化、分享或恢复运行历史。
