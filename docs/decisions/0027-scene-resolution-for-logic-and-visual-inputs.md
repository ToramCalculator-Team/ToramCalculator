# 0027 - 场景解析统一产出逻辑输入与静态世界资源

- **状态**: Accepted
- **日期**: 2026-07-10
- **决策层**: 跨层（数据 / 应用编排 / 引擎 / 渲染）
- **相关代码**: `db/repositories/simulatorEngine.ts`、`src/lib/engine/core/`、`src/lib/3dScene/`
- **相关 ADR**: Depends on 0016、0026；Related to 0025

## 背景

Simulator Member 只保存 Character 或 Mob 等领域资源的明确引用。逻辑模拟需要把这些引用展开为属性、装备、技能和难度参数；3D 场景未来还需要从同一资源取得模型、外观和动画信息。当前渲染链路只从逻辑引擎接收 `character | sphere`，并硬编码模型路径，无法在 Character/Mob 绑定模型后选择正确资产。

若把模型详情复制进每个 Member，会产生与 Character/Mob 源数据重复的视觉配置；若让逻辑引擎独自选择模型，又会让计算层拥有静态渲染资产，并使设计阶段必须先启动逻辑引擎才能显示场景。

## 候选方案

### A. Member 复制完整逻辑与模型数据，由逻辑引擎驱动全部渲染

- 优点：引擎加载后即可通过单一命令流提供所有场景信息。
- 缺点：Member 与领域资源形成重复数据；逻辑引擎承担模型解析；设计阶段和 3D 场景被迫依赖引擎生命周期。

### B. 统一解析引用，分别产出逻辑输入与静态世界资源

- 优点：持久结构保持引用式；逻辑和视觉输入来自同一次解析且版本一致；设计场景无需逻辑引擎也能加载正确模型。
- 缺点：需要新增明确的场景解析边界，并在 3D 层汇合静态资源与动态命令。

## 决议

选 B：**应用编排层通过唯一场景解析边界展开 Simulator Member 引用，一次产出同版本的逻辑世界输入与静态 `worldResources`；逻辑引擎只负责模拟动态，3D 场景按 `memberId` 汇合两条数据流。**

1. 场景解析读取设计草稿中的 Character/Mob 引用及其当前领域数据，输出一个已解析场景定义；调用方不得分别查询并拼装逻辑与视觉数据。
2. 已解析场景定义至少包含逻辑引擎所需的成员规则，以及渲染实体所需的 `memberId -> model/appearance/animation resource` 静态描述。二者必须标识并保持同一解析版本。
3. 设计阶段可以直接把当前解析得到的 `worldResources` 投影到 3D 场景，不需要先创建或运行逻辑世界。
4. 从设计进入验证时，逻辑输入与 `worldResources` 一起冻结进运行快照。验证、分析与回放不得混用其他时点解析的视觉资源。
5. 逻辑引擎继续发送位置、朝向、动作、存活和动态实体生命周期等事实，但不查询模型资产，也不独立决定静态外观。
6. 动态协议如需创建运行中实体，只能引用已解析世界资源中的稳定资源标识；不得另传一份可能与 `worldResources` 分叉的原始模型配置。
7. AUI 状态机只持有交互状态与 Simulator 会话阶段，不持有已解析场景定义、模型资产或运行快照。

## 代价

场景解析结果会同时服务逻辑与渲染，需要明确缓存失效、版本身份和加载失败语义。3D 层必须在动态命令到达前后正确处理静态资源就绪与实体汇合；未来运行中生成新实体时，也必须先有可引用的世界资源描述。

## 影响范围

- `db/repositories/simulatorEngine.ts` 当前直接构造引擎专用深关系树的职责需要收敛到场景解析边界。
- `RendererProtocol` 中只有 `character | sphere` 的实体描述以及 `EntityFactory` 的硬编码模型路径不能作为最终模型契约。
- `WorldContentDirector`/`EntitySystem` 消费静态 `worldResources`，实时命令流只更新动态状态。
- Character/Mob 模型绑定属于被场景解析读取的领域资源，不复制进 Simulator Member。
- 迁移与验收见 `docs/plans/0021-aui-interface-state-migration.md`。
