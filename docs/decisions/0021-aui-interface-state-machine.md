# 0021 - AUI 行为状态机作为跨模态交互状态源

- **状态**: Accepted
- **日期**: 2026-07-10
- **决策层**: 跨层（应用层状态 / UI / 渲染）
- **相关代码**: `src/machines/`、`src/components/features/character/EquipmentPanel.tsx`、`src/lib/3dScene/`
- **相关 ADR**: Supersedes 0012

## 背景

2D UI 与 3D 场景需要一致地解释用户当前在检查、编辑或模拟什么。旧模型以通用 `Target`、`Operation` 和独立注意力生命周期同步两端，但它把业务行为压缩成少数视觉操作，还要求具体投影通过 `SETTLED` 回执推进共享状态。

MBUI 的分层思想有助于区分任务、模态无关行为和具体表现，但现有 2D UI 与 3D 场景并非从抽象界面生成，近期也不需要完整的模型驱动开发流程。因此需要决定 AUI 层只承担什么职责。

## 候选方案

### A. UI 与场景双向同步

- 优点：每个投影可以沿用自己的局部状态，单点改动直接。
- 缺点：同一语义存在多份状态，输入模态增加后同步和回声关系迅速增长。

### B. 定义 AUI 形态并生成 CUI

- 优点：符合完整 MBUI/MDD 方法，抽象结构和具体界面的映射可系统化。
- 缺点：需要重塑已有 UI 与场景，成本远高于当前“统一交互状态”的需求。

### C. AUI 层只建立行为状态机

- 优点：保留模态无关的状态与事件语义，同时允许现有 2D/3D 投影保持各自形态和生命周期。
- 缺点：AUI 不再是完整界面模型，投影映射仍需手工维护。

## 决议

选 C：**AUI 层是一台模态无关的行为状态机，作为 2D UI 与 3D 场景共享交互语义的唯一事实源；它不定义抽象组件树或界面形态。**

必须遵守以下边界：

1. 状态路径表达有限交互模式；角色 ID、装备槽等动态目标放在 machine context，不编码进状态名。
2. UI、键盘和 3D 拾取等输入统一转换为语义事件。合法意图由状态转移和 guard 表达，不另建 `inputScheme` 或 capability 表。
3. “注意力”是当前状态路径与 context 的派生含义，不建立独立注意力 actor，也不维护额外注意力生命周期。
4. 2D UI 与 3D 场景分别投影同一 snapshot，彼此不通知。任一投影重建后都能从 snapshot 和领域数据恢复表现。
5. 相机补间、overlay 动画、资源加载等 CUI 状态由投影自行管理，不通过完成回执驱动 AUI 状态。
6. 状态机只持有跨模态共享的交互状态；领域数据、`SceneRuntime` 生命周期和局部 UI 状态仍由原模块负责。

本决议继承 ADR 0012 的“共享交互语义只有一个事实源”，但替换其 `Target + Operation + attention lifecycle` 模型，并明确不建立独立注意力 actor。

## 代价

- 现有投影需要显式把各自输入翻译为语义事件，并从 snapshot 派生表现。
- 状态路径会带有业务语义，新增交互空间时必须维护清晰的层级和命名。
- 不采用完整 AUI 形态模型，无法自动生成 CUI，也不能自动验证所有投影结构一致。
- 如果未来需要从同一抽象界面大规模生成多种全新模态，应重新评估完整 MDD。

## 影响范围

- 应用层只保留一套跨模态行为状态模型，不再并列维护视觉意图机和注意力 actor。
- UI 与场景的依赖方向统一为“语义事件进入状态机，snapshot 流向投影”。
- ADR 0015 的双引擎常驻、共享 `SceneRuntime` 和显式生命周期入口继续有效；其中注意力机相关表述以本决议为准。
- 需求边界见 `docs/concepts/interface-state-requirements.md`，迁移步骤见 `docs/plans/0021-aui-interface-state-migration.md`。
