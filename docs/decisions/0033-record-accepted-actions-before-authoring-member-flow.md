# 0033 - 只用被接纳的语义行动生成成员流程

- **状态**: Accepted
- **日期**: 2026-07-12
- **决策层**: 跨层（输入 / 引擎 / 运行分析 / 持久设计）
- **相关代码**: `src/lib/engine/controller/MemberController.ts`、`src/lib/engine/core/MessageRouter/`、`src/lib/engine/core/World/Member/`、`src/lib/engine/core/thread/`
- **相关 ADR**: Depends on 0024、0032

## 背景

首个闭环采用用户手动操作，但用户希望把满意的实操过程保存为 Member 成员流程，从而不必先学习流程编辑器。一次操作中既有用户发出的输入，也有因冷却、资源、状态或目标条件而未被成员接受的输入；若直接录制 UI 点击或消息派发，会把无效尝试写进持久流程。现有线程协议中的 `action_sequence` 只传连续的技能 ID，既不表达完整语义行动，也不保留权威模拟时序。

## 决策驱动

- 持久成员流程只能来自引擎实际接纳的语义行动，不能来自 UI 点击或消息送达。
- 分析仍需要保留被拒绝输入，不能为了流程生成而丢失操作事实。
- 从运行产出写回持久设计必须由用户显式触发。

## 候选方案

### A. 录制用户原始输入并直接生成流程

- 优点：完整保留用户做过的每次尝试，录制点可以靠近 UI。
- 缺点：重复点击和必然失败的输入会污染流程；主线程时间不等于模拟时间；UI 形态会进入持久设计语义。

### B. 分开输入轨迹与行动录制，只用被接纳行动生成流程

- 优点：分析仍能看到失败输入，生成的流程只包含实际可执行行动；录制不依赖具体 UI；接纳 Tick 可由 Worker 权威确定。
- 缺点：引擎必须暴露明确的行动接纳事实，不能把“消息已派发”误当成“行动已接纳”。

## 决议

选 B：**RunRecord 分别保留输入轨迹和行动录制；只有成员运行时真正接纳的语义行动可以由用户显式转换为 `member.behavior`。**

1. 输入轨迹记录用户发出的语义输入及其接纳或拒绝结果，可用于两次运行的操作差异分析。
2. 行动录制只记录被 Member FSM 或等价权威边界真正接纳的行动；冷却中重复点击、非法目标和其他被拒绝输入不得进入录制。
3. 录制使用运行内逻辑 Tick，不使用 UI 事件时间或 `Date.now()` 作为权威时序。
4. “消息已通过 MessageRouter 派发”不等于行动已接纳；接纳事实必须由实际决定行动开始或入队的运行时边界产生。
5. RunRecord 中的行动录制仍是会话内运行产物。只有用户显式执行“保存为成员流程”时，系统才转换并写入当前 Player Member 的 `behavior`。
6. 运行结束不得自动创建或覆盖成员流程；保存前后的 Simulator 设计变化继续走正常持久化与同步链路。
7. 现有分支任务 `action_sequence` 不是行动录制或成员流程的权威 schema；它可以被后续统一契约替换，但不能因名称相似而直接复用。

## 代价

系统需要同时保留输入轨迹与行动录制，并在 Worker 中建立可靠的接纳回执。用户看到的点击次数可能多于可保存的行动数，UI 必须能解释哪些输入被拒绝。把线性录制转换成 Behavior Tree 还需要定义时序、失败策略和目标引用；这些转换规则不能由录制层猜测。

## 重新评估条件

- Member FSM 不再是行动接纳的唯一权威边界。
- 产品要求录制被拒绝输入并将其转换为显式重试或条件节点。
- 流程创作改为完全独立的编辑器，不再支持从实操运行生成初始流程。

## 影响范围

- MemberController 继续发送语义输入，但不拥有权威录制时间。
- Worker 的运行记录捕获输入结果和行动接纳事实，并关联到运行 Tick 与目标 Member。
- 分析 UI 可以展示失败输入；流程保存入口只消费行动录制。
- Player 行为绑定需要支持由行动录制生成的成员流程；完整流程编辑器仍不是最小闭环前置条件。
- 实施与验收纳入 `docs/plans/minimum-validation-loop.md` 的 Tick 记录和再设计阶段。
