# 成员动作编排链设计（可热插拔执行链）

- **状态**: Draft（讨论中，未实现）
- **日期**: 2026-06-05
- **决策层**: 编排层
- **相关**: ADR-0010（ProcBus/DomainEventBus 角色）、ADR-0011（成员事实成对派发）、Pipeline overlay 机制（同构参照）

## 一、要解决的问题

FSM 单个状态的 entry/action 里现在是**写死的命令式代码**（如 `resolveDamageAndApply` 内联的一长串：求公式→跑管线→算 hpAfter→扣血→判死亡）。这段代码：

- 无法被数据（托环/passive）增量改写——想在"伤害应用"过程中插一段逻辑（如"最后的抵抗"封顶 hp-1），只能改引擎代码。
- 与 Pipeline 形成割裂：Pipeline 已经是"可被 overlay 热插拔指令的纯数值执行链"，但它**只管数值**；而"改属性、发事件、塞延迟事件"这类**副作用编排**没有对应的可插拔结构，只能 inline。

## 二、核心设计

把"action 内部写死的命令式代码"换成一条**可热插拔节点的可编排执行链**。

- **执行链 = 有序的节点序列**，每个节点是一次"成员可执行操作"（改属性、发 ProcBus 事件、向 EventQueue 塞延迟 FSM 事件、跑某条管线取数值……）。
- **节点可被数据增量插入/替换**——就像 Pipeline overlay 按锚点插指令一样，托环可以往执行链的命名锚点插节点。
- **开放给执行链的"可调用操作集"，与行为树 action 的本质同源**：都是"成员能做的事"的有限集合。区别只是调用场景（编排链 vs BT）。

### 与 Pipeline 的关系：同一抽象的两个实例

| | Pipeline | 动作编排链 |
|---|---|---|
| 处理对象 | 纯数值 | 副作用编排（控制流 + 动作） |
| 节点 | 三地址指令（`target = a op b`） | 成员可执行操作 |
| 热插拔 | overlay 按 target 锚点插指令 | 托环按锚点插节点 |
| 纯度 | 纯函数，无对外副作用 | 有副作用（这正是它存在的理由） |

二者**互补且同构**：编排链节点内部可以调用 Pipeline 做纯计算，Pipeline 不再需要 `emit` 算子向外发事件（那是编排链节点的职责）。这也是 Step 5 删除 `emit` 算子的依据——副作用有了正经的家。

## 三、致死事件：用现有 ProcBus，不引入新机制

"最后的抵抗"（队伍最后一人受到致死伤害时，封顶 hp-1，并在 5+tLv 秒后死亡）是"改写结算"类效果，不能用事后订阅普通 `damage.received`（那时 HP 已扣成负、人已死，来不及改写）。

**关键洞察（来自讨论）**：伤害流程本来就需要在"算出致死"时发一个信号去驱动 FSM 转入死亡状态（现状是 `MobStateMachine` 用 `修改属性` 事件 + `满足存活条件` guard inline 判断，见 `MobStateMachine.ts:460-469`）。把这个 inline 判断**提成一个显式的 ProcBus 致死事件**，则一个事件同时满足两个消费方：

1. **FSM**：订阅它，转入"死亡"状态（取代现在的 guard inline 判断）。
2. **最后的抵抗托环**：订阅它，执行"封顶 hp-1 + 调度延迟死亡 FSM 事件"。

因此**不引入** `onProposeFatal` 之类的新拦截器机制（那是平行炉灶，违背"复用总线"原则）。改为：ProcBus 新增一个致死事件类型（暂名 `damage.fatal`），在伤害应用编排链的"致死判断"节点发射。它的语义是**"提议致死"**——发射时机在"算出 hpAfter≤0、尚未真正转入死亡状态"之间，允许订阅者改写结算（把 hpAfter 改成 1 从而取消本次致死）。

> 时机要求：`damage.fatal` 必须在"扣血后、转死亡状态前"发射，且订阅者对它的处理（如封顶）要能反映回最终的存活判定。这与 GAS 的 `PostGameplayEffectExecute`（算出会死→给一次改写机会→再 commit）语义一致，但**载体是 ProcBus 事件而非独立钩子**。

## 四、验证用例：两个托环的真实数据

两个托环对照，说明本设计如何用**同一套 `subscriptions` 词汇**覆盖"事后响应"和"改写结算"两类，无需为后者新增数据行类型。

### 钉鞋（事后响应类）

"陷入迟缓状态时移动速度 +tLv×5%"。纯 ProcBus 订阅，现有机制即可（Step 4 已验证）：

```jsonc
{
  "id": "registlet_spike_shoes", "name": "钉鞋", "maxLevel": 10,
  "subscriptions": [
    { "eventNames": ["status.entered"], "requiredStatusTypes": ["slow"],
      "handlers": [
        { "type": "addModifier", "attribute": "mspd",
          "modifierType": "dynamicPercentage", "value": "{level} * 5",
          "lifetime": "bySource", "sourceIdSuffix": "slow" } ] },
    { "eventNames": ["status.exited"], "requiredStatusTypes": ["slow"],
      "handlers": [
        { "type": "removeModifierBySource", "attribute": "mspd",
          "sourceIdSuffix": "slow" } ] }
  ]
}
```

### 最后的抵抗（改写结算类）

"队伍最后一人受致死伤害时封顶 hp-1，5+tLv 秒后死亡"。**不新增数据行类型**，同样用 `subscriptions`，订阅新的 `damage.fatal` 事件：

```jsonc
{
  "id": "registlet_last_resistance", "name": "最后的抵抗", "maxLevel": 5,
  "subscriptions": [
    { "eventNames": ["damage.fatal"],
      // condition 取自开放给执行链/handler 的成员条件集
      "predicate": "isLastInTeam",
      "handlers": [
        // overrideHpAfter / scheduleFsmEvent 是"成员可执行操作集"的成员，
        // 与 BT action 同源；需要在 RegistletHandler 联合里新增这两种 handler 类型。
        { "type": "overrideHpAfter", "value": "1" },
        { "type": "scheduleFsmEvent", "event": "death",
          "delayMs": "(5 + {level}) * 1000" }
      ] }
  ]
}
```

### 两个用例揭示的待办（实现时落项）

1. **新增 ProcBus 事件 `damage.fatal`**：在伤害应用编排链的"致死判断"节点发射；FSM 改为订阅它转死亡状态（取代 `MobStateMachine.ts:460-469` 的 guard inline）。
2. **`RegistletHandler` 联合新增两种 handler**：`overrideHpAfter`（改写本次结算的 hpAfter）、`scheduleFsmEvent`（向 EventQueue 塞延迟 FSM 事件，回放安全）。这是"开放给执行链的可调用操作集"的扩充。
3. **`subscriptions` 支持 `predicate` 表达式**（如 `isLastInTeam`），目前过滤只有 `requiredStatusTypes`/`requiredDamageTags`，需扩展到通用成员条件。
4. **伤害应用流程归属**：已定为**成员类**（`resolveDamageAndApply` 所在层），非状态机。FSM 只负责"收到 `damage.fatal` 后转状态"。

## 五、范围与边界

- 本文档只确立**方向**，不含实现。第一个落地验证用例是"最后的抵抗"，因为它逼出"改写结算"的最小机制。
- **不做** BT 接管 FSM action（讨论中已否定）：FSM 骨架不变，只把 action **内部**的命令式代码换成可编排链。
- 跨 tick 的 RUNNING 语义**暂不引入**：编排链节点约定同步跑完，跨帧的事（如延迟死亡）一律交给 EventQueue/FSM 转换，绕开 BT 执行状态当前不可 checkpoint 的限制（`BtManager.restoreCheckpoint` 现为空实现）。
