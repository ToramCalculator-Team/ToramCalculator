# 技能类型分析（2026-05-22）

## 结论

`skill_variant` 当前采用三组行为槽表达运行时能力：

| 行为槽 | 字段 | 数量 | 入口时机 |
|---|---|---:|---|
| 主动行为 | `activeBehavior` | 0..1 | 玩家状态机接受释放并支付基础 HP/MP 门槛后 |
| 被动行为 | `passiveBehavior` | 0..N | 成员创建时安装 |
| 长期注册行为 | `registeredBehavior` | 0..N | 主动流程或行为树需要注册长期逻辑时 |
| 自定义主动行为树 | `activeBehaviorTree` | 0..1 | 主动释放时优先执行 |
| 自定义被动行为树 | `passiveBehaviorTree` | 0..1 | 成员创建时启动 |
| 自定义长期行为树 | `registeredBehaviorTrees` | 0..N | 被主动行为树或注册阶段长期挂载 |

默认 DSL 与自定义行为树是同级入口。`activeBehaviorTree` 存在时，主动释放执行行为树；`activeBehavior` 仍可承载 lifecycle、targeting 等释放参数，运行时不会执行它的效果体。

行为树关系放在 `skill_variant` 顶层 relation，行为 JSON 内不保存 `behaviorTreeId`。

Ex 技能本轮保留在技能配置/成员配置表单问题域，未进入行为 DSL 或行为树模型。

`skill.isPassive` 与 `skill.chargingType` 继续保留在 `skill` 表，当前改造只处理 `skill_variant` 的运行时行为形状。

类型结构见 [skillBehaviorTypes.ts](./skillBehaviorTypes.ts)。

本轮不单独抽取“托拉姆物语游戏模型层”。技能行为类型、默认 MDSL 模板与 DSL 参数说明先放在 `skillBehaviorTypes.ts` 附近，后续需要多游戏模型时迁移这一组内容即可。

## 行为类型

`SKILL_BEHAVIOR_KIND` 当前包含 7 类：

| 类型 | 所属行为槽 | 目的 |
|---|---|---|
| `DamageAction` | `activeBehavior` | 主动伤害流程 |
| `StatusAction` | `activeBehavior` | 主动状态流程 |
| `RecoveryAction` | `activeBehavior` | 主动回复流程 |
| `PassiveRule` | `passiveBehavior[]` | 成员创建时安装的被动规则 |
| `WorldObject` | `activeBehavior` | 主动创建世界对象 |
| `WorldZone` | `activeBehavior` | 主动创建世界区域 |
| `RegisteredAction` | `registeredBehavior[]` | 长期注册但可结构化的数据行为 |

自定义行为树是行为槽的替代入口，未列入 `SKILL_BEHAVIOR_KIND`。

## 执行语义

主动释放：

```text
if activeBehaviorTree exists:
  execute activeBehaviorTree
else if activeBehavior exists:
  execute activeBehavior DSL
else:
  finish skill flow
```

成员创建：

```text
install passiveBehavior[]
start passiveBehaviorTree when exists
collect attributeSlots from active/passive/registered DSL and behavior_tree relations
```

长期注册：

```text
registeredBehavior[] is interpreted by the default DSL executor
registeredBehaviorTrees[] is registered by behavior tree actions or registered stage
```

默认 DSL 的执行方式是“DSL 参数 + 内置 MDSL 模板”。模板由 `behaviorKind` 选择，DSL 参数注入模板 Agent；模板表达流程骨架，参数表达技能数据。

当前运行时已经读取新字段；默认 MDSL 模板接入仍是后续工作。无主动行为树且存在 `activeBehavior` 时，玩家状态机会记录 warning 并完成技能流程。

## 默认 MDSL 模板

默认模板作为技能行为类型声明的附属内容维护，不写入 `behavior_tree` 表。

`behavior_tree` 表只保存用户自定义行为树、维护者自定义行为树和迁移保留下来的旧行为树资源。

模板选择规则：

```text
behaviorKind -> SkillBehaviorMdslTemplates[behaviorKind]
behaviorParams -> template agent context
template.definition -> BtManager
```

模板 action 只做编排入口，具体动作由绑定层读取 `behaviorParams` 后调用现有运行时服务：

| 模板 action | 读取的数据 | 调用的运行时 |
|---|---|---|
| `skillBehaviorPrepareActiveLifecycle` | `lifecycle` | 玩家状态机已计算的生命周期毫秒值 |
| `skillBehaviorDispatchDamageEvents` | `damageEvents` | `DamageAreaSystem` |
| `skillBehaviorApplyStatusEffects` | `effects` | `StatContainer`、状态清理调度 |
| `skillBehaviorApplyRecoveries` | `recoveries` | 资源写入与周期调度 |
| `skillBehaviorInstallPassiveRule` | `modifiers / pipelinePatches / subscriptions / thresholdWatchers` | StatContainer、Pipeline overlay、ProcBus、AttributeWatcherRegistry |
| `skillBehaviorRegisterLongRunningAction` | `registeredBehavior.behaviorParams` | BtManager 或长期行为注册器 |

内置模板源码见 [skillBehaviorTypes.ts](./skillBehaviorTypes.ts) 的 `SkillBehaviorMdslTemplates`。

## 释放生命周期与资源

`activeBehavior.behaviorParams.lifecycle` 承载旧时间字段：

| 旧字段 | 新位置 |
|---|---|
| `startupMs` | `lifecycle.startupMs` |
| `actionFixedMs` | `lifecycle.actionFixedMs` |
| `actionModifiedMs` | `lifecycle.actionModifiedMs` |
| `chantingFixedMs` | `lifecycle.chantingFixedMs` |
| `chantingModifiedMs` | `lifecycle.chantingModifiedMs` |
| `chargingFixedMs` | `lifecycle.chargingFixedMs` |
| `chargingModifiedMs` | `lifecycle.chargingModifiedMs` |
| `castingRange` | `lifecycle.targeting.castingRange` |

`effectiveRange` 属于具体行为参数。伤害技能迁入 `damageEvents[].delivery.effectiveRange`。

`elementLogic` 属于伤害公式。伤害技能迁入 `damageEvents[].formula.element.expression`。

基础 HP/MP 技能消耗归 `skill.cost` 管线与玩家状态机。`hpCost/mpCost` 表示释放门槛；无法支付时玩家状态机拒绝发动。

效果阶段的 HP/MP 改变归具体行为 DSL 或自定义行为树。破风之势、献祭这类技能记录为 `hpCost=0`，效果阶段扣 HP 与不足惩罚由自定义行为树处理。

魔法火神这类连续扣 MP 流程按持续行为处理。实测中技能 MP 消耗修正不会影响后续持续扣除，所以该扣除流程不进入 `skill.cost`。

## Pipeline 职责边界

Pipeline 只承载计算修正。

适合进入 pipeline 的内容：

| 内容 | 原因 |
|---|---|
| `skill.cost` | 输入基础 HP/MP 门槛，输出修正后门槛 |
| `skill.startup / charging / chanting / action` | 输入生命周期时间，输出修正后时间 |
| `hitCheck` | 输入命中上下文，输出命中参数或结果 |
| `damageCalc` | 输入伤害上下文，输出伤害值或伤害参数 |
| `ailmentChance / resist` | 输入异常概率上下文，输出修正后概率 |

不进入 pipeline 的内容：

| 内容 | 承载位置 |
|---|---|
| 等待、循环、条件分支 | MDSL / BT |
| 注册行为树 | BtManager action |
| 创建伤害区域 | 默认 MDSL action + AreaSystem |
| 安装事件订阅 | PassiveRule 安装动作 |
| 写入长期状态 | StatContainer 或长期行为树 |
| 效果阶段扣 HP/MP | 默认 MDSL action 或自定义行为树 |

`pipelinePatches` 只表示“安装一个计算槽修正”，不表达流程副作用。

## 伤害行为

`DamageAction` 默认语义：

```text
DamageAction {
  ActiveLifecycle.executeTemplateBody {
    for each damageEvent in damageEvents {
      wait(damageEvent.timing.delayMs)
      snapshotCasterForDamagePayload
      inferDamageTagsFrom(formula.element, formula.damageType, delivery.rangeType, ailments)
      buildDamageAreaRequest(damageEvent.formula, damageEvent.delivery, damageEvent.timing)
      registerDamageAreaRequest
    }
  }
}
```

伤害模板只创建 `DamageAreaRequest`。

`DamageAreaSystem.tick` 负责范围查询、再次命中节流、距离/目标数注入，并向目标成员派发 `受到攻击`。

目标侧 FSM 接收 `DamageDispatchPayload` 后按 `hitCheck → damageCalc → applyDamage` 处理命中、伤害和 HP/MP 写回。

单次技能内的多段伤害保存为多个 `DamageEventParam`；每个事件独立进行一次命中判定。

`DamageRangeType` 直接复用 `db/schema/enums.ts` 的 `DAMAGE_RANGE_TYPE`。

`warningZone` 由范围类型推导，表单不暴露该字段：

| `DamageRangeType` | 警告区 | 表单数据 |
|---|---|---|
| `Single` | none | 无运行时目标 ID 字段 |
| `None` | none | 历史兼容单体语义 |
| `Range` | red | 半径、可选再次命中间隔 |
| `Enemy` | red | 半径、可选再次命中间隔 |
| `MoveAttack` | red | 移动距离、宽度、速度、阻挡、可选再次命中间隔 |
| `Line` | red | 长度、宽度、可选再次命中间隔 |
| `Ground` | blue | 路径类型、数量、速度、宽度、可选再次命中间隔 |
| `Bullet` | blue | 半径、速度、路径命中、着弹命中、可选再次命中间隔 |
| `GroundFixed` | blue | 位置、半径、可选再次命中间隔 |
| `Meteor` | blue | 位置、半径、可选再次命中间隔 |
| `Explosion` | blue | 位置、半径、可选再次命中间隔 |
| `Attraction` | blue | 位置、半径、吸引强度、可选再次命中间隔 |

`targetId` 是运行时上下文，由目标选择和区域系统注入。

`hitIntervalMs` 属于区域再次命中参数，生成时下沉到 `DamageAreaRequest.hitPolicy.hitIntervalMs`。

`damageTags` 由模板推导，来源包括伤害类型、元素公式结果、异常衍生伤害和范围警告区。

`element` 使用公式表达；示例包括固定元素、主手元素、副手元素、元素所及影响后的元素。

改写 `damageCalc` 内部变量的技能使用现有 pipeline overlay 或自定义行为树。

## 命中异常

`AilmentParam` 保留：

| 字段 | 含义 |
|---|---|
| `type` | `AbnormalType` 枚举值 |
| `probability` | 命中后判定概率公式 |
| `durationMs` | 异常持续时间毫秒 |

异常判定时机统一为命中时。

## 状态与回复行为

`StatusAction` 默认语义：

```text
StatusAction {
  ActiveLifecycle.executeTemplateBody {
    recipients = resolveRecipients(scope)
    for each recipient in recipients {
      validate modifiers with StatContainer runtime schema
      add modifiers to recipient.statContainer
      schedule cleanup by source when durationMs exists
      register aura when aura exists
    }
  }
}
```

`StatModifierParam` 的精确结构归 `StatContainer` 运行时定义；`db/schema/jsons.ts` 只保留数组外形。

`source` 缺省时由模板按技能、buff、debuff 或 passive 来源生成。

`RecoveryAction` 使用单个 `amount` 公式。固定回复、额外回复、条件加成都写入同一个公式。

周期回复使用互斥调度结构：

| 调度 | 字段 |
|---|---|
| `Immediate` | 无额外字段 |
| `Periodic` | `durationMs`、`intervalMs` |

周期次数、终止条件、按命中次数回复进入自定义行为树。

## 被动行为

`PassiveRule` 复用 RuntimeAttachment 与托环同构能力：

| 字段 | 对应运行时 |
|---|---|
| `attributeSlots` | `StatContainer` 构造前的属性槽合并 |
| `modifiers` | `StatContainer.addModifier` |
| `pipelinePatches` | `PipelineOverlay` |
| `subscriptions` | `ProcBus` |
| `thresholdWatchers` | `AttributeWatcherRegistry` |
| `skillBranchActivators` | `CommonCondition.isSkillBranchActivated` |

`passiveBehaviorTree` 固定最多一棵；多个被动分支由行为树内部并行节点表达。

当前 CSV 中旧 `passiveEffects` 非空行为 0，迁移后没有生成被动自定义行为树。

## 长期注册行为

第三组命名为 `registeredBehavior`，含义是生命周期超过本次技能释放的行为。

典型例子是魔法炮充能。该行为需要长期运行计时逻辑，适合保存为 `registeredBehaviorTrees`。

`registeredBehavior[]` 预留给能结构化的长期注册 DSL；当前数据中未生成默认 registered DSL。

`registeredBehaviorTrees[]` 可多棵；行为树归属通过 `behavior_tree.registeredOwnerId` 表达。

## 状态槽与运行时数据

新增运行时数据通过 `AttributeSlotDeclarationData.path` 声明到 `StatContainer`。

路径就是槽身份，例如：

```text
skill.magicCannon.charge
passive.hpEmergency.lastTriggeredFrame
buff.blessing.startedAtMs
```

默认模板只保留计时器这类简单跨帧状态。

层数增减、分支选择、消耗层数后执行不同动作、状态转移图进入自定义行为树。

## 世界对象与世界区域

`WorldObject` 和 `WorldZone` 仍是默认行为类型，但首批运行时实现优先级低于伤害、状态、回复、被动。

对象与区域参数只保存领域事实：

| 类型 | 语义 | 关键字段 |
|---|---|---|
| `WorldObject` | 独立实体或对象 | `objectKind`、`spawnAt`、`durationMs` |
| `WorldZone` | 空间区域 | `zoneKind`、`center`、`radius`、`durationMs`、`tickIntervalMs` |

对象或区域内部流程超出默认模板时，使用对应行为槽的自定义行为树表达。

## 表单策略

普通用户上传技能时先选择行为槽，再选择默认行为类型。

`skill.chargingType` 来自技能本体，表单可展示但不写入 `skill_variant`。

表单按行为类型渲染：

| 行为类型 | 表单字段 |
|---|---|
| `DamageAction` | 生命周期、伤害事件、常数、倍率、元素公式、伤害类型、`DamageRangeType`、范围参数、异常 |
| `StatusAction` | 生命周期、作用域、持续时间、modifier、光环 |
| `RecoveryAction` | 生命周期、HP/MP、目标、回复公式、立即/周期调度 |
| `PassiveRule` | 属性槽、modifier、RuntimeAttachment 高级能力 |
| `WorldObject` | 生命周期、对象类型、位置、持续时间 |
| `WorldZone` | 生命周期、区域类型、中心、半径、持续时间、周期 |
| `RegisteredAction` | 属性槽、长期注册参数 |

高级模式直接编辑行为树表，并通过 `activeBehaviorTree / passiveBehaviorTree / registeredBehaviorTrees` relation 关联。

普通表单隐藏 `pipelinePatches / subscriptions / thresholdWatchers / skillBranchActivators` 的原始 JSON 编辑入口；维护者模式使用这些字段。

## 风险

`behaviorParams` 一旦允许任意步骤列表，就会退化成低配行为树。

默认类型只暴露领域事实：公式、范围、持续时间、modifier、订阅、阈值 watcher、对象/区域几何。

当录入内容需要顺序、条件、循环、状态转移、动态插入动作时，使用自定义行为树。
