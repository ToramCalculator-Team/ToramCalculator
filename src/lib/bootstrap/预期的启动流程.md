此架构下，应用启动流程分成“资源一致性检查、本地状态迁移、数据库迁移、数据同步、业务启动”五段。业务应用只在前四段完成后启动。

启动总览

浏览器打开网站
-> SW 处理导航请求
-> 加载最小启动脚本
-> 读取 release manifest
-> 判定资源版本
-> 更新资源或刷新
-> 迁移 localStorage
-> 初始化并迁移 PGlite
-> 注册 Electric sync
-> 等待首轮同步
-> 启动未同步 changes 上传
-> 挂载业务应用
-> 后台 warm cache / 周期检查
1. 浏览器导航
用户访问 / 或任意应用路由。

SW 对导航请求执行 network-first：

能连服务器 -> 获取最新 HTML
离线 -> 使用已缓存且兼容的 App Shell
HTML 不长期 cache-first，避免旧 HTML 引用已经不存在的旧 chunk。

2. 最小启动脚本运行
页面先运行一个最小 BootShell。
BootShell 只负责版本检查、迁移状态展示、恢复流程，不读取完整业务 store，不创建 PGlite，不启动业务组件。

此阶段允许读取的本地数据只有：

lastKnownRelease
updateLock
上次迁移状态
必要的显示配置兜底值
完整 localStorage store 要等 release manifest 判定后再读取。

3. 读取 release manifest
BootShell 请求固定版本入口：

/api/release
请求要求绕过缓存：

cache: no-store
返回内容包含：

releaseId
assetVersion
swVersion
storeSchemaVersion
dbSchemaVersion
minCompatibleStoreSchemaVersion
minCompatibleDbSchemaVersion
请求失败时进入离线判定：

本地存在 lastKnownRelease 且状态兼容 -> 离线启动
本地版本未知或迁移未完成 -> 显示恢复/重试界面
4. 资源版本判定
BootShell 比较本地记录的 assetVersion 和服务器返回的 assetVersion。

一致：

继续后续启动阶段
不一致：

通知 SW 更新资源
-> SW 预缓存新 core
-> 保留旧 asset cache
-> 页面等待 UPDATE_READY
-> 触发一次 reload
刷新后重新从第 1 步进入。
旧 asset cache 在旧客户端关闭或版本确认后再清理，避免动态 import 旧 chunk 失败。

5. SW 更新协商
SW 的启动角色是资源层执行器。

SW 负责：

安装新 SW
预缓存新 core
维护版本化 cache
保留旧版本 asset cache
向页面发送 UPDATE_READY / UPDATE_FAILED
页面负责：

决定是否 reload
记录 reload guard
处理 reload 后的继续启动
reload guard 用来防止资源更新失败时无限刷新。

6. localStorage 迁移
资源版本稳定后，BootShell 开始处理 localStorage。

流程：

读取 persisted store
-> 读取 persisted storeSchemaVersion
-> 检查是否低于 release.storeSchemaVersion
-> 执行 store migrations
-> 校验迁移结果
-> 写回新 store
可迁移：

保存用户偏好
继续启动
部分不可迁移：

丢弃对应分支
保留其他白名单配置
继续启动
完全不可读：

重置为 initial persisted store
记录恢复事件
继续启动
此阶段只保护用户配置。运行时 UI 状态、临时加载状态、弹窗状态不进入迁移契约。

7. 创建 PGlite Worker
localStorage 迁移完成后，BootShell 请求创建 PGlite Worker。

PGlite Worker 内部执行：

打开数据库
-> 创建 migrations 表
-> 读取 applied_migrations / schema_version
-> 对比 release.dbSchemaVersion
-> 执行缺失 client migrations
-> 校验 schemaVersion
createPgWorker() 只有在数据库结构可用后 resolve。

8. 数据库迁移失败分支
迁移成功：

进入 Electric sync 注册阶段
迁移失败且没有未同步 changes：

删除可重建同步库
-> 执行 client.sql baseline
-> 执行 baseline 到 target 的 migrations
-> 继续启动
迁移失败且存在未同步 changes：

停止数据同步
-> 停止业务启动
-> 进入恢复流程
恢复流程可以提供：

重试迁移
上传旧 changes
迁移 outbox
导出本地变更
只读模式
9. 注册 Electric sync
数据库版本确认后，PGlite Worker 注册 Electric sync。

顺序要求：

migration completed
-> schema verified
-> syncShapeToTable(...)
Electric 不应在迁移前启动。
旧 schema 上启动同步会造成表结构不匹配、shape 状态污染或同步失败。

10. 等待首轮同步
主线程进入 electricInitialSync 阶段。

它等待所有启动所需表完成 initial sync：

account
player
world
item
skill
character
simulator
...
此阶段完成后，应用可以认为服务端同步数据已达到启动门槛。

11. 初始化 temporaryAccount
首轮同步完成后，启动 temporaryAccount。

原因是账号恢复依赖同步后的 account/player/character 数据。
这个阶段完成后，用户身份和本地账号状态稳定。

12. 启动 changes 上传
账号和数据库都稳定后，启动 ChangeLogSynchronizer。

顺序是：

db migration completed
-> electric initial sync completed
-> temporaryAccount ready
-> changes upload starts
这样可以避免旧 schema changes 在迁移前上传，也避免账号未恢复时写入端点认证失败。

13. 挂载业务应用
前面阶段完成后，BootShell 挂载完整 Solid 应用。

此时业务组件可以安全假设：

资源版本一致
localStorage 已迁移
PGlite schema 已到目标版本
Electric sync 已完成启动门槛
账号状态已恢复
changes 上传器已进入受控状态
业务模块不需要自己处理版本兼容，只需要声明依赖的数据阶段。

14. 启动计算引擎和 Worker
引擎、模拟器、3D、计算 Worker 在数据阶段之后启动。

推荐依赖：

engine -> electricInitialSync
simulator -> electricInitialSync + pgworker
UI route -> appReady 或对应数据 ready
这样大型 Worker 不会和 PGlite wasm、Electric 初始同步抢首启资源。

15. 后台任务启动
应用 ready 后再启动后台任务：

warm cache
周期版本检查
资源完整缓存
非首屏 chunk 预热
低优先级数据刷新
后台任务失败不影响当前业务启动，只记录状态并等待下次重试。

多标签页规则
如果多个标签页同时打开，只有一个执行资源更新或数据库迁移。

其他标签页进入等待状态：

检测 updateLock
-> 订阅 BroadcastChannel
-> 等待迁移完成或刷新通知
迁移完成后，其他标签页 reload 或重新进入启动流程。

最终状态
启动结束时系统进入其中一种状态：

ready：完整可用
offlineReady：离线可用，跳过远端检查
degradedReady：核心可用，部分同步失败
recovery：存在未解决的本地数据风险
blocked：无法保证数据安全，等待用户操作
这条流程把错误控制在启动层：资源错误在资源阶段处理，配置错误在 store 迁移阶段处理，数据库错误在 PGlite 阶段处理，同步错误在 Electric 阶段处理。业务应用只接收已经判定过的运行环境。
