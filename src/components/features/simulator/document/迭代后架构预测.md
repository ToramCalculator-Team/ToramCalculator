# 迭代后架构预测

> 本文档在引擎运行时架构重构（Phase 0 ~ Phase 5）开始前撰写，描述迭代完成后项目前端页面与计算服务的预期架构。迭代结束后应对照本文件逐项检查。

---

## 一、迭代前现状概要（基线）

### 页面与引擎的关系

| 页面 | 是否使用引擎 | 引擎交互方式 |
|------|-------------|-------------|
| `/simulator/:id` | 是 | `realtimeSimulatorPool` (单 worker)，完整生命周期 |
| `/character/:id` | 否 | 纯 Babylon 3D 渲染 + DB 直读，无计算引擎 |
| `/wiki/:subName` | 否 | DB 直读 + VirtualTable 展示，无衍生指标 |

### 引擎生命周期

```
页面 Mount → new GameEngine(config) → initialize(data) → start → run → stop → cleanup → 页面 Unmount
```

- 引擎在 Simulation.worker.ts 中以模块级常量创建，Worker 启动即构造
- `JSProcessor`、`PipelineRegistry` 在 GameEngine 构造函数内 new，跟随 GameEngine 实例生死
- `cleanup()` / `reset()` 会连带清空编译缓存
- `realtimeSimulatorPool` 是模块级单例，但只有 `/simulator` 页面使用
- `batchSimulatorPool` 已定义但未接入任何 UI

### 引擎 runState

```
"stopped" | "running" | "paused"
```

无 idle/ready 概念；不支持"已构造但未开始仿真"状态下的数据查询。

### 2.2 引擎生命周期

```
App 启动
  → Worker 创建
  → Infrastructure 构造 (JSProcessor, PipelineRegistry)
  → GameEngine 构造 (接收 infra 注入)
  → runState = "idle"
  │
  ├─ loadScenario(data)  →  runState = "ready"
  │   ├─ setProfile(RealtimeProfile)  →  start()  →  "running"
  │   ├─ setProfile(PreviewProfile)   →  runPreview()  →  "running" → 自动停止 → "ready"
  │   └─ 直接查询(idle/ready 均可)
  │
  ├─ reset()  →  重建仿真层，保留 infra  →  "idle"
  │
  └─ 配置变更 (character 页)
      → patchMemberConfig(memberId, diff)
      → 只 rebuild 受影响 member (编译缓存命中不变的表达式)
      → 重新 runPreview()
```

**关键变化**：
- `runState` 新增 `"idle"` 和 `"ready"` 状态
- `idle` 下引擎可响应静态数据查询
- `ready` 下引擎可响应查询 + 接受 start/preview 指令
- `loadScenario()` 替代原 `initialize()`
- `reset()` 不再销毁 Infrastructure 层的编译缓存

### 2.3 SimulationProfile

零散配置（`engineMode`、`FrameLoopMode`、`enableIntentInput`）统一收敛为多维度 profile：

```
SimulationProfile {
  driveMode:            "clocked" | "unclocked"
  executionSemantics:   "full" | "previewSafe"
  stopPolicy:           "manual" | "untilFrame" | "untilBattleEnd" | "untilSequencesDone"
  outputPolicy:         "streamRealtime" | "collectFrameSnapshots" | "returnPreviewReport"
  probePolicy:          "disabled" | "rollbackAfterSkillProbe"
  acceptExternalIntents: boolean
}
```

三个预设：

| Profile | driveMode | executionSemantics | stopPolicy | outputPolicy | probePolicy | acceptExternalIntents |
|---------|-----------|-------------------|------------|--------------|-------------|-----------------------|
| Realtime | clocked | full | manual | streamRealtime | disabled | true |
| FastForward | unclocked | full | untilBattleEnd | collectFrameSnapshots | disabled | false |
| Preview | unclocked | previewSafe | untilSequencesDone | returnPreviewReport | rollbackAfterSkillProbe | false |

Profile 可在 `idle` / `ready` 状态下通过 `setProfile()` 运行时切换，无需重建引擎。

## 三、各页面的引擎消费方式

### 3.1 Wiki 页 (`/wiki/:subName`)

**迭代前**：纯 DB 直读，VirtualTable 展示，无引擎参与。

**迭代后**：

```
Wiki 页 → EngineService.queryDPSImpact(itemId, activeCharacterId)
        → Worker: engine 在 idle/ready 下查询当前 loadScenario 的成员属性
        → 对比基线属性面板，计算 DPS 增减
        → 返回 { dpsDelta: +15.3, dpsPercent: "+2.1%" }
```

- Wiki 页的表格列新增 DPS 影响指标（如"DPS +123 / -45"）
- 依赖当前"活动角色配置"（即 character 页最后打开/编辑的角色）
- 不需要 start 引擎，不需要帧推进
- 引擎在 idle/ready 下通过 `data_query` 通道即可响应
- 如果用户未配置任何角色，DPS 列可回退到"无基准"提示

### 3.2 Character 页 (`/character/:characterId`)

**迭代前**：纯 Babylon 3D 渲染 + DB 直读，无计算引擎。

**迭代后**：

```
Character 页 Mount
  → EngineService.loadScenario(characterConfig → SimulatorWithRelations)
  → runState = "ready"
  → 引擎立即以当前配置计算属性面板
  │
  ├─ 用户修改装备/技能
  │   → EngineService.patchMemberConfig(memberId, diff)
  │   → 引擎 rebuild 受影响 member（编译缓存复用）
  │   → 重新计算属性面板
  │   → 对每个技能执行 PreviewRunner.runSkillProbe()
  │   → 返回 PreviewReport[] 到 UI
  │
  └─ UI 显示
      ├─ 属性面板（ATK, DEF, MATK 等实时数值）
      ├─ 技能伤害预测列表
      │   ├─ 技能名 | 预测伤害 | MP消耗 | 施法时间
      │   └─ (每次配置变更后自动刷新)
      └─ Babylon 3D 模型（保留，与引擎无关）
```

**关键设计**：
- Character 页使用 Preview profile
- 每次配置变更 → `patchMemberConfig()` → 增量 rebuild → 技能 probe
- `patchMemberConfig` 而非 `reset + initialize` 以复用编译缓存
- PreviewRunner 在 probe 前后使用 checkpoint/restore 自动回滚
- 响应速度目标：配置变更后 < 100ms 更新属性面板，< 500ms 更新技能预测

### 3.3 Simulator 页 (`/simulator/:simulatorId`)

**迭代前**：完整实时模拟，`realtimeSimulatorPool` 单 worker。

**迭代后**：

```
Simulator 页 Mount
  → EngineService.loadScenario(simulatorData)
  → EngineService.setProfile(RealtimeProfile)  // 或 FastForwardProfile
  → start()
  │
  ├─ 实时模式
  │   ├─ FrameLoop 提供 clocked tick
  │   ├─ MemberController 发送 intent (acceptExternalIntents = true)
  │   ├─ frame_snapshot / domain_event_batch / render_cmd 流式推送
  │   └─ Babylon 渲染器消费渲染快照
  │
  └─ 快速模式
      ├─ unclocked，fast-forward 分片调度
      ├─ collectFrameSnapshots 输出
      └─ 运行到 stopPolicy 条件满足后自动停止
```

与迭代前的主要差异：
- `EngineMode` enum 和 `FrameLoopMode` 被 `SimulationProfile` 替代
- `FrameLoop` 瘦身为纯时钟驱动，不再直接调用 `stepFrame()`
- `GameEngine.start()` 完全拥有 run loop 责任
- 支持运行中切换 profile（stop → setProfile → start）

## 四、主线程服务架构

### 4.1 EngineService（新增）

```
EngineService (主线程单例)
  ├─ realtimePool: SimulatorPool   // 单 worker，常驻
  ├─ batchPool: SimulatorPool      // 多 worker，按需
  ├─ activeCharacterId: Signal<string | null>  // 当前活动角色
  ├─ engineState: Signal<EngineRunState>  // 镜像引擎状态
  │
  ├─ loadScenario(data)        // 加载场景（→ realtimePool）
  ├─ setProfile(profile)       // 切换 profile
  ├─ start() / stop() / pause() / resume()
  ├─ patchMemberConfig(id, diff)  // 增量更新成员
  │
  │  // 自动分发：少量 → 单 worker，批量 → 多 worker 并行
  ├─ runSkillPreview(memberId, skillId): PreviewReport
  ├─ runBatchPreview(memberId, skillIds[]): PreviewReport[]
  ├─ runBatchDPSCompare(itemIds[], memberId): DPSImpactResult[]
  ├─ queryMemberStats(memberId): StatSnapshot
  ├─ queryDPSImpact(itemId, memberId): DPSImpactResult
  │
  └─ events (from worker)
      ├─ onFrameSnapshot
      ├─ onTelemetry
      ├─ onDomainEventBatch
      └─ onPreviewReport
```

**分发策略**：
- 任务数 <= 5：直接在 realtimePool 单 Worker 内顺序执行（避免分发开销）
- 任务数 > 5：从 realtimePool 捕获 checkpoint + 编译字典，分发到 batchPool 并行执行
- 阈值可根据实际性能数据调整

**与现有 `SimulatorPool` 的关系**：
- `EngineService` 包装两个 `SimulatorPool` 实例，提供业务语义 API
- `SimulatorPool` 保持为底层 worker 通信层，不变
- 页面不再直接 import `realtimeSimulatorPool`，改为通过 `EngineService`

### 4.2 上下文共享

```typescript
// SolidJS Context Provider（在 App Shell 层注入）
const EngineContext = createContext<EngineService>();

// App Shell
<EngineContext.Provider value={engineService}>
  <Router>
    <FileRoutes />
  </Router>
</EngineContext.Provider>

// 各页面
const engine = useContext(EngineContext);
```

所有页面共享同一个 `EngineService` 实例 → 同一个 realtimePool（常驻）+ 同一个 batchPool（按需）。

## 五、Worker 内部架构

### 5.1 迭代后 Simulation.worker.ts 结构

Primary Worker 和 Batch Worker 共用同一个 `Simulation.worker.ts`，但行为因接收的命令不同而分化：

```typescript
// Worker 级长期持有 —— 跨 engine reset 存活
const infra: EngineInfrastructure = {
  jsProcessor: new JSProcessor(),
  pipelineRegistry: createBuiltInPipelineRegistry(),
};

// 引擎实例 —— 接收 infra 注入
const gameEngine = new GameEngine(defaultConfig, infra);

// 消息处理 —— 根据角色不同执行不同命令
port.onmessage = async (event) => {
  // === Primary Worker 常用 ===
  // engine_command: 状态机命令
  // data_query: 数据查询（含 queryDPSImpact 等新命令）
  // load_scenario: 加载场景数据
  // set_profile: 切换 SimulationProfile
  // patch_member: 增量更新成员配置
  // run_preview: 执行技能预览
  // capture_checkpoint: 捕获 checkpoint（可返回序列化数据给主线程）
  // get_expr_dict: 导出已变换的表达式字符串字典

  // === Batch Worker 常用 ===
  // restore_checkpoint: 从序列化 checkpoint 恢复引擎状态
  // import_expr_dict: 导入表达式字典预热本地 JSProcessor 缓存
  // run_probe_chunk: 执行一批技能 probe 并返回 PreviewReport[]
  // run_dps_compare_chunk: 执行一批装备替换 DPS 计算
};
```

### 5.2 Checkpoint 流

```
                    captureCheckpoint()
                          │
    ┌─────────────────────┼──────────────────────┐
    │                     │                      │
    ▼                     ▼                      ▼
EngineCheckpoint     WorldCheckpoint       EventQueueCheckpoint
    │                     │
    │           ┌─────────┼──────────┐
    │           ▼         ▼          ▼
    │    MemberMgr    AreaMgr    SpaceMgr
    │    Checkpoint    Checkpoint  Checkpoint
    │           │
    │     ┌─────┼──────┬──────────┬──────────┐
    │     ▼     ▼      ▼          ▼          ▼
    │   FSM   Stat   Status    BtMgr    Pipeline
    │   Snap  Cont   Store     Chkpt    Mgr Chkpt
    │
    └─────────────── restoreCheckpoint() ──────────►
```

每个子系统实现 `Checkpointable<T>` 接口，顶层 `GameEngine` 递归组装。

## 六、数据流全景

### 6.1 Character 页配置变更流

```
用户修改装备
  → UI dispatch
  → EngineService.patchMemberConfig(memberId, { weaponId: newId })
  → Primary Worker: gameEngine.patchMember(memberId, diff)
      → member.statContainer 部分重算（编译缓存命中不变的表达式）
      → member.pipelineManager.invalidateAffected(diff)
  → EngineService.runBatchPreview(memberId, allSkillIds)
  → 分发策略判断：
      │
      ├─ 技能数 <= 5（单 Worker 顺序）
      │   → Primary Worker: PreviewRunner.runSkillProbe() × N
      │       → captureCheckpoint → probe → restore × N
      │   → 返回 PreviewReport[]
      │
      └─ 技能数 > 5（多 Worker 并行）
          → Primary Worker: captureCheckpoint() + getExprDict()
          → 主线程: 将 checkpoint + exprDict 分发到 batchPool
          → Batch Worker 1: restore → probe 技能 1~10 → 返回 reports
          → Batch Worker 2: restore → probe 技能 11~20 → 返回 reports
          → Batch Worker 3: restore → probe 技能 21~30 → 返回 reports
          → 主线程: 聚合 → 返回 PreviewReport[]
  → UI 更新属性面板 + 技能伤害预测列表
```

### 6.2 Wiki 页 DPS 查询流

```
Wiki 表格渲染行 (items[])
  → EngineService.runBatchDPSCompare(itemIds, activeCharacterId)
  → 分发策略判断：
      │
      ├─ 项目数 <= 5（单 Worker 顺序）
      │   → Primary Worker: checkpoint → 换装 → 算DPS → restore × N
      │   → 返回 DPSImpactResult[]
      │
      └─ 项目数 > 5（多 Worker 并行）
          → Primary Worker: captureCheckpoint() + getExprDict()
          → 主线程: 将 checkpoint + exprDict 分发到 batchPool
          → Batch Worker 1~N: 各自 restore → 换装 → 算DPS
          → 主线程: 聚合 → 返回 DPSImpactResult[]
  → 表格列渲染 DPS 增减指示器
```

### 6.3 Simulator 页实时模拟流（与迭代前基本一致）

```
start()
  → FrameLoop tick (rAF/setTimeout)
  → GameEngine.handleFrameLoopTick()
  → runSingleFrame() × N
      → stepFrame()
      → emitFrameSnapshot() (节流)
  → postMessage → SimulatorPool → UI
```
