# 架构决策治理标准

本规范组合 arc42 的文档分工、MADR 的决策结构、ATAM 的架构显著性思想、Nygard 的 ADR 生命周期，以及 Architecture Knowledge Management 的决策关系与视图。它不是要求完整执行这些方法，而是抽取一套可重复执行的最小治理协议。

本文使用以下规范词：

- **必须**：违反时会破坏决策知识的一致性。
- **应该**：默认执行；只有明确理由才能偏离。
- **可以**：由项目上下文决定。

## 信息模型

架构知识由不同时间视角的信息组成：

| 信息 | 回答的问题 | 权威载体 |
|---|---|---|
| 当前架构事实 | 系统现在如何组成和运行 | 代码、schema、测试、必要的架构概览 |
| 领域或产品概念 | 业务术语和稳定语义是什么 | `CONTEXT.md`、概念文档 |
| 架构决策 | 为什么选择当前边界而放弃其他可行方案 | ADR |
| 实施计划 | 如何迁移、按什么顺序交付 | 计划、Issue、PR |
| 问题或调查 | 观察到了什么、尚未决定什么 | Issue、调查或问题清单 |
| 代码契约 | 调用方必须遵守什么可执行规则 | 类型、schema、测试、必要注释 |

一项信息必须有一个主要权威载体。其他文档可以链接它，不得复制成第二份会独立演化的事实。

ADR 是历史理由的证据，不是当前实现清单、产品需求全集、会议纪要或任务跟踪器。

## 文档分类

按顺序询问以下问题，命中后停止：

1. **能否由代码、类型、schema 或测试完整表达？**
   - 能：使用代码契约。只有理由反直觉且通过显著性门槛时，另加 ADR。
2. **是否定义领域术语、用户可见语义或产品规则，而没有选择技术边界？**
   - 是：使用 `CONTEXT.md` 或概念文档。
3. **是否描述迁移顺序、首版范围、任务、命令、验收或进度？**
   - 是：使用计划。
4. **是否只是问题、风险、调查事实或尚未收敛的候选？**
   - 是：使用 Issue、调查或计划。
5. **是否选择了会长期约束系统的架构边界，并通过显著性门槛？**
   - 是：使用 ADR。
6. **是否描述当前系统整体结构，而不是历史选择？**
   - 是：使用架构概览。

常见误分类：

- “首版不做重放”通常是范围；“运行产出永不持久化”可能是架构决策。必须辨别它是暂缓还是长期边界。
- “UI 展示哪些差异”通常是产品语义；“UI 只能消费权威 Worker 事实”可能是架构决策。
- “字段使用扁平对象”只有在它构成跨线程或长期兼容契约时才可能是 ADR；局部对象结构不是。
- “修复同步 Bug”不是 ADR；改变确认身份、写入权威或回滚语义可能是。

## 架构显著性门槛

候选必须同时通过 G1 至 G4。

### G1：架构或质量属性影响

至少影响以下一项，并说明具体边界：

- 数据、领域对象或运行事实的权威所有者；
- 应用、会话、Worker、引擎或资源的生命周期所有者；
- 模块或上下文之间的依赖方向；
- 持久化、同步、线程、网络或外部接口契约；
- 跨模块的一致性、可恢复性、可复现性或安全边界；
- 性能、可靠性、安全性、可修改性、可观测性等质量属性。

“影响多个文件”不能代替 G1。“只影响一个核心模块”也不能自动否定 G1。

高风险决策应该把质量属性写成可检查的场景：

```text
在何种环境和刺激下，哪个系统部分必须产生什么响应，使用什么指标判断。
```

### G2：真实可行的替代方案

必须存在至少两个在决策当时真实可行的方案。稻草人方案、明显违法需求的方案和从未考虑过的补写方案不计入。

若只有一个方案能满足硬约束，应记录该约束或代码契约，而不是伪造 ADR 权衡。

### G3：有意义的逆转成本

至少存在一项：

- 数据或 schema 迁移；
- 已发布协议或兼容性迁移；
- 多个边界必须协调改变；
- 基础设施、依赖或供应商锁定；
- 已持久化用户语义需要重新解释；
- 大量调用方或长期运行状态需要迁移。

只需重命名、移动函数、替换局部实现或删除少量代码的选择通常不通过 G3。

### G4：理由不可见或反直觉

未来维护者仅查看最终代码时，无法可靠推导选择理由，或者一个合理维护者很可能“修正”掉刻意设计的边界。

若类型、测试和常见惯例已经充分解释选择，通常不通过 G4。

### 判定输出

每个候选必须留下以下评估结果，但评估本身默认不创建新文件：

| 门槛 | 通过 | 依据 |
|---|---|---|
| G1 架构或质量属性 | 是/否 | 具体边界或场景 |
| G2 真实替代方案 | 是/否 | 方案名称 |
| G3 逆转成本 | 是/否 | 需要迁移什么 |
| G4 理由不可见 | 是/否 | 未来为何会困惑 |

只有全部为“是”时进入颗粒度分析。

## 颗粒度与决策聚合

### 决策节点

先把讨论拆成候选节点，但不要立刻映射为文件。节点可能是：

- 核心选择；
- 核心选择确立的不变量；
- 直接推论；
- 独立的下游选择；
- 产品范围；
- 实施方式。

只有“核心选择”和能独立通过显著性门槛的“下游选择”可能成为 ADR。

### 聚合规则

候选 A、B 满足以下任一强条件时必须放在同一 ADR：

- A 是 B 的不变量、直接推论、接受代价或否决理由；
- 推翻 A 会使 B 的主要理由失效，反之亦然；
- 两者共享同一问题、同一组主要驱动和同一重新评估条件；
- 两者只能作为一个迁移单元被采用或取代。

A、B 只有同时满足以下条件才应该拆分：

- 各自独立通过 G1 至 G4；
- 各自有不同的候选方案或主要驱动；
- 可以只取代其中一个而不重新讨论另一个；
- 生命周期或重新评估条件不同。

### 独立取代测试

使用反事实句验证：

```text
假设未来只改变 A，B 的决议、理由和代价是否仍完整成立？
```

- 是：倾向拆分。
- 否：必须聚合。
- 不确定：继续澄清核心不变量，不能靠多建 ADR 回避判断。

### 不构成拆分理由

- 涉及不同文件或目录；
- 可以写成不同句子；
- 分属数据层、引擎层和 UI 层；
- 每项都有实现任务；
- 为了让单篇文档更短；
- 为了保存完整讨论顺序。

一次设计会话可以产生零篇、一篇或多篇 ADR，但数量不得与问题清单或讨论轮次数量绑定。

## ADR 内容契约

ADR 使用项目模板并保持可快速阅读。必须包含以下语义，标题名称可以遵循仓库惯例：

### 决策问题

说明现在必须决定什么。问题应该允许多个可行答案，不能把答案写进问题。

### 决策驱动

列出真正改变选择结果的约束或质量属性，通常不超过五项。一般背景不算驱动。

### 候选方案

只记录当时真实可行且实际考虑过的方案。每个方案同时记录优势和代价。

### 决议

第一句给出选择。随后记录决议确立的不变量和明确排除的边界。不要混入执行步骤。

### 代价

记录主动接受的不利后果、锁定和尚未解决的风险。没有真实代价通常意味着决策尚未理解或根本不需要 ADR。

### 重新评估条件

使用可观察条件，例如：

- 性能或容量超过阈值；
- 引入某类新成员或新运行模式；
- 需要持久化此前仅存在于会话的数据；
- 某项外部约束消失；
- 依赖方数量或兼容期改变。

避免“需求变化时重新评估”这类恒真句。

### 关系与证据

使用有类型关系连接 ADR。代码、调查和计划作为证据链接，不成为决议正文的替代品。

## 生命周期

允许的状态：

| 状态 | 含义 |
|---|---|
| `Proposed` | 正在主动评审，尚未成为权威 |
| `Accepted` | 已采纳的当前或历史权威决策 |
| `Rejected` | 完整评估后明确未采纳 |
| `Superseded by NNNN` | 被另一 ADR 取代 |
| `Deprecated` | 原决策对象已消失且没有替代决策 |
| `Withdrawn` | 草稿撤回或确认并非架构决策 |

状态规则：

- 探索性想法首先留在计划，不要求创建 `Proposed` ADR。
- `Proposed` 必须有正在等待解决的明确问题；没有活跃评审的 Proposed 必须收敛或撤回。
- `Rejected` 仅保留值得避免重复讨论的真实权衡，不保留随口提出的想法。
- `Deprecated` 不能代替 `Rejected` 或 `Withdrawn`。
- 状态必须与索引完全一致，不允许自由文本进度状态。

### 已发布与未发布

“已发布”至少指 ADR 已提交到共享版本历史、被代码或计划正式引用，或被团队作为权威传播。

- 未发布草稿：直接重写、合并、重新编号或删除。
- 已发布 Accepted：保持决议历史不变；语义变化使用新 ADR 取代。
- 错别字、链接、代码位置和不改变论点的澄清可以直接修订。
- 当时背景事实写错但结论不变时追加勘误，不创建新决策。
- 实现后来偏离 ADR 时，先判断是实现缺陷、决策失效还是新决策，不能自动改写 ADR。

## 决策关系

优先使用以下关系：

| 关系 | 含义 |
|---|---|
| `Supersedes NNNN` | 本决策取代旧决策 |
| `Depends on NNNN` | 本决策成立依赖另一决策 |
| `Refines NNNN` | 在独立边界上细化上层决策 |
| `Conflicts with NNNN` | 两项决策不能同时成立 |

`Related to` 只能用于确实无法表达方向的弱关系，不得成为默认关系。

取代关系必须双向可追踪：新 ADR 声明 `Supersedes NNNN`，旧 ADR 状态改为 `Superseded by MMMM` 并链接新 ADR。

如果一项所谓 `Refines` 不能独立取代，它很可能只是上层 ADR 的不变量，不应单独建档。

## 当前决策视图

决策目录必须提供两个不同入口。

### 当前决策视图

由人策展并按问题域组织，只展示当前 `Accepted` ADR。每个问题域说明这些决策共同建立的边界，例如：

```text
Simulator
  会话与任务生命周期
  设计事实与运行事实
  场景解析与 Worker 输入
  Tick 记录与分析投影
```

一个 ADR 可以跨多个模块，但应该选择一个主要问题域作为导航归属，其他位置使用交叉链接。

### 完整历史目录

按编号列出所有 ADR、状态和日期，服务追溯与机械检查。它可以自动生成，但不能取代人工策展的当前决策视图。

默认阅读路径是：当前架构或问题域概览 -> 当前 ADR -> 必要时进入历史取代链。不得要求读者按编号顺序重建架构。

## 存量整理

按以下顺序进行一次性整理。

### 1. 建立清单

记录编号、标题、状态、日期、主要问题域、关系、是否已发布、是否被代码或计划引用。

### 2. 处理未发布草稿

重新执行文档分类、显著性门槛和颗粒度聚合。可以删除、移动、合并和重新编号，不保留虚构的取代历史。

### 3. 处理错误分类

把活跃内容移到概念、计划、Issue 或代码契约。若原 ADR 已发布，保留编号和简短说明，状态使用 `Withdrawn`；不要继续更新错误载体。

### 4. 终结悬空状态

逐篇检查长期 `Proposed`：

- 已成为当前权威：`Accepted`；
- 明确未采纳：`Rejected`；
- 不再评审或不是 ADR：`Withdrawn`；
- 仍在评审：保留 `Proposed` 并写明未决问题。

### 5. 检查当前权威冲突

同一事实只能有一个权威来源。若多个 Accepted ADR 对所有权、生命周期、依赖或数据语义给出不兼容结论，必须通过新决策收敛或确认其中一项只是实现偏差。

### 6. 建立导航

先创建当前决策视图，再保留完整历史目录和取代链。不要通过移动大量已发布文件制造链接 churn。

### 7. 保持历史真实性

不得为了减少文件数量合并已发布 ADR 并伪装成原始决策，也不得为了编号连续删除真实历史。整理目标是降低默认阅读负担，不是缩短编号。

## 治理例外

本标准不能消除所有架构判断，但必须让例外显式。

当用户坚持创建未通过门槛的 ADR 时：

1. 列出未通过的门槛和推荐载体；
2. 说明这会增加哪类维护成本或歧义；
3. 获得用户对例外的明确确认；
4. 在 ADR 中注明它是治理例外及原因，不伪造不存在的权衡。

当仓库已有规则与本标准冲突时，先提出统一修改方案。不得同时保留两个模板、两组状态或两套索引规则。

## 理论来源

- Michael Nygard, *Documenting Architecture Decisions*：轻量 ADR 与不可变历史。
- MADR：决策驱动、候选方案与后果的 Markdown 结构。
- arc42：当前架构、概念、风险和决策的文档分工。
- SEI ATAM：质量属性、敏感点、权衡点和架构风险。
- Jansen and Bosch, *Software Architecture as a Set of Architectural Design Decisions*：架构决策及其依赖。
- Kruchten, Capilla and Duenas, *The Decision View's Role in Software Architecture Practice*：面向读者的决策视图。
