# 实时模拟页面启动慢 — 排查结论

> 本文记录针对"导航进入实时模拟页 → 看到画面/视角就位"链路耗时过长（实测 4–6s，含一次约 6s 的 worker 初始化）的排查结论。诊断用 `[perf]` 日志已全部移除，本文是其留存结论。

## 一、整条进入链路（按时间顺序）

```
SimulatorPage.onMount
  └─ createResource: selectSimulatorByIdWithRelations(simulatorId)   ← DB 查询
        └─ getDB → createPgWorker()                                  ← 等 worker init
              └─ PGlite.worker init:
                   createDatabase → applyMigrations → syncTable×N     ← 逐表 electric 同步
        └─ 嵌套关联 SQL 执行（simulator + camp + team + member ...）
  └─ 门控：等 store.database.tableSyncState.simulator = true 才 refetch
RealtimeSimulator.onMount
  └─ engine.loadScenario(simulatorData)
  └─ engine.setRuntimeConfig
  └─ refreshMembers
  └─ addMemberController
  └─ activateSceneSession
        └─ acquireRealtimeSession:
             await initializationPromise（等 SceneRuntime 场景就绪）
             getRenderSnapshot
             applyRenderSnapshot → 逐成员 spawn/teleport/face/action
                  └─ createCharacter: loadGLB → instantiateHierarchy → cloneAnimations
```

## 二、三个延迟大头（按量级排序）

### 1. PGlite worker 初始化 ≈ 6s（最大头，首次进入）
- `createDatabase` + `applyMigrations` + **逐表 electric 同步**（`syncTable` 对每张表做一次 `syncShapeToTable` 握手）。
- 同步是**串行逐表**完成的，`simulator` 表排在同步队列靠后的位置，意味着即使只需要 simulator 数据，也要等前面的表同步握手依次完成。
- 这是"冷启动"成本：worker 建库 + 迁移 + 首次全量 shape 同步。二次进入若 worker 已存活则可跳过。

### 2. 嵌套关联 SQL 查询（第二大头）
- `selectSimulatorByIdWithRelations` 是深层嵌套关联查询：simulator → campA/campB（team[]）→ members（member[]）→ 各成员子关系。
- 在 PGlite（WASM 内运行的 Postgres）里，深层 `jsonArrayFrom` 嵌套子查询开销不小。
- 诊断时对 >50ms 的单条 SQL 做过打点，嵌套关联查询是除 worker init 外的主要耗时来源。

### 3. 角色模型创建（进入后的视觉停顿）
- `createCharacter` 三段：`loadCharacterModel`（GLB 加载）→ `instantiateHierarchy`（层级实例化）→ 动画克隆。
- 每个 Player 成员都会走一遍；GLB 加载与动画克隆是其中较重的部分。
- `applyRenderSnapshot` 对每个成员串行执行 spawn/teleport/face/action。

## 三、门控与时序

- `[simulatorId].tsx` 用 `createEffect(on(tableSyncState.simulator, ...))` 做门控：**只有 simulator 表同步完成才触发 refetch**。
- 这意味着即便 DB 查询本身很快，也必须等 electric 同步把 simulator 表标记为就绪——这把"同步耗时"串进了首屏关键路径。
- `acquireRealtimeSession` 内 `await initializationPromise`：还要等 SceneRuntime 的 babylon 场景 `executeWhenReady` 完成，才能 getRenderSnapshot。

## 四、可考虑的优化方向（未实施，供后续决策）

1. **并行化逐表同步**：`syncTable` 目前串行 await，若改并行或按需优先同步 `simulator` 相关表，可显著缩短首屏门控等待。
2. **同步与查询解耦**：考虑让 simulator 这类首屏必需表享有更高同步优先级，而非排在长队列尾部。
3. **嵌套查询拆分/懒加载**：首屏只取渲染必需字段（成员 id/位置/类型/队伍），其余子关系（技能、装备明细等）进入后再懒加载。
4. **角色模型预热/缓存**：`loadCharacterModel` 结果可跨成员复用（已用 instantiateHierarchy 复用层级），GLB 本身可在导航前预加载。
5. **worker 预启动**：在用户进入模拟页之前（如在列表页 hover/点击时）提前触发 `createPgWorker()`，把 6s 冷启动挪到用户感知之外。
6. **骨架/占位**：首屏先用白球占位渲染全部成员（mob 已是白球），角色模型加载完成后再替换，避免视觉空窗。

## 五、本次会话其他改动与发现（非性能，相关上下文）

### 已完成的改动
- **成员初始队形**：新增纯函数 `src/lib/engine/core/World/Member/memberFormation.ts`（`computeMemberFormation(campA, campB)`），径向同心对峙规则；引擎 `loadScenario` 与渲染层相机预摆位**共用同一函数**，保证位置一致。配套单测 `memberFormation.test.ts`（node:test，7 用例全过）。
- **mob 白球渲染**：spawn 链路加 `entityType`，Mob 走 `createSphere`（白色，半径 0.4），其余走 `createCharacter`。两条 spawn 路径（snapshot 路径 + Member 直发）都已接通。
- **相机可被鼠标控制**：根因是 `scene.activeCamera` 一直停留在初始化时设的 `backgroundCamera`，而 babylon **只把鼠标/滚轮输入路由给 activeCamera**——`acquireRealtimeSession` 创建 `realtimeCamera` 后从未切换。已在 `SceneRuntimeCore` 创建相机后加 `scene.activeCamera = realtimeCamera`。
- **相机限位**：给 ArcRotateCamera 设 `lowerRadiusLimit/upperRadiusLimit/lowerBetaLimit/upperBetaLimit`，防滚轮缩放穿模/翻转。
- **鼠标事件被 DOM 拦截**：`#mainContent`（`(app)/(features).tsx`）是 `z-40` 全屏容器，盖在 `z-0` canvas 上吃掉鼠标事件。已在实时模式下给它加 `pointer-events-none`（内部 UI 面板各自 `pointer-events-auto`，仍可点，空白区穿透到 canvas）。

### 待实现（功能缺失，非 bug）
- **WASD 控制成员移动这条链路根本没有接**。现状：`MemberController` 有完整意图 API（`move(x,y)` / `stopMove()`，经 `sendIntent` 发"移动"意图），Player FSM 有"移动状态"（静止↔移动中），但**全代码库没有任何键盘监听去调用 `controller.move()`**，中间这段缺失。
- 实现前需先明确 `MemberController.move(x, y)` 的 `x,y` 语义（目标坐标点 or 方向增量）——这决定 WASD 如何映射。FSM 的"移动/移动中"只是状态切换，实际位移逻辑需进一步确认在哪（action / runtime tick 的位移积分）。
- 注意：键盘=人物移动、鼠标=相机转向，是既定职责划分。键盘不应驱动相机。
