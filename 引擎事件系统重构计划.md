## 总目标

把引擎里七套都叫「事件」的机制按职责收敛为三个角色：
- **EventQueue** = 延迟 FSM 消息调度器（cross-frame scheduler）
- **ProcBus** = 成员内响应总线（passive/buff/registlet 在此声明「XXX 时 YYY」）
- **DomainEventBus** = 出 UI 投影（成员内事实 → 表现层）

分五步推进，风险递增。Step 1–3 已完成，Step 4–5 待做。

### Step 3 关键设计点（实现时已确认，供回顾）

- **等价性**：旧 watcher 每注册一个 (path,threshold) 配一个 handler；新模型每个 register 仍独立 emit 一次 `attr.crossed`，订阅者按 sourceId 一一对应订阅、predicate 按 `path + threshold` 过滤。N 个注册 → N 次 emit → 各订阅者各收各的，不丢不错配。
- **冷却**：installer 的 `lastFiredTimeMs` 冷却保留在订阅闭包里（本就不进 checkpoint，靠重放重建）。
- **checkpoint**：Member.captureCheckpoint 本来就没有 attributeWatchers 段（只 checkpoint statContainer/statusStore/btManager），所以删旧 watcher 不涉及 checkpoint 改动。AttributeThresholdSource 自带 captureCheckpoint 仅调试用，靠安装逻辑重放恢复。
- **EventCatalog**：维持 worker 启动一次性冻结现状，`attr.crossed` 只占一个内置位（起步方案，不拆 per-path）。

## Step 4 — 消除双总线（中风险，要 ADR）

### 问题

受击结算后当前要手工 emit **两次**：一次 `damage.received` 到自身 ProcBus（成员内响应），一次 domain event 到 DomainEventBus（出 UI）。Member 成了「接线盒」，同一事实派发两处，容易漏派/不一致。`healHp`/`healMp` 等叶子也直接调 `notifyDomainEvent`，是双总线在叶子层的残留。

### 目标方向

成员内总线（ProcBus）成为**唯一 emit 点**。DomainEventBus 降为 ProcBus 的**下游投影适配器**：订阅 ProcBus 的相关事件，转成 UI 投影。Member 不再手工双发，叶子不再直接碰 notifyDomainEvent。

### 动手前要调研的点

1. `notifyDomainEvent` 的所有调用点：`grep -rn "notifyDomainEvent" src/`。重点看 DamageResolution.ts（受击结算）和 healHp/healMp 叶子。
2. DomainEventBus 当前如何被消费（谁订阅它、投影成什么）：找 `ControllerEventProjector` 或类似。确认「ProcBus 事件 → UI 投影」的映射是否信息足够（ProcBus 事件 payload 是否够投影用，不够则需补字段）。
3. 确定性回放影响：DomainEventBus 改为订阅 ProcBus 后，投影是否仍只在「真实结算」时触发、预览模式是否一致。
4. 跨成员事件：ProcBus 是成员内的，出 UI 投影可能需要聚合多成员 → 确认 DomainEventBus 是否本就是引擎级单例（跨成员），这决定「下游订阅」是每成员一个 projector 还是引擎级一个。

### 落地步骤（ADR Accepted 后）

1. 先写 ADR（任务 #5，Proposed），把上面 4 点的结论写死。
2. DomainEventBus 增加「订阅 ProcBus」的接线：每成员 setEventCatalog 时，把该成员 ProcBus 的相关事件转发到 DomainEventBus。
3. 删除 DamageResolution 里手工的第二次 emit；改为只 emit 到 ProcBus，投影由下游自动发生。
4. healHp/healMp 等叶子移除直接 `notifyDomainEvent`，改为 emit 成员内事件（如 `hp.healed`），由下游投影。
5. capabilities 移除 `notifyDomainEvent`（BtManagerEnv 接口里它现在标着「双总线残留，待 ADR 消除」）。
6. tsc + 运行时验证：受击/治疗后 UI 投影仍正确，且只发生一次。

## Step 5 — Pipeline emit 纯度（高风险，要 ADR，最后做）

### 问题

计算层 Pipeline 通过 `emit` 算子（Member.ts 的 `pipelineEventSink`）在管线执行**中途**主动推编排层 ProcBus。这破坏了计算层的纯度和确定性回放：管线本应是「输入 → 输出」的纯计算，中途产生副作用（emit）会让重放结果依赖派发时序。

### 目标方向（已选）

移除管线中途 emit，改为**返回值**：管线执行收集 `emittedEvents[]` 作为输出的一部分，编排层在 `runPipeline` 返回后统一派发。计算层回到纯函数，派发时机由编排层掌控。

### 动手前要调研的点

1. `emit` 算子在所有 overlay / stage 的使用点：找 Pipeline instruction 里 `emit` 的定义和用例。
2. `pipelineEventSink` 当前被谁调用、emit 的事件去向（Member.ts setEventCatalog 里接的那个）。
3. 收集返回值的改造面：Pipeline 执行结果结构要加 `emittedEvents` 字段，所有 `runPipeline` 调用点要消费它。
4. 对预览模式确定性的影响：改返回值后，预览/回放是否更可控（这是收益，要在 ADR 里讲清）。

### 落地步骤（ADR Accepted 后）

1. 先写 ADR（任务 #6，Proposed）。
2. Pipeline 执行结果加 `emittedEvents: Array<{name,payload}>`；`emit` 算子改为往这个数组 push，不再调 sink。
3. 所有 `runPipeline` 调用点：拿到结果后把 `emittedEvents` 逐个 emit 到 ProcBus。
4. 移除 `pipelineEventSink`。
5. tsc + 运行时验证：管线触发的 proc（如命中后效果）仍正确派发，且回放确定。
